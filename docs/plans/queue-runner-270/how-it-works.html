EPIC #270

Autonomous work-item queue runner

An unattended, fleet-level wrapper around the already-working single-issue /implementation-loop. The Gateway gains named work lists plus a thin queue runner that drains them; the product skill and Cockpit are clients of the same lists; the Director stays dumb. The only genuinely new contract is the loop's terminal signal.

Sliced 2026-06-10 into 1 epic + 5 children· Dependency-ordered· Single-repo, GitHub-issue source for v1

1. The one idea

Today a human opens a session and types /implementation-loop 171 by hand, watches it run DEV → QA → squash-merge, then starts the next issue manually. This epic automates exactly that loop — the per-item unit of work is identical to what you run today.

What is automated
A named work list at the Gateway holds the order, e.g. Today: [#262, #256, #270, …]. The queue runner does what you do by hand: for each item it starts one implementation session, seeds it with that same /implementation-loop <N> string, watches it to a terminal signal, then advances. One item in flight per list, never starting the next until the current one finishes.

The runner's loop, step by step

  1. Claim the list (single-consumer claim — a second drainer is refused). #273
  2. Start one session on a Director and seed it with /implementation-loop <N> for the first item. #274
  3. Watch that session until it emits the machine-readable terminal signal. #272
  4. Record which of done / needs-human / failed the item ended on. #274
  5. Advance to the next item — never overlapping two items within one list. Repeat until the list is drained. #274

6. Decisions to confirm before/while building

Settled 2026-06-10 — Option B: item-ref model promoted now

The board shows three item sources (GH / DevOps / JIRA) and per-item area chips (Gateway / Core / Installer / Cockpit). Rather than hard-code GitHub and defer multi-source, we promoted the model:

  • #273 stores a structured ref { source, id, area? } — id is a string, source is an enum (github runnable; devops/jira accepted + stored). The store never rejects a source and never holds a status field.
  • #274 (runner) reads item.source/item.id and in v1 drives only github items (seeds /implementation-loop <id>). Non-github refs are skipped per a documented rule — queued and visible, never started, until a per-source adapter exists.
  • #275 renders source badge + area chip from the ref (real, not hard-coded); #276 appends { source: "github", id, area? }.

So the multi-source board is real on day one; making devops/jira runnable is the only piece deferred to a later phase (source adapters).

The one genuinely new contract (#272)
Assumption A1: the terminal signal rides the session transcript as a fenced sentinel block (IMPL-LOOP-TERMINAL / signal: done|needs-human|failed), matching the diagram's "FINISHED" arrow — not a new GitHub comment or REST call. If we'd rather the loop POST the signal to the Gateway directly, that overrides A1 and #272 + #274 re-point to that transport.

Constraints carried from the epic

  • Same-machine parallelism is gated by the single slot-5 test Director (one build output, one Control API port, strictly sequential sub-agents). v1 parallelism is across machines only; same-machine worktrees come after slot/build/port isolation exists.
  • One consumer per list — enforced by the claim, so two drainers never race the same items.
  • Single source of truth for item state — status lives in GitHub flow:* labels; the list stores order + assignment only, never a status field.
  • Parallel merges to main raise the conflict-escalation rate — safe because the loop escalates needs-human rather than force-merging.

Roadmap

  • v1: named lists over REST; GitHub issues as the item source; product agent + Cockpit as clients; cross-machine parallelism (one list per machine); the terminal-signal contract.
  • Later: same-machine worktree parallelism; source adapters that make devops / jira refs runnable (the model already carries them); list persistence across a Gateway restart.
