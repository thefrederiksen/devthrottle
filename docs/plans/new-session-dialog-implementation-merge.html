New Session dialog — merge Developer + QA into one Implementation session

Revises #254. Developer and QA stop being two separate sessions. They become roles inside one Implementation session that loops build↔verify until the task is done. The Product group shrinks from four tied sessions to two: Product + Implementation. Support leaves the group (the type stays selectable solo).

The Product group now uses two types — Product and Implementation. Developer, QA, and Support stay defined (old sessions still load, you can still spin one up solo) but are no longer part of the group. Integer enum values stay stable; legacy names keep deserializing, same pattern #254 established.

Why merge Developer + QA into one session

  • Same-repo parallelism doesn't work. Two sessions on one repo fight over the working tree, build output, the running app, ports, and slots. You can't reliably test a build while another agent is mid-edit on the same checkout.
  • Dev↔QA is a tight loop, not a one-way handoff. QA finds a defect → dev fixes → QA re-verifies → repeat. Across two sessions every turn costs a label handoff, a context reload, and a re-orientation. In one session the loop keeps full context and runs fast.
  • Single owner of the working tree. The Implementation session is the only thing that edits, builds, and runs the app — no coordination protocol between two agents over one filesystem.
  • Simpler human model. Product (decide + specify) → Implementation (build + verify, looping) → ship.

CenCon note: the four agent roles and the flow:* labels are unchanged — Developer and QA now run as roles inside the Implementation session, looping over flow:ready-dev / flow:ready-qa internally. The SOC 2 "QA is a different identity from the developer session" line in DEVELOPMENT_METHOD.md softens to role separation within one session.
