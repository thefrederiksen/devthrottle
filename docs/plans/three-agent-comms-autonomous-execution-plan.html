Autonomous Execution Plan: Three-Agent Communication

Claude Code + Codex + Pi  |  Close every gap, prove each capability with tests, run start-to-finish without stopping, and emit a large QA HTML report at the end.

This is the runbook a machine follows. It turns the gap list into ordered phases, each gated by automated tests, with pre-decided answers to every choice so it never has to pause for a human. It ends by generating a QA report that shows, per capability per agent, PASS with evidence, or NOT POSSIBLE with the reason it genuinely cannot be done.

Two honest outcomes only. For each capability and each agent the run produces exactly one of: PASS (proven by a test and a captured artifact) or NOT POSSIBLE (the agent's CLI genuinely lacks the mechanism, cited from the agent-expert reference). A third, temporary state BLOCKED-BY-ENV (e.g. a CLI is not installed) is allowed only while the environment is being fixed; it must resolve to PASS or NOT POSSIBLE before the report is final.

1. Autonomy model - how it runs without stopping

  • Phases are gated by tests, not by humans. A phase proceeds only when its test suite is green. Nothing waits for approval.
  • Every choice is pre-decided (section 2). The runner never reaches an "ask the user" branch; if it would, that is a planning bug, and the pre-decisions list is extended.
  • Failure handling is bounded fix-retry, then classify. On a failing gate the runner attempts a fix and re-runs the gate up to MAX_FIX_ATTEMPTS = 3. If still failing, it classifies: a bug in our code is logged and the phase is marked DEFERRED (run continues; the QA report lists it); a genuine feature gap is recorded as NOT POSSIBLE with a citation and the run continues. It never halts the whole run for one capability.
  • Resumable. Phase status is written to a state file (docs/cencon/proof/three-agent-comms/run-state.json). A restart skips completed phases. This is what lets the run survive a crash or a context reset and keep going.
  • Self-contained environment. The runner builds to its own test slots, launches test Directors, spawns and tears down its own sessions, and cleans up - all without touching the user's Directors (section 2, D-6 / D-7).
  • Progress is observable. Each phase appends a line to a run log and updates the state file, so the user can watch without intervening.

4. Test strategy - the proof layers

Layer 1 - Unit tests (xUnit, in the solution)

  • Each installer (Pi extension, Codex capability, readers) and helper (shared echo-verified submit) gets focused unit tests, same style as CodexHookInstallerTests / CodexDriverTests.
  • Transcript readers are tested against captured sample files (a real Pi JSONL, a real Codex rollout) checked into the proof folder.
  • Gate: dotnet test for the touched projects is green.

Layer 2 - Integration tests (Control API, single process)

  • Spin the Control API in-process where possible; exercise /fleet/*, /sessions/{id}/fleet-preamble, the new cc-status / cc-history endpoints with fakes.
  • Gate: integration suite green.

Layer 3 - Live trio end-to-end (the real proof)

  • Build to a test slot, launch a test Director via Task Scheduler, spawn real Claude + Codex + Pi.
  • Drive the capability inventory C1-C10 against the live sessions, capturing an evidence artifact for every assertion (a rollout/transcript excerpt, a cc-* output, a buffer snapshot).
  • The six-pair round-trip (C10) is the headline test: for each ordered pair (Claude-Codex, Claude-Pi, Codex-Claude, Codex-Pi, Pi-Claude, Pi-Codex) prove send + ask succeed.
  • Gate: every capability is PASS or a justified NOT POSSIBLE.

Phase 5 - Live trio verification L

Goal
Prove the whole matrix on real sessions.
  1. Build to slot 8, launch via Task Scheduler, spawn Claude + Codex + Pi.
  2. Run C1-C9 for each agent, capturing an evidence artifact per assertion.
  3. Run C10 - the six ordered pairs - send + ask each way; capture transcripts of each round-trip.
  4. Tear down sessions; stop slot 8 (D-7).
Gate
Every capability is PASS or a justified NOT POSSIBLE; all evidence written to evidence/.

Phase 6 - QA report generation M

Goal
The large, human-readable QA HTML report (section 7).
  1. Collect unit/integration results and the live evidence into one dataset.
  2. Render three-agent-comms-qa-report.html with the capability x agent matrix, per-cell evidence, and NOT-POSSIBLE reasons.
  3. Open it; write the final run-state summary.
Gate
Report exists, every cell is PASS or NOT POSSIBLE (no BLOCKED-BY-ENV remaining), every PASS links real evidence.

6. The NOT-POSSIBLE rule (the honesty gate)

A cell may be marked NOT POSSIBLE only when all of the following hold, so "we gave up" can never masquerade as "the agent cannot":

  • The agent's CLI has no documented mechanism for the capability, cited from the agent-expert reference (file + line/section), AND
  • A live probe confirms the mechanism is absent or inert on the installed binary (e.g. the Grok-style "hook output is ignored" finding), AND
  • No alternative family (A/B/C/D) achieves it either.

For these three agents we expect very few NOT-POSSIBLE cells - the inventory is achievable for Claude, Codex and Pi. The rule exists so that if, say, Pi's idle-detection proves fundamentally unreliable for a clean cc-devthrottle message ask, the report says exactly that, with the probe that shows it, rather than a green cell that is really flaky.

7. The final QA HTML report

Deliverable: docs/plans/three-agent-comms-qa-report.html - a standalone, human-readable report proving the cross-communication.

Structure:

  • Verdict banner - "N of 10 capabilities PASS for all three agents", run date, binary versions.
  • The headline matrix - rows = C1-C10, columns = Claude / Codex / Pi, each cell PASS (with an "evidence" link) or NOT POSSIBLE (with the cited reason). Colour + text, no symbols.
  • The six-pair round-trip grid - a 3x3 (minus diagonal) showing each ordered pair's send and ask result, each linking the captured round-trip transcript.
  • Per-capability sections - for each C#, what was tested, the exact assertion, and the evidence artifact (a rollout/transcript excerpt, a cc-* output block) inline.
  • NOT-POSSIBLE appendix - any cell that could not be done, with the citation and the live probe output that proves the feature is absent.
  • Test ledger - every unit/integration test name and result that backs the matrix.

8. Orchestration runbook (the loop the runner follows)

  1. Load run-state.json; determine the first incomplete phase.
  2. For that phase: do the work, run the phase tests (the gate).
  3. If green: mark the phase COMPLETE, commit (D-9), append to the log, advance.
  4. If red: fix-retry up to MAX_FIX_ATTEMPTS; then classify DEFERRED (bug) or NOT POSSIBLE (feature gap) and advance - never halt.
  5. Repeat until Phase 6 emits the QA report.
  6. Stop only at the very end (or on a true environment block it cannot self-resolve, which it records and continues past where possible).

This loop can be driven directly by the agent in a sustained run, by the /loop skill (self-paced), or as a multi-agent workflow. The state file makes any of them resumable.

9. Definition of done

  • Phases 0-6 COMPLETE (or their gaps classified) in run-state.json.
  • All new code committed to the feature branch with green unit + integration suites.
  • The live trio test ran and captured evidence for C1-C10.
  • The QA report exists, every cell is PASS or NOT POSSIBLE, every PASS links real evidence, and the three agents demonstrably start, find each other, message, ask, report status, and read history - in all directions that are possible.
  • Pushing the branch, opening a pull request, and tagging the release remain the only human steps (D-9).
