Three-Agent Communication: Release Plan

Claude Code  +  Codex  +  Pi  |  Goal: start them, they always find each other, and they communicate every way possible - then cut a release.

Scope is deliberately narrow: make these three agents excellent at talking to each other. Start any of them, they discover each other instantly, send messages, ask and get answers, see what the others are doing, and read each other's history - on one machine today, across machines through the Gateway. This plan lists exactly what already works, what is missing, and the work to a release.

The plumbing already exists. Sessions reach each other through their own Director with no token: cc-devthrottle session list (list the fleet), cc-devthrottle session whoami (your id), cc-devthrottle message send (one-way message, or all), cc-devthrottle message ask (ask and wait for the answer), cc-devthrottle session spawn (open a session). A message is delivered into the recipient as a framed prompt; cc-devthrottle message ask reads the reply from the recipient's terminal once it returns to idle. The fleet preamble injected at launch tells each agent these commands exist. The gaps are about making all three equally good at every one of these.

1. How the communication actually works (so the gaps make sense)

  • Discovery is the fleet list. On one Director, cc-devthrottle session list shows every session; across machines, each Director relays to the Gateway which aggregates. An agent only knows to use it because the fleet preamble was injected at launch.
  • Send (cc-devthrottle message send) frames your message with your identity and delivers it into the target as a prompt. This depends on the target's submit path working (typing the text and pressing Enter reliably).
  • Ask (cc-devthrottle message ask) sends the question the same way, then waits for the target to leave and return to Idle and reads everything it printed in between (ANSI cleaned). So cc-devthrottle message ask needs two things from the target: a working submit and accurate Working/Idle detection. It does NOT need transcript parsing.
  • Read history (past turns of another session) is the one thing that needs a transcript reader for that agent.
Correction worth stating: earlier notes said "cc-devthrottle message ask only works for Claude because of TranscriptRead". That is the standalone hosted-ask path. The fleet cc-devthrottle message ask is buffer-based and works for any agent whose submit and idle-detection work. Transcript reading is for history, not for asking.

3. What we are missing (the gap list)

Eight items take the three from "mostly works" to "excellent in every direction". Effort tags are rough: S = small, M = medium, L = larger.

G1. Pi fleet preamble MISSINGM

Why
Without it, a Pi session does not know it can list/message the fleet - it cannot "always find" Claude and Codex. This is the single most important gap for Pi.
How
Family C: ship a small Pi TypeScript extension (loaded from ~/.pi/agent/) that on session_start fetches GET /sessions/{id}/fleet-preamble using the injected CC_DIRECTOR_API + CC_SESSION_ID and injects it via before_agent_start; re-injects on /new (session_start reason=new) and on compaction. A PiExtensionInstaller drops/updates it on Pi launch (idempotent), mirroring ClaudeHookInstaller / CodexHookInstaller.

G2. Pi echo-verified submit RISKS

Why
PiDriver.SubmitAsync is a blind submit ("composer echo layout is unverified, so no echo gate yet"). That is exactly the bug that parked every programmatic prompt for Codex. If Pi drops the Enter, cc-devthrottle message send and cc-devthrottle message ask into Pi silently fail.
How
Reuse the shared echo-verified submit (extracted from ClaudeDriver/CodexDriver): type, wait for Pi's composer to echo the text, then a separate Enter. First verify live whether Pi even needs it; adopt if it drops the blind Enter.

G3. Codex: declare TranscriptRead PARTIALS

Why
CodexTranscriptReader is built and tested but the driver does not declare the capability, so the uniform "read this session's history" path treats Codex as unreadable even though the data is there.
How
Wire CodexDriver's ReadWidgets/ReadUsage/ListTranscripts to the existing reader and add DriverCapabilities.TranscriptRead. Cheap unlock for clean ask answers and history.

G4. Pi transcript reader MISSINGL

Why
Pi's conversation is JSONL under ~/.pi/agent/sessions/ but unparsed, so "look at Pi's history" and clean ask-answer extraction are impossible.
How
Reverse-engineer the Pi JSONL schema from a real session, write a PiTranscriptReader (same shape as Claude/Codex readers), declare TranscriptRead, and surface it through SessionHistoryReader / the History tab.

G5. Fix the cc-devthrottle message ask Windows print crash BUGS

Why
The data round-trip works, but cc-devthrottle message ask crashes while printing the answer on Windows (a Unicode encoding error on a prompt glyph). It exits with an error even though the answer arrived, which reads as "broken".
How
Force UTF-8 / ASCII-safe console output in the cc-devthrottle message ask Python tool (and audit the other cc-* tools for the same). This makes "ask and read the answer" clean for all three.

G6. Cleaner ask-answers for TUI agents PARTIALM

Why
cc-devthrottle message ask scrapes the terminal buffer. For Claude that is clean; for Codex/Pi the answer is interleaved with heavy TUI repaint, so the returned text is noisy.
How
When the target has a transcript reader (Claude now, Codex after G3, Pi after G4), read the last assistant message from the transcript instead of the buffer scrape. Falls back to the buffer when no reader exists. Gives crisp answers across all three.

G7. "Know what each other are doing": cc-status NEWM

Why
cc-devthrottle session list shows running/working/waiting, but not what a session is working on. The Director already computes per-session summaries and briefs.
How
Add a cc-status <id|all> tool that surfaces, per session: activity state, current repo/branch, and the latest one-line summary (from the existing /sessions/{id}/summary or turn-summaries). Optionally a --watch that streams changes. This is read-only and works for any agent.

G8. "Look at their history": cc-history NEWM

Why
An agent should be able to read another session's recent conversation, not just ask it a live question.
How
Add a cc-history <id> [--last N] tool that returns the last N turns of a target session via the transcript readers (Claude, then Codex G3, then Pi G4). Read-only; the relay goes through the Director like the other cc-* tools.

4. Per-agent: what is left

Claude Code essentially complete

  • Fleet preamble wired; discoverable; receives and answers; history readable.
  • Only shared item: benefits from G5 (cc-devthrottle message ask print fix) and gains G7/G8 as new tools.

Codex two small unlocks

  • Fleet preamble wired; submit fixed; discoverable; receives and answers.
  • G3 (declare TranscriptRead) - small. Then G6 makes its ask-answers crisp and G8 reads its history.

Pi the bulk of the work

  • G1 fleet preamble extension - so Pi can find and address the others (most important).
  • G2 echo-verified submit - so messages and asks actually deliver into Pi.
  • G4 transcript reader - so others can read Pi's history and get clean ask-answers.
  • After these three, Pi is a first-class peer to Claude and Codex.

5. Release plan

Release definition of done: on one machine, start Claude + Codex + Pi; each appears in the others' cc-devthrottle session list; every ordered pair can cc-devthrottle message send and cc-devthrottle message ask successfully; cc-status shows what each is doing; cc-history reads each one's recent turns; and a single self-test proves all of it.
  1. Land the cross-cutting bits first: shared echo-verified submit helper (used by G2), and G5 (cc-devthrottle message ask print fix).
  2. Codex unlocks: G3 (declare TranscriptRead).
  3. Pi to parity: G1 (preamble extension), G2 (submit), G4 (transcript reader).
  4. Crisp comms: G6 (transcript-based ask answers), then the new tools G7 (cc-status) and G8 (cc-history).
  5. Three-way verification: extend cc-devthrottle selftest into a trio test - spawn Claude, Codex, Pi; run send/ask/status/history across all six ordered pairs; assert PASS. Run it live before tagging.
  6. Cut the release: version bump and release notes ("super communication between Claude Code, Codex, and Pi: start, discover, message, ask, status, history - all directions").

6. Stretch (nice, not required for the release)

  • Start the trio in one shot: a convenience that spawns Claude + Codex + Pi as a named group (the Director already has session groups) so "start my three" is one action and they boot knowing they are a team.
  • Subscribe to a peer: a cc-watch <id> that pushes a short note to you when a peer finishes a turn or needs input - "know what each other are doing" without polling.
  • Cross-machine: the same flows already relay through the Gateway, so the trio works across machines once each Director is attached - verify as a follow-up.

7. Risks and checks

  • Pi submit may or may not need the echo gate - verify live before assuming; cheap either way.
  • Pi JSONL schema is undocumented - the transcript reader (G4) needs a real session file to reverse-engineer; this is the largest single item.
  • TUI answer noise - until G6, Codex/Pi ask-answers are buffer scrapes; acceptable but not crisp.
  • Idle-detection accuracy - cc-devthrottle message ask relies on Working->Idle; confirm it is reliable for Pi (Pi's continuous-output behaviour, if any) so asks do not time out or return early.
  • All three installed - Claude and Codex are confirmed on this machine; confirm Pi is installed before the live trio test.
