Transcription Consolidation

Issue #585  |  Source: mobile voice note 2026-06-20 (recording 238e948d...)  |  Status: DRAFT for review

Why this document exists. Transcription has grown into several independent code paths that behave differently. This document (1) states the ONE target workflow, (2) inventories every place we transcribe today with concrete file:line evidence, (3) gives a verdict on whether the desktop path changes the user's text, (4) lists the exact changes needed to make every path identical, and (5) proposes one shared visual component in a full desktop size and a small embeddable size. Read the flow and the gap list first, confirm them, then we implement.

1. The target workflow (the one flow, everywhere)

Every transcription surface - desktop dictation, desktop voice, phone recorder, phone push-to-talk - must follow exactly these four steps. No variations.

1

Start

User starts transcription. Capture audio locally. Show a live waveform and a timer. Show NO text yet.

->
2

Finish or Hold

When the user hits Finish or Hold, flush and capture ALL audio buffers, including the final buffer. Nothing is dropped.

->
3

Send whole to Gateway

Send the COMPLETE audio to the Gateway. The Gateway transcribes it once, using the selected transcription method. No partials, no streaming.

->
4

Dictionary words only

Replace known dictionary words. Never re-write the text. Never change the meaning. Finish auto-uses the text; Hold shows it for review.

The two rules that matter most. (a) Send the WHOLE utterance to the Gateway after the user stops - never transcribe live in pieces. (b) After transcription, the ONLY transformation allowed is swapping known dictionary words (for example a mis-heard product name). The wording, order, grammar, and meaning of the user's speech must survive untouched.

1a. The Audio Completeness Gate (non-negotiable)

We ALWAYS capture and upload ALL sound. This is a GATE, not best-effort.

Transcription must be impossible on partial audio. Every byte the microphone produced between Start and Finish/Hold must reach the Gateway and be verified before a single word is transcribed. There is no "transcribe what we got" path. If the audio is incomplete, the system blocks and retries - it never silently transcribes a truncated clip.

The gate has three checkpoints, all required:

  1. Capture seal (client). On Finish/Hold, the recorder's final buffer is flushed and the finalized file(s) read in full. The client computes total bytes, duration, and a per-segment SHA256, and writes a sealed capture manifest. An empty capture fails loud - never an empty transcript.
  2. Durable upload (transport). Every segment is uploaded with its SHA256; the complete call carries the manifest (expected segment count + total bytes + hashes). The client retries upload and complete until the Gateway acknowledges. App-kill between capture and acknowledgement cannot lose audio - the recording stays queued on disk and resumes.
  3. Acceptance gate (Gateway) - THE gate. Before transcribing, the Gateway verifies every index 0..N-1 is present and contiguous, each stored segment's SHA256 matches the manifest, and the total byte count matches. Only an all-pass result advances to transcription. Any miss returns an "incomplete" response listing the missing/bad segments; the client re-sends them. No partial audio is ever transcribed.

Consistent with the no-fallback rule: a missing buffer is fixed (re-sent), not worked around. "Transcribe whatever arrived" is exactly the fallback we are removing.

Where today's code violates the gate. Desktop dictation streams audio live and, if the connection ends before a clean stop, it recovers whatever bytes arrived and transcribes them when they exceed a byte threshold (DictationEndpoint.cs:255-265, 317+, TryRecoverOrDiscard / MinRecoverableAudioBytes). That is best-effort truncation, the direct opposite of the gate. The phone paths are close to compliant (per-chunk SHA256, complete-call retried until acknowledged), but the gate is not yet a uniform, enforced invariant across all surfaces.

3. Verdict: is the desktop path changing the text?

Yes - in two ways, and one of them is the serious one.
  1. The transcription model itself is not verbatim (desktop dictation, BYO mode). Desktop dictation streams to OpenAI Realtime with gpt-4o-transcribe (DictationEndpoint.cs:179-181). That is a language model, not a literal speech-to-text engine like Whisper - it tends to tidy disfluencies and lightly normalize phrasing as it transcribes. The text can change at the transcription layer, before any post-processing runs. Moving to "send the whole clip, batch-transcribe with Whisper" produces a more literal transcript.
  2. The voice paths run a free-text LLM "cleanup" whose output replaces your words. Desktop voice mode and phone push-to-talk call WingmanService.CleanVoiceTranscriptAsync (VoiceService.cs:91 and :169). Its prompt says "return verbatim", but it returns the model's own free text - the engine does not check the output against the source. An LLM told to "clean up" can and sometimes will re-word, fix grammar, or shift meaning. This is the path that violates rule 4, and it is the one to remove.
What is already correct and must be kept. The dictionary corrector (CleanupOrchestrator + TranscriptEditEngine, used by desktop dictation, phone recorder, and as the final step of the voice paths) is safe by construction: the language model only proposes find/replace pairs, and a deterministic engine rejects any edit whose "find" text is not literally present in the raw transcript and whose "replace" is not a known dictionary term. The user's transcript never round-trips through the model, so it cannot be reworded. This is the model for "step 4" everywhere.

One decision needed before G1/G4. The current dictionary corrector uses a language model to propose the find/replace pairs (then validates them deterministically). A fully offline, pure find-and-replace dictionary already exists in the Vosk path. Do we keep the model-proposes / engine-validates design (smarter at catching mis-hearings, still cannot change meaning), or switch to pure deterministic replacement (no model at all in step 4)? Both satisfy "never change meaning". This choice affects G4 and G8.

5. Target architecture

A

Shared capture

One capture layer. Records locally, shows waveform + timer, flushes the final buffer on Finish/Hold. No text produced here.

->
B

Gateway transcribe

Whole audio uploaded and verified by the completeness gate (section 1a). Only when every segment is present and hash-checked does the Gateway resolve the selected method and transcribe once. Returns raw text.

->
C

Dictionary corrector

Validated find/replace against the shared glossary. Deterministic apply. No re-wording. Returns final text + which words changed.

->
D

Terminal action

Finish: insert / use the text automatically. Hold: show it for review and editing first.

Every surface differs ONLY in step A's host (Avalonia, web, Android) and step D's destination (a session prompt, a voice turn, a recording transcript). Steps B and C are identical code for all.
