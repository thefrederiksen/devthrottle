4. The two fixable gaps (not "impossible")

Gap A - cc-* tools not runnable by bare name in Git Bash (fix applied to test machine; ship a one-liner)

The cc-* tools are installed as bin\<name>.cmd shims. Agents drive their shell through Git Bash, which does not resolve .cmd by bare name - so an agent running cc-ask got command not found. This is the exact friction that started this whole effort. I added bare-name shell shims to the bin directory (a tiny #!/bin/sh; exec ...exe "$@" per tool), after which Pi's cc-ask worked. The permanent fix is to have the setup engine (PythonToolsInstaller.WriteShims) write these bare-name shims alongside the .cmd ones (and remove them in RemoveManagedShims). One contained change; not yet committed.

Gap B - clean answers when the ask TARGET is a heavy-repaint TUI (Pi) (server fix landed; needs a follow-up for Pi)

The fleet ask captured the answer by scraping the target's terminal buffer, which is noisy for heavy-repaint TUIs (Pi, Codex). G6 (committed) now reads the answer from the target's transcript (clean parsed text) for any agent with a reader, falling back to the buffer only when the transcript yields nothing. This makes asking Claude/Codex crisp and removes the cc-ask print-crash risk. Asking Pi is still flaky: Pi tends to echo the @.temp delivery reference rather than answer, and the transcript read fell back to the buffer within its retry window. Follow-ups: deliver short fleet questions to Pi inline (not via the @-temp-file path) and widen the transcript-flush retry for Pi.
