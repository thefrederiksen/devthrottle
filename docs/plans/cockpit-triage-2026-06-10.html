The headline finding: Cockpit voice recording is still broken from a non-Gateway machine (this laptop), and it is not a dictation bug — it is one architectural gap that also breaks the live terminal cross-machine. The per-session WebSocket legs connect browser → Director directly using each Director's own advertised address; when that address is loopback (ws://127.0.0.1:7887) a remote browser hits its own loopback and the socket never connects. Fixing that one backbone (route the WS legs through the Gateway, same-origin) makes voice, terminal, and every future per-session WS feature work from any machine. The work is organised into two epics below.

Update — 2026-06-10 (later that day): the backbone #268 has since landed and merged (PRs #283 + #289) and is flow:done. The per-session WS legs now route through the Gateway, verified cross-machine (SORENLAPTOP → SOREN_NORTH: live dictation + terminal, and the offline-owner 503 / unknown-session 404 path). EPIC Remote-first reliability #279

Make the Cockpit fully usable from any non-Gateway machine. Reliability only — visual redesign is explicitly deferred. view epic →

One root cause, not a pile of bugs. The Gateway already reverse-proxies HTTP /sessions/{sid}/* and the Blazor circuit WS to the owning Director, but the two raw per-session WebSocket legs are not proxied: dictation (DictateWsUrl(Selected.TailnetEndpoint)) and terminal (TerminalPane.BuildWsUrl). Both target the Director's own address, so a remote browser cannot reach them. Route them through the Gateway and the browser only ever talks same-origin.
