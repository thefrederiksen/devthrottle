Fleet Preamble: All-Agents Implementation Plan

CC Director / DevThrottle  |  Goal: every supported agent has a plugin, a driver, and launch-time fleet-awareness injection - or an explicit, honest declaration that it cannot.

Two agents are done (Claude Code, Codex). This plan covers the remaining six, plus the cross-cutting work needed so that "every agent knows the fleet the moment it starts" is a property of the whole fleet, not a per-agent accident - and so that an agent that genuinely cannot do it says so rather than failing silently.

What "done" means for one agent. At launch, the agent receives a short preamble - its own session id and the cc-* fleet commands - injected into its context with no skill lookup, and that preamble is re-injected after the agent clears or compacts its context. The shared piece (already built) is the Control API endpoint GET /sessions/{id}/fleet-preamble; each agent differs only in how the text is surfaced and whether the agent emits a reset signal.

1. Executive summary

CC Director runs nine agent kinds. Eight have a first-class plugin and an agent launch class; five have a bespoke protocol driver and three share a GenericDriver; the ninth kind, RawCli, is a deliberate "run any command" case with no plugin. Today the launch-time fleet preamble is wired for 2 of 8 agents (Claude, Codex).

  • Plugin coverage: 8 of 8 real agents have a plugin. RawCli has none by design - a decision to confirm (section 9).
  • Driver coverage: every kind resolves to a driver, but Gemini, OpenCode and Grok use the shared GenericDriver (minimal capabilities). Giving them bespoke drivers is part of this plan.
  • Fleet preamble: 2 wired, 5 to wire, 1 to declare unsupported. The agents split across four working strategies plus a "cannot" case.
  • A bug found while wiring Codex (programmatic prompts dropped the Enter) likely affects other agents driven through the same path - audited as a cross-cutting item.

6. The five injection strategies

Every agent falls into one family. The work per agent is "implement this agent's family", reusing the shared preamble endpoint. The fifth case is the honest "cannot".

A - Native lifecycle hook (best)

A config file registers a command we own; the command prints the preamble as the agent's additionalContext. The agent re-fires the hook on clear and compact, so re-injection is automatic and free. Agents: Claude (done), Codex (done), Gemini (after upgrade), Cursor.

Pattern proven twice: ClaudeHookInstaller (via --settings) and CodexHookInstaller (merge into ~/.codex/hooks.json + --dangerously-bypass-hook-trust).

B - Out-of-process event bus

We subscribe to the agent's event stream over a local socket and push the preamble when we see a new / cleared / compacted session. Richer than a hook for observing resets. Agents: OpenCode (opencode serve + GET /event SSE).

C - In-process extension or plugin we author

We ship a small extension loaded into the agent that handles its lifecycle events and re-injects. Agents: Pi (TypeScript extension with session_start / before_agent_start / compact hooks). OpenCode could also use this as an alternative to B.

D - Instruction file the agent re-reads every prompt (passive, self-healing)

We write the preamble into a file the agent reads on every prompt (AGENTS.md, CLAUDE.local.md, copilot-instructions.md, ...). No event needed; it survives clear and compact because it is re-read, not stored in the conversation. Costs: it lives in a file (some pollute the repo), and there is no reset signal. Agents: Grok (hook output is ignored, so this is the only route), Copilot (CLI hook output ignored), and a universal fallback.

None - cannot inject

The agent is an arbitrary command with no instruction-file, hook, or event contract. Agents: Raw CLI. The plan makes this an explicit, declared state, not a silent gap (section 9).

7. Cross-cutting work (do this first)

Three pieces of shared scaffolding make the per-agent work small, consistent, and honest.

7.1 Declare the strategy on every plugin new

Add a plugin-owned metadata record, for example AgentPluginFleetMetadata(FleetPreambleStrategy Strategy, FleetPreambleStatus Status, string Note), exposed on IAgentPlugin. This is where each agent says how it injects, or that it cannot.

  • FleetPreambleStrategy: NativeHook, EventBus, Extension, InstructionFile, None.
  • FleetPreambleStatus: Wired, Planned, Unsupported.
  • Surfaces in the Control API session DTO and the desktop UI, so a session can be badged "fleet-aware" or "no fleet awareness (this agent cannot)". This is the answer to "how do we say they can't do it".
  • Unit test: every built-in plugin returns a non-default strategy; None is allowed only with a written reason.

7.2 Extract the echo-verified submit into a shared helper new

Wiring Codex exposed that the blind programmatic submit (type text, wait 50 ms, one Enter) is dropped by a repainting composer, parking the prompt unsubmitted. Claude and Codex now type, wait for the composer to echo the text, then press Enter separately. Extract this into TerminalSubmit.EchoVerifiedAsync(...) and audit each driver:

  • Gemini, OpenCode, Grok, Copilot, Cursor, Pi: confirm whether their composers drop the blind Enter when driven over the REST prompt path; adopt the shared echo-verified submit where they do.
  • This is independent of the preamble but blocks reliable automated turns (and therefore hook firing for family A agents, as Codex showed).

7.3 Give the three generic-driver agents bespoke drivers new

Gemini, OpenCode and Grok currently share GenericDriver (only Cancel + Interrupt). To declare honest capabilities (each has its own clear/compact, transcript, model flag) and to host the echo-verified submit and family-specific behaviour, give each a bespoke driver: GeminiDriver, OpenCodeDriver, GrokDriver. The plugin already owns the driver, so this is a contained change per agent.

9. Handling agents that cannot inject

The user requirement: if an agent cannot do it, we must say so. Two sub-cases.

  • Raw CLI (structurally cannot): mark FleetPreambleStrategy.None, Status.Unsupported, with the reason "arbitrary command - no context-injection contract". The environment still carries CC_SESSION_ID / CC_DIRECTOR_API and the cc-* tools still work, so a human at the terminal can run cc-whoami; we just cannot pre-load the model's context.
  • Degraded agents (partially can): e.g. Grok and Copilot inject at launch via a file but get no reset signal; record Status.Wired with a note "re-injection is passive (file re-read), no reset event". The matrix and the session badge tell the truth.
Surfacing it. The FleetPreambleStatus + note from 7.1 flow into the session DTO and the desktop session card, so every session visibly states whether it is fleet-aware and, if not, why. No silent gaps.

Plugin for Raw CLI? Today RawCli has no plugin (it is the explicit "Custom" case). Two options: (a) leave it plugin-less and special-cased, adding only the declared unsupported state where the UI reads it; (b) add a thin RawCliAgentPlugin so "every kind resolves to a plugin" holds uniformly and the strategy declaration lives in one place. Recommendation: (b) - a minimal plugin that declares Strategy.None - because it makes the invariant "every agent has a plugin" literally true and removes the special case.

10. Sequencing and roadmap

  1. Cross-cutting first (7.1, 7.2, 7.3 partial): the FleetPreambleStrategy declaration + the shared echo-verified submit helper. Small, unblocks everything, and immediately lets the UI tell the truth about all agents.
  2. Codex finish: declare TranscriptRead (cc-ask). Tiny.
  3. Grok (D): installed and understood; instruction-file flavour. No upgrade needed.
  4. Copilot (D): instruction-file; same shape as Grok. Confirm install.
  5. Gemini (A+D): after upgrading the binary; high value (hook parity with Claude/Codex).
  6. Pi (C): author the extension; self-contained.
  7. OpenCode (B): the event-bus transport is the most new infrastructure; do after the simpler wins.
  8. Cursor (A): after installing cursor-agent; verify the clear re-fire.
  9. Raw CLI: add the minimal plugin + declared unsupported state (can land any time with 7.1).

Rationale: ship the cheap, already-installed, already-understood agents first (Grok, Copilot), do the high-parity hook agents when their binaries are ready (Gemini, Cursor), and leave the most new infrastructure (OpenCode event bus, Pi extension) for last.

11. Open questions and risks

  • Per-session id in instruction files (Grok, Copilot): the preamble carries the session id, but instruction files are static. Resolve per agent: per-session file written at launch vs commands-only + cc-whoami. Recommended: per-session file for full parity.
  • Global config edits: the hook/instruction installers write the user's own agent config (Codex hooks.json, Gemini settings.json, etc.). For a shipped product, prefer managed/elevated config paths where they exist, and always merge (never clobber) the user's content.
  • Binary churn: these CLIs are young and changing. Every field name (hook output shapes, event names) needs a live check against the installed build before depending on it; the agent-expert skill records the per-agent verification checklists.
  • OpenCode transport: whether a Director-driven TUI exposes a reachable event server is unconfirmed; may require running serve ourselves.
  • Submit audit: the echo-verified submit fixed Codex; assume the other agents may need it too and budget a live check per agent.
  • Cursor / OpenCode / Copilot / Pi are not confirmed installed here, so their plans rest on documentation until a binary is available.
