Wingman: Terminal State Detector + Human Feedback

End-to-end implementation report · generated 2026-05-24 · verified against a live Director (slot 4, port 7881)

One line: the Wingman now reads only the terminal to decide what a session is doing — no Claude Code hooks, no shared .jsonl files, so it works for any agent and cannot cross-contaminate. When it gets a state wrong, you correct it from the phone in two taps and the correction is filed to a GitHub issue with full context.

What was built

StepStatusWhat it does
1. Terminal drives stateDONE A TerminalStateDetector derives ActivityState and turn boundaries from the session's own PTY byte stream. Output flowing → WORKING; quiet 5s → the LLM judge reads the screen and labels it READY / NEEDS YOU / PERMISSION. Process exit → ENDED.
2. Stop relying on hooksDONE + deferred cleanup Hooks are now inert — the new build ignores them for state and does not install them in terminal mode. Physical deletion of the hook code is deferred (see why).
3. Human vote → GitHubDONE A "Status wrong? Tell us" panel on the mobile session view. Pick the correct state + a note → saved locally and appended to one GitHub tracking issue with the terminal tail.

Why terminal-only (the architecture)

Before: hooks

Claude Code event
  → spawn powershell relay
  → write JSON file to a
     SHARED events folder
  → every Director's watcher
     sees every file
  → route by ClaudeSessionId
  → Session state
  • Claude-Code-only (no Pi/Gemini)
  • broadcast + filter by an id that rotates on /clear — the cross-contamination bug
  • breaks if Claude changes hooks

After: terminal

claude.exe stdout
  →(in-process pipe)
  → this Session's own Buffer
  → TerminalStateDetector
  → Session state
  • agent-agnostic — any TUI
  • per-session, no id lookup → cannot cross-contaminate
  • the model reads the screen like you do; survives TUI changes

The LLM judge earns its keep

When the terminal goes quiet, the judge reads the screen and cites the actual evidence:

[TerminalStateDetector] … terminal=QUIET (5.0s no output) | hook=Working
[TerminalStateDetector] … LLM verdict=waiting_for_input
   ("The prompt (❯) is displayed and ready for the next command;
     no spinner or active operation visible at the bottom.")

[TerminalStateDetector] … LLM verdict=working
   ("Elapsed-time counter 'Brewed for 2s' and active hook execution
     ('running sp hooks... 0/3') with esc to interrupt footer
     indicates active processing.")

Note the second line: the byte gate flipped to active and the judge correctly distinguished real work ("Brewed for 2s", "esc to interrupt") from an idle prompt. This is exactly the case the old hooks got stuck on.

Step 3: how you vote (and that it works)

At the bottom of every session view there is a "Status wrong? Tell us" panel. Three steps:

  1. It shows what the detector currently thinks (Detected: WORKING (working)).
  2. Pick what it should have been from the dropdown (WORKING / NEEDS YOU / WAITING FOR PERMISSION / READY / ENDED).
  3. Optionally type why, then tap Send feedback.

The vote panel expanded: detected state, the "it should have been" picker set to READY, and a free-text reason.

What's working, reversible, and deferred

Working now

  • Terminal-only state detection drives the live status (verified above).
  • Turn summaries built from the terminal, not the shared .jsonl — no cross-contamination.
  • Mobile vote → local + GitHub #131 (verified).
  • Builds clean (0 warnings); the terminal-detector, Wingman, buffer, ActivityState, TurnAccumulator and EventRouter test suites are green (218 relevant tests).

Reversible (no rebuild)

  • CC_DIRECTOR_TERMINAL_STATE=0 — fall back to the old hook path.
  • CC_DIRECTOR_TERMSTATE_LLM=0 — keep terminal mode but skip the LLM judge (byte-activity only).

Deferred on purpose (step 2 cleanup)

  • Physically deleting the hook code (EventRouter, PipeMessage, relay, file watcher) is staged, not done, for two concrete reasons:
    • The desktop app has a live hook-event viewer UI bound to EventRouter — rewriting that is Avalonia UI I can't run-verify here.
    • ~/.claude/settings.json hooks are shared; auto-uninstalling them would break your other live Directors mid-session.
    The new build already ignores hooks (terminal is authoritative) and stops installing them, so the reliance is gone. Final deletion should happen once all Directors are on this build.
