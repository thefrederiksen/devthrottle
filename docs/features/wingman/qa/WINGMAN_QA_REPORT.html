3. Goal Management - how it works

Goal Management is the first expanded responsibility. It is strictly observational in this slice: the Wingman reports a verdict but never changes the status color or acts on the session.

Flow

  • User sets a goal via the UI or POST /wingman/goal.
  • Setting a goal triggers an immediate assessment (AssessGoalNowAsync).
  • After every completed turn, TurnSummaryCache re-assesses the goal against the recent turn summaries.
  • A one-shot Haiku call returns one of: on_track complete drifting unknown.
  • The verdict + reason are stored on the session and surfaced in GET /wingman and the Session View.

Safety properties

  • On any failure (no CLI, parse error, timeout) the verdict is unknown with a reason - a verdict is never fabricated.
  • Setting a new goal resets the verdict to unknown so a stale verdict cannot linger.
  • No status-color change, no message sent to the session. Read-only.
  • The riskier "act on your behalf" slices (auto-answering prompts) are deliberately NOT in this build.

7. Known limitations & next steps

  • SCOPE Goal Management UI is wired in the web Session View only. The desktop Avalonia WingmanView does not yet show the goal (needs SessionViewModel plumbing).
  • DEFERRED The "act on your behalf" responsibilities (Smart Decision Filtering / auto-answering trivial prompts, Interview-style questioning) are intentionally NOT built. They cross the read-only boundary and need an explicit autonomy guardrail + kill switch before implementation.
