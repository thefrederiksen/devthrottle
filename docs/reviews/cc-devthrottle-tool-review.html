cc-devthrottle Tool Review

Readiness and code-review report for the shipped Fleet tool. The review compares the current Python CLI to the user-facing Director UI, Gateway Cockpit, Control API, Gateway API, docs, shipped manifest, and tests. The goal is to turn cc-devthrottle into the one reliable command surface for DevThrottle automation before reviewing the remaining tools.

Executive Read

What Works

  • The source CLI cleanly groups Fleet sessions/messages, settings, Gateway schedules, and setup under one Typer app.
  • Gateway cron routes are mostly wrapped: list, get, runs, create, run, enable, disable, delete, endpoint.
  • Settings read/write was successfully folded into cc-devthrottle settings.
  • Focused Python schedule/settings tests pass in the current tree.

Readiness Judgment

  • Not ready to call complete as the one DevThrottle tool yet.
  • It is not installed on this live session's PATH even though older fleet aliases are.
  • The advertised agent action list is incomplete and diverges from the actual CLI.
  • Full UI/Gateway/Director parity requires new command groups and contract tests.

High Settings writes bypass the live Control API path and lack Settings UI parity. tools/cc-devthrottle/src/settings_ops.py:59 mutates config objects and calls config.save() at settings_ops.py:90. The live Director API uses PUT /settings and re-applies Gateway changes through src/CcDirector.ControlApi/SettingsEndpoint.cs:39 and ControlApiHost.cs:653. The Agents settings API exposes full CRUD/detect/quick-check/catalog at src/CcDirector.ControlApi/AgentsEndpoint.cs:25-44. Changing gateway.url from the CLI can leave a running Director stale until restart, and the CLI cannot manage the Agents tab, detection wizard, quick checks, command-line previews, or catalog. Default cc-devthrottle settings to CC_DIRECTOR_API when inside a session, with --offline for direct file edits. Add structured settings agents, settings detect, settings test-gateway, and settings screenshots detect subcommands.
High Schedule command lacks full Cockpit parity because it cannot edit an existing job's fields. Cockpit's Schedule page uses edit/create modal behavior and calls UpdateCronJobAsync at src/CcDirector.Cockpit/Components/Pages/Schedule.razor:530. Gateway exposes PUT /cron/jobs/{id} at src/CcDirector.Gateway/Api/CronJobEndpoints.cs:64. The CLI has enable/disable as narrow PUT wrappers, but no schedule update command. Agents can create or toggle jobs but cannot safely modify target machine, repo, prompt/worklist, cron, timezone, or notification policy after creation. Add cc-devthrottle schedule update <id> with the same fields as create, plus optional patch-style flags that preserve unspecified values.
High Director session UI controls are mostly not wrapped. Control API supports hold (ControlEndpoints.cs:735), wingman toggle (763), prompt (1890), interrupt (2044), escape (2068), upload image (2203), delete session (2893), history (SessionHistoryEndpoint.cs:25), usage (SessionUsageEndpoint.cs:19), recap (ControlEndpoints.cs:1309, 1346), and queue routes (ControlEndpoints.cs:1954, 1968, 2006). The CLI can list/rename/spawn/message/ask, but it cannot drive many operations users can perform from the Director or Cockpit UI. Add a cc-devthrottle session expansion: prompt, interrupt, escape, kill, hold, wingman, history, usage, recap, queue, upload-image, and screenshot operations.
Medium Gateway/Cockpit pages expose major automation surfaces that are missing from the CLI. Work lists are exposed by src/CcDirector.Gateway/Api/WorkListEndpoints.cs:32-145 and run via WorkListRunnerEndpoints.cs:38. Account/telemetry/dictionary/transcripts/exes/wingman queue and wingman instructions are exposed in src/CcDirector.Cockpit/Services/GatewayClient.cs:176, 219, 252, 606, 666, 744, and src/CcDirector.Gateway/Api/WingmanInstructionsEndpoint.cs:29. If the standard is "everything available in the UI/Gateway/Director", cc-devthrottle is missing whole product domains. Define explicit groups: gateway account, gateway telemetry, gateway dictionary, gateway transcripts, gateway exes, gateway wingman, and worklist. If some are out of scope, document that boundary.

Recommended Implementation Path

  1. Fix install/runtime first: make cc-devthrottle present on PATH in a live Director session and add a release smoke test for it.
  2. Replace static action metadata with a declarative command registry and make actions --json complete.
  3. Switch settings writes to Control API by default and add settings agents parity subcommands.
  4. Add schedule update and full action metadata for all schedule commands.
  5. Add session control parity commands for prompt/interrupt/escape/kill/hold/wingman/history/usage/recap/queue.
  6. Add Gateway domain groups only after explicitly deciding that cc-devthrottle owns full Gateway/Cockpit coverage.
