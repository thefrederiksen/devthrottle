Overall Verdict

No blocking defects found. The core mapping behavior is coherent across desktop and Cockpit, the desktop handler is wired after persisted state is applied, the missing-transcript cache path is handled, and the filtered-empty copy is present. I would still address the warnings below before calling the feature polished, especially the Cockpit scroll-listener lifecycle regression that the new filters make easy to trigger.

What Is Done Well

  • Desktop handler sequencing is correct. The checkboxes are populated before IsCheckedChanged handlers are subscribed, which avoids overwriting persisted config during construction.
  • The desktop missing-transcript path clears cached history. This prevents a later filter toggle from resurrecting a transcript after its path disappears.
  • The filtered-empty state is implemented on both surfaces. A transcript with messages hidden by filters now reports "No messages match the current filters."
  • Desktop and Cockpit mapping are mostly kept in parity. Text is preserved, tool calls/thinking/results are filtered per part, and pure tool-result bubbles disappear when results are hidden.
  • Persistence defaults are conservative. Desktop config defaults to show-all, save uses merge-patch, and malformed typed keys in HistoryFilterConfig.Get throw rather than silently coercing.
  • The Cockpit render signature includes the filter. A toggle re-renders cached history without waiting for a transcript change.

Test Gaps / Follow-ups

  • Add a Cockpit component or browser-level test for the empty-to-nonempty filter flow to catch the scroll-listener lifecycle issue.
  • Add UI-level desktop coverage that proves applying saved checkbox state during construction does not call Save and overwrite config.json.
  • Add Cockpit localStorage coverage for load, persist, invalid stored values, and the first-render re-map path.
  • Add component-level empty-state tests on both surfaces for "messages exist but filters hide all" versus "no history yet."
  • Add coverage for assistant-side ToolResult parts with results hidden, not only user-role tool-result bubbles.
  • I did not run the test suite during this review because the request was review-only and constrained commands to read-only inspection plus the single HTML artifact.
