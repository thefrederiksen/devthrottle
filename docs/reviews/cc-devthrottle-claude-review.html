Independent Code Review — cc-devthrottle

Unified DevThrottle command-line surface (fleet, sessions, messages, settings, Gateway schedules, setup)
Scope: tools/cc-devthrottle, tools/cc_shared/director.py, Control API + Gateway endpoints, registry/manifest, settings/setup surfaces, UI overlap
Reviewer: Claude (session 3429220a, SOREN_NORTH) Date: 2026-06-27 Tool version: 1.3.0 Repo: D:\ReposFred\devthrottle @ main

This review assesses the cc-devthrottle CLI for missing coverage, inconsistencies, bugs, install/path risk, incomplete action metadata, and gaps against what the Director, Gateway, and UI can do. The good news first: the runtime wire contracts are sound. Every fleet/session Control API call and every Gateway cron call the tool makes was traced to a matching backend route with matching request/response field names (camelCase or PascalCase tolerated via director.field()). The serious problems are concentrated in the installer (which does not work against the real release at all), in drift from the tool registry, and in incomplete agent-facing action metadata.

Critical

Critical C1 — setup install/update/repair downloads nothing against the real release (asset-name mismatch)

The installer asks GitHub for the latest release and looks up assets by name. It expects a per-tool asset <tool>.exe for every Python tool, <tool>.zip for every Node/.NET tool, and the literal name cc-director.exe for the app. None of those names exist in the published release.

What the installer looks for
for tool in PYTHON_TOOLS:
    asset_name = f"{tool}.exe"        # e.g. "cc-gmail.exe"  -> never present
    if asset_name not in assets: skipped += 1; continue
...
if "cc-director.exe" in assets:       # literal "cc-director.exe" -> never present
    _download_file(assets["cc-director.exe"], ...)
tools/cc-devthrottle/src/setup_ops.py:181-184, 215-240
What the release actually publishes
cc-director-win-x64.exe          (the app  - NOT "cc-director.exe")
cc-python-win-x64.zip            (shared CPython runtime)
cc-tools-pyenv-win-x64.zip       (wheelhouse + requirements.lock + tools-manifest.json)
devthrottle-gateway-win-x64.exe  cc-launcher-win-x64.exe
devthrottle-cockpit-win-x64.zip  devthrottle-setup-win-x64.exe  ...
.github/workflows/release.yml:404-426 (Collect release files) and :437-451 (required assets)

Consequences, against a genuine releases/latest:

  • Every Python tool is skipped (no <tool>.exe asset exists; Python ships as one shared-venv bundle).
  • Every Node/.NET tool is skipped (no <tool>.zip per-tool assets exist).
  • The app is never downloaded — the asset is cc-director-win-x64.exe, so the "cc-director.exe" in assets check is always false.
  • The installer has no concept of the Python bundle (runtime zip + wheelhouse + venv creation), which is the only way Python tools actually ship.

Net effect: cc-devthrottle setup install prints "Installation complete." while having installed nothing. This is a silent no-op masquerading as success — the worst failure mode under the project's "no fallback / fail explicitly" rule. The command is advertised to agents in the fleet preamble (cc-devthrottle setup status) and exposed as the setup-install action, so it will be invoked.

Note: the real, working installer is a separate component (see H1). The pragmatic fix may be to make these subcommands defer to / delegate the setup engine rather than re-implement download logic that is already out of step with the release format.

High

High H1 — The installer duplicates and contradicts the canonical setup engine

The product already has a dedicated installer pipeline: the setup wizard (devthrottle-setup-win-x64.exe), the setup CLI (devthrottle-setup-cli-win-x64.exe), and a ComponentRegistry that knows the real asset names (ComponentRegistry.Gateway.WindowsAsset = devthrottle-gateway-win-x64.exe, etc.). Tool directories cc-director-setup, cc-director-setup-cli, and cc-director-setup-engine exist for exactly this.

setup_ops.py is a second, parallel installer with its own hardcoded asset names, its own PATH logic, and its own component model. It is unaware of the Gateway, Launcher, Cockpit, setup-CLI, and the Python shared-venv bundle. Two installers with divergent asset-name expectations is a standing maintenance and correctness hazard: a release-format change (as already happened, per C1) silently breaks one of them.

Evidence
tools/cc-devthrottle/src/setup_ops.py:130-345 (DevThrottleInstaller) vs .github/workflows/release.yml:166-345 (gateway/launcher/cockpit/setup-cli/python-bundle); ComponentRegistry asset names referenced in release.yml:166-168, 202-204, 237-239

High H2 — Installer tool list ignores registry.json, the documented single source of truth

tools/registry.json states plainly: "Single source of truth for all tools… A tool ships in the installed product ONLY when "ship": true." The shipped allowlist is exactly nine tools: cc-pdf, cc-html, cc-word, cc-gmail, cc-outlook, cc-playwright, cc-vault, cc-comm-queue, cc-devthrottle.

The installer's hardcoded PYTHON_TOOLS / NODE_TOOLS / DOTNET_TOOLS lists ignore that flag entirely. They enumerate ~32 tools, most marked ship: false (e.g. cc-crawl4ai, cc-twitter, cc-computer, cc-browser), and omit a tool that IS shipped: cc-comm-queue (ship: true) is absent from the list.

Missing (shipped)cc-comm-queue is ship: true but not in PYTHON_TOOLS
Over-broad~23 ship: false tools are listed and would be downloaded in alpha mode

tools/cc-devthrottle/src/setup_ops.py:24-63 vs tools/registry.json:3 (ship rule), :246-252 (cc-comm-queue ship:true), ship flags throughout

The list should be derived from registry.json (filter ship == true) rather than hand-maintained.

High H3 — actions metadata is incomplete and partly incorrect

The actions --json output is the contract agents use to discover how to call the tool without hardcoding syntax (it is advertised first in the fleet preamble). It must mirror the real command surface. It does not.

Missing actions (commands that exist but are not advertised)

  • settings list, settings path
  • schedule get, schedule runs, schedule enable, schedule disable, schedule delete, schedule endpoint
  • setup update, setup repair, setup doctor
  • the top-level actions command itself

Incorrect / stale action definitions

  • schedule-create is wrong. Metadata lists args cron_or_at, tz, seed_or_worklist and a command string --cron <expr> --tz <tz> --seed <prompt>. The real command uses distinct flags --at/--cron (mutually exclusive), --seed/--worklist (mutually exclusive), plus --notify-on and --notify-webhook. An agent following the metadata would build an invalid command.
  • session-spawn advertises only the repo arg, omitting --agent, --prompt, --name, --type, --command, --command-args.
  • message-ask omits --timeout-ms.
Evidence
tools/cc-devthrottle/src/cli.py:50-182 (_ACTIONS) vs the actual commands at cli.py:247-270 (spawn), :282-291 (ask), :388-423 (schedule create), :334-347 (settings list/path), :370-458 (schedule get/runs/enable/disable/delete/endpoint), :475-492 (setup update/repair/doctor)
The metadata is hand-maintained and will keep drifting. Consider generating _ACTIONS from the Typer command tree, or adding a test that asserts every registered command has a corresponding action entry.

Medium

Medium M1 — director.py docstring contradicts how schedules actually talk to the Gateway

The shared helper states, as a design invariant: "cc-devthrottle only ever calls its own Director. The Director relays to the Gateway on its behalf, so it never needs the Gateway URL or the fleet token. The fleet token stays on the Director… The fleet token is deliberately never read here."

That is true for fleet/session commands, but schedule_ops.py does the opposite: it reads config.gateway.url and config.gateway.token from config.json and calls the Gateway directly (default http://127.0.0.1:7878). There is no Director relay for cron (confirmed: cron routes live only in CcDirector.Gateway, none in CcDirector.ControlApi). So the stated trust model ("never reads a token, never knows the Gateway URL") is false for the entire schedule surface.

This is mainly a documentation/architecture inconsistency rather than a runtime bug — the schedule path works — but the invariant as written is misleading and would steer a future maintainer wrong. Either route schedules through the Director too, or correct the docstring to scope it to fleet/session calls and acknowledge the direct-to-Gateway schedule path.

Evidence
tools/cc_shared/director.py:1-10, 38-52 vs tools/cc-devthrottle/src/schedule_ops.py:21,44-66 (reads gateway.url + gateway.token, calls Gateway directly)

Medium M2 — The gateway section is write-only via the CLI until it already has a value

CCDirectorConfig.to_dict() only emits the gateway block when it is non-empty (by design, to avoid injecting empty sections). But settings read paths are all built on to_dict():

  • settings get gateway.url → "Unknown key" (not in the flattened dict).
  • settings list → no gateway.* rows.
  • settings show gateway → "Unknown section: gateway".

Yet settings set gateway.url <url> works (it goes through getattr(config, "gateway"), which always exists). The result is a confusing asymmetry precisely around the setting a user most needs to discover — the Gateway URL/token that makes schedule commands reach a remote Gateway. A fresh install cannot read or list the key it is supposed to set.

Evidence
tools/cc_shared/config.py:497-501 (gateway emitted only when not is_empty) and :212-213 (is_empty); tools/cc-devthrottle/src/settings_ops.py:28-57 (read paths via to_dict) vs :59-91 (set_value via getattr)

Medium M3 — No tests for the highest-risk modules

The project standard is "all public methods need unit tests." Tests exist for schedule_ops and settings_ops (good, and the notify-flag tests are solid). But there are no tests at all for:

  • session_ops.py — the core fleet messaging surface (list, whoami, rename, spawn, send, ask, selftest, target resolution).
  • setup_ops.py — the installer. A single test asserting expected release asset names would have caught C1.
  • cc_shared/director.py — resolve_target ambiguity/prefix/exact-name logic, field() casing.
  • the cli.py _ACTIONS metadata (no test ties it to the real command tree; would have caught H3).
Evidence
tools/cc-devthrottle/tests/ contains only test_schedule_ops.py, test_settings_ops.py, test_schedule_cli_notify.py

Medium M4 — No schedule update command (coverage gap vs the UI)

The CLI can create, list, get, run, enable, disable, and delete schedules, but cannot edit one. Enable/disable use a get-then-PUT round-trip (set_enabled), so the underlying PUT /cron/jobs/{id} is available and the Gateway supports full edits — the CLI simply does not expose them. To change a cron expression, seed, machine, time zone, or notify policy, a user must delete and recreate, which discards run history. The Avalonia SchedulerView and Cockpit Schedule.razor both allow editing, so the CLI is behind the UI here.

Evidence
tools/cc-devthrottle/src/cli.py:362-458 (no update command); schedule_ops.py:120-121,133-136 (update_job exists, used only for enable/disable); src/CcDirector.Avalonia/Controls/SchedulerView.axaml, src/CcDirector.Cockpit/Components/Pages/Schedule.razor

Low

Low L1 — settings set coercion can raise a raw exception; bool coercion is silent

set_value coerces by the current value's type: int(value) / float(value) will raise an uncaught ValueError (stack trace to the agent, violating the project's "fail with a clear message" rule) if a non-numeric string is passed for a numeric field. Bool coercion (value.lower() in ("true","1","yes")) silently maps any unrecognized string to False — e.g. settings set ...enabled maybe quietly disables it. Currently latent (no reachable scalar int/float key in the schema), but it is a trap waiting for the first one.

Evidence
tools/cc-devthrottle/src/settings_ops.py:80-91

Low L2 — settings set cannot set None-valued, list-valued, or new keys

set_value requires hasattr(obj, attr) and walks existing attributes, so it cannot set a value whose intermediate node is None, cannot edit list-typed settings (e.g. photos.sources), and cannot create a key that the dataclass does not already model. Acceptable for a guarded settings surface, but worth documenting as a deliberate limitation so callers do not assume arbitrary keys are writable.

Evidence
tools/cc-devthrottle/src/settings_ops.py:59-91

Low L3 — Three different HTTP stacks inside one tool

director.py and setup_ops.py use urllib; schedule_ops.py uses requests. Same tool, three code paths for auth headers, timeouts, and error mapping — and requests is pulled in only for the schedule surface. Standardizing on one client would simplify the tool and unify the (currently duplicated) connection-error / timeout / "extract server error message" logic.

Evidence
tools/cc_shared/director.py:12-15,65-90; tools/cc-devthrottle/src/setup_ops.py:7-9,94-123; tools/cc-devthrottle/src/schedule_ops.py:10,68-108

Low L4 — schedule create --worklist still sends an empty seed

When a work list is specified, the job body sets action.seed = seed or "" — an empty string is sent alongside workListName. The CLI already validates that exactly one of seed/worklist is given, so the empty seed is redundant payload. Harmless today (the Gateway accepts it), but cleaner to send None/omit it for the worklist branch.

Evidence
tools/cc-devthrottle/src/schedule_ops.py:290-318

Low L5 — cc-shared declared as a dependency that cannot be resolved from an index

pyproject.toml and requirements.txt both list cc-shared as a dependency, but it is not on any package index; at runtime the ops modules import it by inserting the tools directory onto sys.path. The declared dependency is therefore decorative and would break a clean pip install -r requirements.txt. Either package cc-shared properly (path/local dependency) or drop it from the requirement files and rely on the documented sys.path mechanism consistently.

Evidence
tools/cc-devthrottle/pyproject.toml:6-11; tools/cc-devthrottle/requirements.txt:4; sys.path insert in session_ops.py:17-21, schedule_ops.py:15-19, settings_ops.py:14-18

Low L6 — Minor UX / consistency notes

  • message-ask marked mutatesState: true — defensible (it prompts the target) but inconsistent with treating queries as read-only; worth a deliberate decision. cli.py:100-108
  • Schedule fallback port http://127.0.0.1:7878 is correct for a local Gateway (confirmed default), but a local-only Director with no Gateway will always fail there with a "not reachable" message; that is acceptable, just noting the assumed-local behavior. schedule_ops.py:21,44-47
  • Message send/ask have no desktop UI counterpart (only CLI). Not a defect, but the inter-session messaging surface is CLI-only while sessions/schedules/settings have UI. CcDirector.Avalonia, CcDirector.Cockpit
  • Selftest is Windows-only (hardcodes cmd + /k prompt); fine for the platform, but it will fail noisily if ever run elsewhere. session_ops.py:291-303,311-385

Recommendations (priority order)

  1. Fix or retire the installer (C1, H1). The strongest option is to make setup install/update/repair delegate to the existing setup engine / setup-CLI rather than re-download assets. If it must stay standalone, update it to the real asset names (cc-director-win-x64.exe) and teach it the Python shared-venv bundle model (cc-python-win-x64.zip + cc-tools-pyenv-win-x64.zip + venv creation). At minimum, do not print "complete" when nothing was installed.
  2. Derive the tool list from registry.json (H2). Filter ship == true and read type for build category; delete the hardcoded lists. This auto-fixes the missing cc-comm-queue and the over-broad set.
  3. Regenerate / validate actions metadata (H3). Generate _ACTIONS from the Typer command tree, or add a test asserting every command (and option) is represented; fix schedule-create, session-spawn, message-ask immediately.
  4. Correct the director.py invariant (M1) to scope the "only ever calls its own Director" claim to fleet/session calls, and document the direct-to-Gateway schedule path and its token use.
  5. Make the gateway section readable (M2) — surface gateway.* keys in list/get/show even when empty (or special-case the section), so the Gateway URL/token can be discovered before it is set.
  6. Add tests for session_ops, setup_ops, director, and the action metadata (M3) — a setup test pinned to expected asset names would have prevented C1.
  7. Add schedule update (M4) to reach parity with the UI and avoid delete/recreate losing run history.
  8. Address the Low items opportunistically: guard numeric coercion (L1), document settings-set limits (L2), unify on one HTTP client (L3), omit the empty seed for worklists (L4), and reconcile the cc-shared dependency declaration (L5).
