Result

PASS. Codex is no longer routed through the generic driver. It now has a plugin-owned driver, settings metadata, launch strategy, command presets, slash commands, context-clear control, and history support declaration.

Implemented Flow

1. SettingsAgent settings and the Control API catalog read plugin metadata: presets, model support, capabilities, and history support.
2. ConfigAgentToolConfig resolves presets and model flags through AgentPluginRegistry.
3. LaunchRuntime factories ask the plugin to create the CLI launch strategy. Codex creates CodexAgent.
4. ControlsSessions resolve the active driver through the plugin registry, so Codex exposes its own stop, interrupt, and clear-context behavior.

Important Notes

  • This slice proves the new architecture with Codex first. Other CLIs still use compatibility adapters unless they already had dedicated drivers.
  • Codex conversation history remains read through the existing SessionHistoryReader and CodexRolloutLocator. The plugin declares that the shared history tab supports Codex, while the driver refuses an unverified in-terminal history picker.
  • External DLL loading is not part of this slice. The new contract and registry are the foundation for that next work item.
  • No live Codex process was launched by these automated tests. The verified launch path proves Director constructs the Codex agent through the plugin; live manual QA should start a Codex session and confirm /clear, stop, interrupt, and history binding in the UI.
