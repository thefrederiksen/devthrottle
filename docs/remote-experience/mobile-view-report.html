CC Director — Mobile View: Implemented and Tested

2026-05-23  .  Live test against a fresh build (slot 5, port 7879) at mobile viewport 390x844, real Claude session with a real turn.
What this is: the mobile remote-experience view, built and verified end-to-end in a browser. The headline new capability — the proactive supervisor briefing (P1.2 from the plan) — is implemented and working: for a session in mobile mode, the supervisor regenerates a plain-language Opus briefing at each turn-end and caches it, so the phone shows it instantly on open, no spinner. All three context modes (terminal / supervisor digest / voice) were exercised live.

1. The three modes, working

Session tab: the proactive briefing auto-loaded instantly (opus - auto), and the Mobile: ON toggle.

Agent activity + supervisor log
Tool cards (Find Files, Terminal) and the supervisor status timeline.

2. What I implemented

  • Proactive explain + cache — new ProactiveExplainService hooks the WaitingForInput turn-end transition; for mobile-mode sessions it builds the supervisor context and runs the Opus "explain" in the background, caching the result on the session.
  • Mobile-mode flag on Session (MobileMode) gating the cost, plus CachedExplainText/At/Model with a SetCachedExplain that ignores empty results (a failed/timed-out regen preserves the last good briefing, never blanks the screen).
  • Endpoints — GET /sessions/{id}/supervisor/explain (instant cached read) and POST /sessions/{id}/mobile-mode (toggle; warms the cache immediately on enable).
  • Session view wiring — on open it loads the cached briefing into the Explain panel instantly (marked "auto"), polls for newer briefings, auto-enables mobile mode on a phone-sized viewport, and a "Mobile: ON/off" toggle. Shared SupervisorContextBuilder so the on-demand and proactive paths build identical context.
turn finishes (-> WaitingForInput)
   -> ProactiveExplainService (mobile-mode only): build context, Opus "explain" in background
   -> session.SetCachedExplain(text)          [keeps last-good on failure]
   -> phone opens session view -> GET /supervisor/explain -> rendered instantly, marked "auto"

5. Comments and recommendations

DONEFidelity rule held under a real test

The briefing correctly reported the session as idle and explicitly noted there was no pending question - it did not confabulate a user action. That is the trust floor (P1.1) doing its job on live data.

DONECost gating works as designed

Proactive Opus generation only runs for mobile-mode sessions; the flag is visible and toggleable via the Mobile button. Enabling it warms the cache immediately rather than waiting for the next turn.

NEXTSupervisor view-link minting (P1.4) - the clear next feature

Not built yet. In mobile view, file paths the agent references should render as tap-able Tailscale HTTPS links the user can open in-tab (HTML/PDF), per the solo-tailnet no-sandbox decision. This is the next thing to implement on top of this foundation.

NEXTGateway directory mobile icon

Surface MobileMode on the session DTO so the fleet directory shows which sessions are phone-armed. Small, and it makes the cost/coverage visible at a glance.

NEXTDecide auto-enable stickiness

Today the view auto-enables mobile mode on a phone-sized viewport for the current load. The open question from the plan: should that persist server-side so a session stays phone-armed after you close the tab, so the briefing keeps regenerating while you are away?

NOTEComposer placeholder clipped on narrow screens - fixed in source

The long '(Enter to submit, Shift+Enter for newline)' hint clips in the collapsed one-row textarea on a phone. Shortened to 'Send to session...' on mobile in source; the screenshots above predate the rebuild.

NOTERegen-while-busy is skipped, not queued

If a turn-end fires while a briefing is still generating, it is skipped (in-flight guard); the next turn regenerates. Fine for now; if briefings ever lag, switch to a coalescing 'regenerate once more when the current one finishes' flag.
