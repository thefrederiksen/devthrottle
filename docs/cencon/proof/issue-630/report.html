Issue #630 - Re-point the Director's login telemetry to the Gateway

Gateway Centralization Phase 1 · Containers: CcDirector.Core (DevThrottleLoginTelemetryReporter), CcDirector.Core.Tests · Developer Agent proof

Release notes

The Director no longer calls devthrottle.com directly to record a login. After a successful sign-in it now POSTs the login event to the configured Gateway (<gateway.url>/telemetry/login), which forwards it to the cloud (the relay shipped in issues #628 and #629). The body and the authorization header are unchanged (Authorization: Bearer <access token> plus { "source": "app", "app_version" }). When no Gateway is configured, the reporter is a transitional no-op that logs a skip line (Phase 1; the Gateway becomes mandatory in Phase 3). The report stays best-effort and fire-and-forget, so a telemetry error can never block or fail the user's login.

What was implemented

Changes

  • src/CcDirector.Core/Account/DevThrottleLoginTelemetryReporter.cs - the POST target is now resolved from the configured Gateway. A new gatewayUrl constructor parameter defaults to gateway.url in config.json (via GatewayConfig.Load()). The target is <gateway.url>/telemetry/login (a trailing slash on the Gateway URL is collapsed). When neither a Gateway URL nor the DEVTHROTTLE_TELEMETRY_URL override is configured, the method logs a skip line and returns - it never falls back to a direct devthrottle.com call. The former DefaultEndpoint constant (the cloud URL) was renamed ForwardTargetEndpoint and kept in source as documentation of the Gateway's forward target (it is no longer the Director's target).
  • src/CcDirector.Core.Tests/Account/DevThrottleLoginTelemetryReporterTests.cs - tests now assert the reporter POSTs to <gateway.url>/telemetry/login with the Bearer token and the source="app" body, that a trailing slash does not double-slash the path, and a new test proves the no-Gateway path is a logged no-op that makes no HTTP call. Each test passes an explicit gatewayUrl so it never reads the test machine's config.json.

Out of scope (left untouched, per the issue): the Gateway endpoint itself (#628/#629, already merged) and removing the devthrottle.com constant from the codebase (it remains as the Gateway's forward target; the Director simply stops calling it).

How this was tested

A full interactive browser sign-in is impractical headless, so - exactly as the issue permits - the proof drives the real FirstRunLoginCoordinator.RunAsync end to end (real loopback listener plus a stand-in sign-in completion that posts the credential to the callback) with the real DevThrottleLoginTelemetryReporter, against a stub Gateway that mirrors the production POST /telemetry/login relay route. A separate stub cloud (a devthrottle.com stand-in) records any request it receives; it would only ever be hit by a direct cloud call, so its emptiness proves the "no direct devthrottle.com call" property. The harness source is committed alongside this report (proof-harness-Program.cs.txt); its console output is harness.log and the machine-readable verdicts are results.json. The Director-side reporter log lines are director-reporter-log-lines.txt.

CenCon impact

No architecture or security-posture drift. The change is internal to one Core class and its tests; it moves the login-telemetry egress from a direct cloud call to the already-merged Gateway relay, which the Gateway Centralization Phase 1 plan (docs/architecture/GATEWAY_CENTRALIZATION_PLAN.md) already describes. Security rule DT-05 (never log the token) is preserved - only the target URL and the outcome status are logged, proven above.

Verdict

I believe this is finished. All four acceptance criteria are met and proven, the access token is never logged, the full solution builds with 0 warnings / 0 errors, and the unit tests pass.
