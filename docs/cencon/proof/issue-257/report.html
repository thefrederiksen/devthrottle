Scope & proof note (human-directed). Issue #257 is a large feature whose ACs 4-7 require a real, destructive uninstall (wiping PATH, scheduled tasks, Tailscale Serve, the app tree, and - on full-wipe - vault secrets and signed-in browser sessions). This is the user's production machine with no isolated VM, so a real run was explicitly deferred. The user directed Phase 1 = engine + CLI inverse operations, proven by unit tests + a sandbox (no real uninstall), with the skill-ownership question resolved by adding an ownership manifest. This report covers Phase 1 only. The whole of #257 is NOT finished - Phase 2 (in-app Tools->Uninstall UI, Start Menu uninstall shortcut, the install-side Add/Remove Programs registration call-site + skill-manifest write, safe-shutdown roster snapshot, detached self-delete, and the real before/after destructive proof) remains.

What Phase 1 implemented (engine: tools/cc-director-setup-engine)

  • SkillManifest - records the skill names CC Director installed, at config/setup/skills.json. The single ownership-truth that makes AC8 safe (Assumption 3 resolved by "add an ownership manifest").
  • ScheduledTaskRemover - removes cc-director-launch / cc-director-gateway-launch if present; absent = skipped (Assumption 4). OS call behind a seam.
  • TailscaleServeTeardown - removes ONLY the 443 front-door mapping (serve --https=443 off); CLI-absent = clean no-op; already-off = success (Assumption 5).
  • AddRemovePrograms - Register / IsRegistered / Unregister the HKCU\...\Uninstall\CcDirector entry (key name parameterized for test isolation).
  • Uninstaller - new UninstallKinds and Apply() steps wire all four in (skills + tasks for both roles, Tailscale + autostart for Gateway, ARP on Windows), with the existing idempotent removed/skipped/failed report. Plan() inventories manifested skills + the ARP entry.

The CLI uninstall / uninstall --dry-run command surfaces all of this through the existing tools/cc-director-setup-cli path - no CLI code change was needed.

Why no real uninstall was run

The OS-touching removals are machine-global, not scoped to CC_DIRECTOR_ROOT: scheduled tasks, the Tailscale 443 mapping, and skills under %USERPROFILE%\.claude\skills. Running the real Apply() on this machine - even with a fake root - would delete the user's real cc-director-launch tasks, tear down the live Gateway front door, and remove real skills. So the destructive paths are proven instead by (a) seam-injected unit tests and (b) real round-trips on uniquely-named throwaway artifacts (a sandbox scheduled task and a throwaway registry key), and the read-only --dry-run Plan against a sandbox root.
