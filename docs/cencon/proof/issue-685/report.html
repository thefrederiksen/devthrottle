Proof - Issue #685

Title: PortAllocator burns a port slot per ungraceful shutdown; stale .port files exhaust 7879..7898 and disable the Control API

Branch: issue-685-portallocator-stale-reservations

File changed: src/CcDirector.ControlApi/PortAllocator.cs + src/CcDirector.Gateway.Tests/PortAllocatorTests.cs (new)

What was implemented

A .port reservation file is only deleted on a graceful Release. Every hard-kill, crash, or reboot left one behind. ReadPortsInUseByOtherDirectors trusted every file on disk, so ghost reservations accumulated until the whole 7879..7898 range looked busy even when every TCP port was actually free - and a starting Director threw "All ports in range 7879..7898 are busy", bringing up no Control API (Remote, Gateway, and phone access all off).

The fix, in PortAllocator.cs:

  • The per-file decision logic was extracted into an internal, testable method CollectLivePortReservations(directory, except, isPortFree) that takes the port-liveness predicate by injection (so tests do not touch real OS ports or the real user's reservation directory). ReadPortsInUseByOtherDirectors is now a thin wrapper that passes the real IsPortFree.
  • A reservation is counted as "used by others" only when its claimed port is genuinely NOT bindable. A live Director holds its port, so a port that is free to bind proves the reservation is a ghost from an ungraceful exit.
  • Ghost reservations (claimed port free to bind) and malformed files are pruned (deleted) in passing, so the directory cannot fill up with ghosts and exhaust the range over time. Pruning is best-effort and logged; a transient delete failure does not abort the scan.
  • A narrow per-file guard logs and skips a reservation file that another starting Director deleted mid-scan, instead of aborting the whole allocation.
