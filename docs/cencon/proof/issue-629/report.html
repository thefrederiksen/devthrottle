Issue #629 - Durable retry queue for telemetry forwarding (survive cloud outages)

Gateway Centralization Phase 1, plan decision #3 · Containers: CcDirector.Gateway, CcDirector.Gateway.Tests · Developer Agent proof

What was implemented

A durable, bounded, restart-surviving retry queue behind the Gateway's login-telemetry relay (the forwarder shipped in issue #628). The relay no longer forwards inline; it enqueues every accepted event and answers the caller 202 Accepted immediately. The queue owns delivery: it flushes in FIFO order, retries with backoff while the backend is unreachable, is bounded (the oldest event is evicted when full, with a logged warning), and persists across a Gateway restart (one JSON file under the Gateway config directory, written through atomically on every mutation and reloaded on construction).

Changes

  • src/CcDirector.Gateway/Api/TelemetryRetryQueue.cs - NEW. The durable bounded FIFO queue: enqueue (with oldest-eviction at the bound), a background flush loop with a retry interval, FIFO head-first drain that stops at the first failure (preserving order), atomic temp+rename write-through persistence, corrupt-file quarantine, and a clean async dispose. The access token is stored on disk for replay but is never written to a log line - logs record only the target URL, the queue depth, and a bearerPresent flag.
  • src/CcDirector.Gateway/Api/TelemetryRelayEndpoint.cs - the relay endpoint now takes the queue and Enqueues instead of forwarding inline.
  • src/CcDirector.Gateway/GatewayHost.cs - constructs the queue (config-dir path, short-timeout forwarder client, 30s production retry interval), wires it into the relay, starts the flusher in StartAsync, and disposes it in StopAsync. New test-injectable constructor params: telemetryQueuePath, telemetryQueueMaxSize, telemetryRetryInterval.
  • src/CcDirector.Gateway.Tests/TelemetryRetryQueueTests.cs - NEW. 10 unit tests.
  • src/CcDirector.Gateway.Tests/TelemetryRelayEndpointTests.cs - updated the #628 wire tests to the enqueue-then-flush flow (delivery is asynchronous now).

CenCon impact

No architecture or security-posture drift. This wraps the existing #628 forwarder with a queue; it adds no new container, no new egress, and no new event type. The egress target and the Gateway-as-single-egress posture are unchanged. The #628 security property (token never logged) is preserved and re-proven here. No blocking security rule (DT-*) is affected. Persistence lives under the Gateway config directory, which the issue's assumptions explicitly permit.
