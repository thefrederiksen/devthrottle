Issue #870 - Flaky throughput test

Branch issue-870-flaky-throughput (off main).

I believe this is finished.

1. Root cause (measured)

TerminalThroughputTests.Parse10MbInChunks_EachChunkCompletesWithin100Ms parsed 10 MB in 160 x 64 KB chunks and failed if any single chunk exceeded a 100 ms wall-clock budget. Measured distribution across runs: the typical (median) chunk is ~25 ms, max usually 67-85 ms, but it occasionally spikes to ~128 ms from GC / OS-scheduler jitter (one run: max 128 ms, 2 offenders -> FAIL). The parser is fine; a single jitter spike at the 100 ms line flaked the gate.

2. Fix

Assert the signals that separate a responsive parser from a real regression, instead of an absolute per-chunk deadline (the approach the issue's criteria endorse):

  • Median chunk <= 100 ms - the typical chunk stays within the UI budget (jitter-robust: a few outliers do not move the median; a real slowdown does).
  • <= 5% of chunks may exceed 100 ms - tolerates rare GC/scheduler jitter; a systemic slowdown pushes far more over.
  • No chunk > 1000 ms - a true catastrophic per-chunk UI block still fails.

Test-only change in src/CcDirector.Core.Tests/TerminalThroughputTests.cs (renamed to Parse10MbInChunks_TypicalChunkStaysWithinUiBudget to match the assertion). No production code change.
