Issue #650 - First-run consent screen shown once at Gateway first launch

Gateway Centralization Phase 3. Branch: issue-650-gateway-consent. Verified on a real, running Gateway build with an isolated config root (a genuine first launch).

What was implemented

  • src/CcDirector.Core/Configuration/GatewayConsentAck.cs (new) - the gateway-side persisted acknowledgement flag, the relocation of the per-Director FirstRunConsent flag. Persists gateway_consent.acknowledged in the Gateway's config.json via the non-lossy CcDirectorConfigService.MergePatch (the same store the centralized #649 setting uses). Deliberately a separate config section from the per-Director consent section.
  • src/CcDirector.Gateway/Account/GatewayConsentSurface.cs (new) - the pure, unit-tested decision-and-persistence library (mirrors GatewaySignInTraySurface): ShouldShowConsentOnLaunch(), CurrentUsageSharingChoice() (reads the #649 default), and Acknowledge(shareUsage) which writes BOTH the centralized consent setting (TelemetryConsentConfig.Set, #649) and the gateway acknowledgement (GatewayConsentAck.MarkAcknowledged) in one step.
  • src/CcDirector.GatewayApp/GatewayConsentWindow.axaml(.cs) (new) - the consent screen, modeled on the per-Director FirstRunConsentDialog (dark theme #1E1E1E, accent #007ACC, per docs/VisualStyle.md), relocated onto the Gateway. The checkbox is the fleet-wide centralized consent; "Got it - continue" calls GatewayConsentSurface.Acknowledge off the UI thread (button disabled during the write).
  • src/CcDirector.GatewayApp/GatewayTrayController.cs - after host start (alongside the #637 first-launch sign-in), shows the consent screen once via ShowConsentIfNeeded() only when not yet acknowledged. Closed on Quit; tracked so a Gateway restart within one run never stacks two screens.
  • Tests: CcDirector.Gateway.Tests/Account/GatewayConsentSurfaceTests.cs (6) + CcDirector.Core.Tests/Configuration/GatewayConsentAckTests.cs (4).
