Issue #268 — Route per-session WebSocket legs (dictate + terminal stream) through the Gateway

Developer Agent proof report · branch issue-268-gateway-ws-proxy · 2026-06-10 · build 0 / 0 380 tests green

1. What the issue asked for

A remote Cockpit (not on the Gateway host) could not use dictation or the live Terminal because the browser built the WebSocket URL from the owning Director's advertised TailnetEndpoint. When that was loopback (http://127.0.0.1:7887) the remote browser dialed its own loopback and the upgrade never landed. The decided fix (epic #279): the Gateway reverse-proxies the two raw per-session WS legs to the owning Director, and the Cockpit builds both WS URLs same-origin to the Gateway — never a Director address.

2. What was implemented

Area / fileChange
CcDirector.Gateway/Api/SessionWsProxyEndpoints.cs new Maps GET /sessions/{sid}/stream and GET /sessions/{sid}/dictate. Resolves the owning Director from the registry (parallel fan-out, mirrors LocateSessionAsync), then reverse-proxies the WS upgrade via the shared YARP IHttpForwarder (same forwarder that already carries the Blazor circuit). Path is rewritten to the Director's own endpoint (/sessions/{sid}/stream resp. /dictate). 404 unknown session, 503 unreachable owning Director. Logs open / resolved / close per leg.
CcDirector.Gateway/GatewayHost.cs Wires SessionWsProxyEndpoints.Map before the fallback Cockpit proxy so the explicit WS routes win.
CcDirector.Gateway.Contracts/CockpitWsUrls.cs new Pure static builder: Stream(origin,sid) / Dictate(origin,sid) — same-origin to the Gateway, http→ws / https→wss, sid-scoped path, default-port aware, deep-link path dropped. Shared by both Cockpit surfaces, unit-tested.
CcDirector.Cockpit/Components/TerminalPane.razor Builds the stream WS URL with CockpitWsUrls.Stream(Nav.BaseUri, sid) (Gateway origin). DirectorBase retained only for the non-WS raw-input REST call (Assumption A2).
CcDirector.Cockpit/Components/Pages/Cockpit.razor Dictation WS URL now CockpitWsUrls.Dictate(Nav.BaseUri, sid) (Gateway origin); old Director-targeted DictateWsUrl removed.
CcDirector.Cockpit/Components/BriefPane.razor The brief-feedback dictation path had the same Director-targeted WS bug; switched to CockpitWsUrls.Dictate(Nav.BaseUri, SessionId) so no Cockpit dictation surface targets a Director address.
CcDirector.Cockpit/wwwroot/js/cockpit-dictate.js Loud failure: a never-opened upgrade now reads “Could not reach the owning Director through the Gateway (it may be offline or unreachable)” instead of a bare “WebSocket connection failed.”

6. CenCon impact

Two new components registered in architecture_manifest.yaml: SessionWsProxyEndpoints (gateway container) and CockpitWsUrls (gateway_contracts container). No security-posture change: the new endpoints sit behind the Gateway's existing token/auth middleware, reach the Director over the same control endpoint every other /sessions/{sid}/* forward uses, and add no new trust boundary — no DT-* rule in security_profile.yaml is touched.

7. Build environment note

The local machine's SDK 10.0.103 has a Razor-compiler regression that mis-parses a relational-pattern switch arm (< 1_000 => ...) in the pre-existing Cockpit.razor — it fails identically on clean main, and CI's 10.0.x compiles the same file with 0 errors. Proof was produced with SDK 10.0.301 (latest 10.0 GA), under which the full solution builds 0 warnings / 0 errors. This is environmental and unrelated to issue #268.

8. Honesty note

The two cross-machine acceptance criteria (remote browser devtools showing the Gateway-origin WS, a remote dictation transcription, and Director-side /dictate arriving from the Gateway) were not captured here — this is a single-machine session with no second tailnet box or live mic. Every mechanism those criteria depend on is proven: the URL the Cockpit builds (unit tests, never a Director address), the Gateway's owning-Director resolution and proxy routing (live 404 + logs + wire tests), and the loud-failure message. QA should run the two-box capture to close them.

I believe this is finished for the parts provable on one machine: the Gateway WS-proxy endpoints, the same-origin URL builders, the loud-failure message, and the CenCon doc updates are implemented, build clean (0/0), and are proven by 380 green tests plus a live Gateway run. The two genuinely cross-machine criteria are flagged above for a two-box QA pass.
