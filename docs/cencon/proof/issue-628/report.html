Issue #628 - Gateway login-telemetry relay endpoint

Branch: issue-628-telemetry-login-relay   Containers touched: CcDirector.Gateway, CcDirector.Gateway.Tests   Date: 2026-06-22

What was implemented

Gateway Centralization Phase 1: a new inbound POST /telemetry/login endpoint on the Gateway that forwards the login-telemetry event to the cloud backend, so the Gateway becomes the single egress. The Director will (in a later issue) POST its login telemetry here instead of calling the cloud directly.

  • Endpoint: src/CcDirector.Gateway/Api/TelemetryRelayEndpoint.cs - maps POST /telemetry/login. Reads the inbound JSON body and the inbound Authorization: Bearer <access token>, then forwards BOTH UNCHANGED to the backend login endpoint.
  • Target URL: default https://devthrottle.com/api/v1/telemetry/login, overridable on the Gateway via the environment variable DEVTHROTTLE_TELEMETRY_URL (ResolveTargetUrl()).
  • Wiring: TelemetryRelayEndpoint.Map(_app) in GatewayHost.cs, mapped alongside the other literal endpoints so the host-wide gateway-token auth middleware (the existing convention, applied when Gateway auth is enabled) governs it exactly like the other Gateway endpoints.
  • Best-effort: a backend 5xx or an unreachable backend is logged and the caller still gets a non-5xx (202 Accepted). The outbound call is wrapped in a try/catch boundary so a backend error NEVER throws back to the Director.
Security (acceptance criterion): the inbound access token (the Bearer) is forwarded unchanged but is NEVER written to the Gateway log. Every log line on this path records only the target URL, whether a Bearer was present, and the outcome - never the token value. This was verified live by grepping the real Gateway log for the token string (zero matches).

How the live proof was produced

A real Gateway was run via the Gateway dev console host (dotnet run --project src/CcDirector.Gateway -- --port N) with DEVTHROTTLE_TELEMETRY_URL pointed at a local stub backend (stub/, which logs every request it receives) and an isolated CC_DIRECTOR_ROOT so the Gateway's real on-disk FileLog could be grepped. The success run used a stub returning 200; the AC3 runs used a stub returning 500 and then no stub at all (a dead port). All proof Gateway processes were launched from this worktree's bin and stopped by exact PID/image-path after capture - no cc-director.exe and no other session's Director was touched.
