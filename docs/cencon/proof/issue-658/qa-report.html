QA verification - Issue #658: installer hands the captured sign-in to the app

Independent QA Agent report - CenCon Development Method. Branch: issue-658-handoff-signin (pull request #670). The QA Agent is a separate identity from the Developer Agent and did NOT reuse the developer's binary or proof.

How this was verified independently. A fresh QA driver (Qa658Driver, not the developer's throwaway) was written that drives the real SignInRunner default persistence path - the production PersistToDirectorCredentialStore -> DevThrottleAccountFactory.CreateForWindows() -> WindowsProtectedTokenStore chain, with NO test seam - under an isolated CC_DIRECTOR_ROOT in a throwaway temp directory. This means the user's real credential blob at %LOCALAPPDATA%\cc-director\config\director\devthrottle-credential.bin was never read, written, or deleted. A properly HMAC-SHA256-signed development access token (dev signing secret) plus a refresh token were captured over the same loopback contract the Director uses, then read back exactly as the Director's account gate does. The 8 unit tests in SignInRunnerTests were also re-run from a clean QA checkout of the pull-request branch.

Note on criterion 2 and issue #664 (verified, not taken on trust). The Director's startup account gate was disabled in #664 (AccountGatePolicy.Decide() always returns Start), so a launched Director boots to the main window regardless of the stored credential - the developer's director-startup-log.txt even shows the slot Director booting with no stored credential -> false, decision=Start. Therefore "no gate UI" is NOT proof of #658. QA verified the part #658 actually delivers and that is durable: the credential the installer writes makes the gate's IsLoggedIn() return true. Launching a full Director to observe this was deliberately avoided because that path reads the shared real credential file; the isolated-root driver proves the same logged-in behavior without risking the user's credential.
