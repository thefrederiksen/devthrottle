Issue #658 - Installer hands the captured sign-in to the app

Developer Agent proof report - CenCon Development Method. Branch: issue-658-handoff-signin

What was implemented

Issue #657 added a forced Sign-in step to the installer that captures an access-plus-refresh token pair over the same loopback contract the Director uses, then deliberately discarded that token (persistence was reserved for this issue). This change makes the installer persist the captured pair into the same operating-system credential store the Director reads, so launching the Director after install reaches the main window already signed in.

The persistence is done inside tools/cc-director-setup/Services/SignInRunner.cs:

  • A new persistCredential collaborator is injected into SignInRunner (no fallback construction; an explicit default). Its default, PersistToDirectorCredentialStore, hands the captured pair to the existing DevThrottleAccountService built by DevThrottleAccountFactory.CreateForWindows() - the exact credential service the Director uses - which encrypts the pair at rest through WindowsProtectedTokenStore to %LOCALAPPDATA%\cc-director\config\director\devthrottle-credential.bin.
  • Persistence runs only after a successful capture. Every non-success path (cancel, timeout, browser-cannot-open, malformed hand-back) returns before reaching the persistence step, so no credential is written for an incomplete sign-in.
  • A persistence failure is surfaced as a failure result (no fallback): if the credential cannot be stored, the install has not achieved "open already signed in", so the runner does not report success.
  • The token value is never returned and never logged - only the fact that a credential was captured and persisted is written to the installer log.

PASS - end-to-end driver (criterion 2): IsLoggedIn() == true. See also the note below on issue #664.
3. The token is written only to the encrypted store, never to the installer log. The runner logs only "credential captured" and "credential persisted"; the value is never passed to a log call. The blob is encrypted at rest. PASS - the captured setup log contains no eyJ (JWT), access_token, refresh_token, or Bearer text. See setup-log-no-token.txt.
4. If sign-in did not happen (error path), no credential is written. Cancel, timeout, browser-cannot-open, and a persistence failure all return before / instead of writing a credential. PASS - unit tests RunAsync_UserCancels_ReturnsCancelled, RunAsync_NoHandBackWithinTimeout_ReturnsTimedOut, RunAsync_BrowserCannotOpen_ReturnsFailed assert persistence was never called; RunAsync_PersistenceFails_ReturnsFailed proves the no-fallback behavior.
Honest note on criterion 2 and issue #664. The Director's startup account gate was disabled in issue #664 (interim Gateway-account unblock, already on main): AccountGatePolicy.Decide() now always returns Start, so the Director boots straight to the main window regardless of whether a local credential exists. The Director therefore does not show the gate today even without #658. The part of criterion 2 that #658 genuinely delivers - and that this proof verifies directly - is that the credential the installer writes satisfies the gate's logged-in check (IsLoggedIn() == true), which is the durable behavior the Gateway-verified gate (issue #641) will rely on. This is stated plainly so QA is not misled into attributing the "no gate" behavior solely to this change.
Dev/QA vs production. The real production sign-in page is not live, so this was proven against the local stand-in tools/devthrottle-dev-signin (per the issue). A stand-in token returns 401 against production - expected; a production pass follows once devthrottle_internal#46 ships. Locally the full path - capture, encrypt, store, read back, IsLoggedIn() - is exercised end to end.

CenCon impact

No architecture or security-posture drift. The change reuses the existing credential store and service (no new containers, no new credential path). It strengthens compliance with the "never log a secret" rule by keeping the captured token out of the installer log and the encrypted blob. No docs/cencon/ file required updating.

Conclusion

I believe this is finished. All four acceptance criteria are met and proven: the captured sign-in is persisted to the Director's encrypted credential store on success only, it decrypts to the captured pair, it satisfies the gate's logged-in check, the token is never logged, and no credential is written on any error path.
