Issue #631 - Director-startup telemetry endpoint (record + best-effort cloud-forward)

Gateway Centralization Phase 1 · Containers: CcDirector.Gateway, CcDirector.Gateway.Tests · Developer Agent proof

What was implemented

A new Gateway endpoint POST /telemetry/director-startup accepting { "director_id": "...", "machine_name": "...", "app_version": "..." }. It always records the event Gateway-side (a log line carrying director_id + app_version) so a Director startup is observable on the Gateway, then best-effort forwards it to the cloud only when a startup endpoint is configured via the DEVTHROTTLE_STARTUP_TELEMETRY_URL environment variable. The backend has no Director-startup endpoint yet (a flagged dependency), so forwarding is gated on configuration: with no URL set, the event is recorded locally, a "no cloud startup endpoint configured" line is logged, and the caller still gets 202 Accepted - no error. When a URL IS configured, the event is enqueued into the same durable retry queue the login relay uses (issues #628 / #629), so delivery, retry-with-backoff, FIFO flush, the bound, and restart survival are reused - this endpoint adds no new forwarder. Unlike the login relay, a startup event carries no inbound Bearer (Phase 1 startup is unauthenticated to the cloud per the provisional contract).

Changes

  • src/CcDirector.Gateway/Api/DirectorStartupTelemetryEndpoint.cs - NEW. Maps POST /telemetry/director-startup; resolves the optional startup URL (no default); records the event; conditionally enqueues into the shared TelemetryRetryQueue; returns 202 on both paths. A malformed body still records (with (none) placeholders) and still returns 202 - the inbound event is best-effort observability, never a validation gate on the 202.
  • src/CcDirector.Gateway/GatewayHost.cs - wires the new endpoint, reusing the existing _telemetryQueue (no new construction).
  • src/CcDirector.Gateway.Tests/DirectorStartupTelemetryEndpointTests.cs - NEW. 5 tests covering the record-only path, the configured-forward path, the 202-for-valid-body criterion, and ResolveTargetUrl (no-env -> null, with-env -> override).
  • docs/cencon/architecture_manifest.yaml - lists the new endpoint component next to its sibling TelemetryRelayEndpoint (keeps the CenCon container inventory current).

CenCon impact

No architecture or security-posture drift. This endpoint is structurally parallel to the already-documented TelemetryRelayEndpoint (issue #628) in the same container; it adds no new egress class (it reuses the #629 queue as the forwarder) and no new credential surface (a startup event carries no Bearer). No blocking security rule (DT-*) is affected. The new endpoint component is added to docs/cencon/architecture_manifest.yaml beside its sibling so the container inventory stays accurate. Forwarding is gated on an explicit operator configuration, consistent with the issue's flagged backend dependency.
