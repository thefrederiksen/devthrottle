Proof - Issue #687

[Recorder] On restart, reconcile and upload ALL local recordings in the background (some recordings never sync)

Branch: issue-687-recorder-reconcile-all

What was implemented

The CC Recorder phone app now reconciles and uploads every local recording on each app start and keeps draining reliably in the background while closed. Five concrete gaps from the issue are closed, mirrored into both phone projects (phone/CcRecorder - the shipping recorder - and phone/CcDirectorClient - the combined client, which shares the same recorder code path).

  1. Recover orphaned recordings on launch (AC1). New pure decision RecordingUploadGate.NeedsRecovery(state, hasAudioSegments) identifies an interrupted recording (state "Recording", i.e. EndedAt == null, but with captured segments on disk). New AndroidAudioRecorder.RecoverOrphanedRecordings() finalizes each such recording (EndedAt set, State = "Queued") so it enters the normal upload path. It runs at the start of every ProcessUploadQueueAsync pass (launch and the background worker) and skips the live in-progress recording so a current capture is never torn out from under itself.
  2. Reconcile-everything pass on launch is not UI-dependent (AC2). The launch reconcile (ProcessUploadQueueAsync + EnqueueBackgroundUpload) was moved into App.CreateWindow (both apps), so it fires at app start regardless of which page/shell screen is shown - no longer only in MainPage.OnAppearing.
  3. Background worker can run over Tailscale (AC3). UploadScheduler's WorkManager constraint changed from NetworkType.Connected to NetworkType.NotRequired, so a Tailscale / VPN-only / constrained network that Android does not classify as "Connected" no longer blocks the periodic and one-time workers from starting. Reachability is governed by the in-app NetworkAccess == None check plus per-attempt failure→Retry, the same Tailscale-aware logic the foreground path already uses.
  4. Survive reboot / force-stop (AC4). New BootReceiver (BroadcastReceiver on BOOT_COMPLETED) re-arms EnsurePeriodic + EnqueueNow after a reboot, plus the RECEIVE_BOOT_COMPLETED permission in both AndroidManifest.xml.
  5. No data loss / no false deletion (AC5). The server→phone deletion sync is unchanged - it still only deletes IsDeletable (fully-delivered) recordings. A recovered orphan becomes "Queued" (not fully delivered, not deletable), so it can never be deleted before confirmed delivery. Pinned by a unit test.

Why no desktop test-Director slot screenshot. This is a phone-only change: the recorder runs on an Android device, not on a desktop CC Director. There is no desktop Control API surface to drive for it. Proof is therefore the headless pure-logic unit tests (the orphan-recovery decision the issue explicitly asks to cover) plus a clean net10.0-android build of both shipping apps. The three device-runtime scenarios in the issue's Verification section (force-kill mid-recording, Tailscale-only drain, reboot drain) require a physical Android device with a paired Gateway and are for QA to exercise on-device; the code paths that satisfy them are in place and build clean.

CenCon impact

No architecture or security drift. The phone recorder is not a desktop container in architecture_manifest.yaml; no new secrets, no new network surface (same Gateway upload path). RECEIVE_BOOT_COMPLETED is a normal-protection-level Android permission. No docs/cencon/ file needed updating.
