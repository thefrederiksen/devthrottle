Issue #175 - GatewayTrayInstaller inherits stdio: piped CLI callers hang forever

Role: Developer Agent (CenCon Development Method) · Repo: thefrederiksen/cc-director · Branch: loop/175-installer-stdio

1. What was implemented

The Gateway tray-app launch in tools/cc-director-setup-engine/GatewayTrayInstaller.cs (step 4 of InstallAsync) started the long-lived tray Gateway with UseShellExecute=false, so the Gateway inherited the setup CLI's stdout/stderr handles and held them open for its whole lifetime. Any caller that PIPES the CLI (... | Select-Object, ... | tail) therefore hung forever after the CLI itself had exited (the live v0.6.6 90-minute phantom hang).

The fix extracts the launch ProcessStartInfo into a testable static seam BuildTrayLaunchInfo(gatewayExe, workingDirectory) and sets UseShellExecute=true (with NO RedirectStandard*), exactly mirroring the already-proven fix on the GatewayApp self-update relaunch path (src/CcDirector.GatewayApp/Program.cs startGateway). A code comment references issue #175 and that precedent.

2. The change (diff)

// tools/cc-director-setup-engine/GatewayTrayInstaller.cs - step 4
-            var psi = new ProcessStartInfo
-            {
-                FileName = gatewayExe,
-                Arguments = InstalledArguments,
-                WorkingDirectory = _layout.GatewayDir,
-                UseShellExecute = false,
-            };
+            var psi = BuildTrayLaunchInfo(gatewayExe, _layout.GatewayDir);

// new static seam (UseShellExecute=true, no redirection; see issue #175):
+    public static ProcessStartInfo BuildTrayLaunchInfo(string gatewayExe, string workingDirectory)
+    {
+        ArgumentException.ThrowIfNullOrWhiteSpace(gatewayExe);
+        ArgumentException.ThrowIfNullOrWhiteSpace(workingDirectory);
+        return new ProcessStartInfo
+        {
+            FileName = gatewayExe,
+            Arguments = InstalledArguments,
+            WorkingDirectory = workingDirectory,
+            UseShellExecute = true,
+        };
+    }

5. How the end-to-end proof reproduces the hang

PipedCaller_ReturnsWithFix_AndWouldHangWithoutIt opens an anonymous pipe whose write end is marked inheritable, then starts a long-lived child (ping -n 30 127.0.0.1) two ways and tries to read the pipe to EOF:

  • OLD (UseShellExecute=false): Process.Start creates the child with bInheritHandles=TRUE, so the child inherits the write handle - exactly how the real Gateway inherited the setup CLI's stdout. The reader gets NO EOF within the window: the hang, reproduced. The test asserts this.
  • NEW (the production BuildTrayLaunchInfo, UseShellExecute=true): ShellExecuteEx launches with bInheritHandles=FALSE, so the handle is NOT passed to the child; the reader reaches EOF promptly even though the child is still alive - the piped caller returns. The test asserts this.
