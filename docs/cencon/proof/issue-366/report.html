Issue: #366 [Voice] Fix stale transcript: voice-turn endpoint speaks previous-turn content
PR: #379 (branch issue-366-stale-transcript)

What was implemented

The voice-turn endpoint (POST /sessions/{sid}/voice-turn) read the most recent assistant message of the WHOLE JSONL transcript after its turn poll loop. When the agent had not yet written its reply to the current utterance, the endpoint summarized and spoke the PREVIOUS turn's output - a systematic stale replay.

  • src/CcDirector.ControlApi/VoiceTurnEndpoint.cs - SnapshotTranscriptLength(session) captures the transcript byte length immediately before session.SendTextAsync(). After the poll loop, ReadNewAssistantText(session, offsetBefore) reads only assistant content appended at/after that offset. ReadLastAssistantText is gone.
  • src/CcDirector.Core/Claude/StreamMessage.cs - new public StreamMessageParser.ParseFileFromOffset(jsonlPath, byteOffset) (seeks to the snapshotted line boundary, reads with FileShare.ReadWrite; an offset beyond EOF - transcript replaced - reads from the start because then all content is new).
  • src/CcDirector.Gateway.Tests/VoiceTurnEndpointTests.cs - regression test VoiceTurn_TwoConsecutiveTurns_SecondTurnSpeaksCurrentReply plus a TranscriptWritingBackend stub that appends scripted assistant replies to a real temp JSONL transcript per turn (and, when the script runs out, ends a turn with NO new content - the exact defect window).
