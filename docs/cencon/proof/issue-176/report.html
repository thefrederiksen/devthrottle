Issue #176 - Self-update test harness pollutes production installed.json

Developer Agent (CenCon) - implementation proof. Repo: thefrederiksen/cc-director. Branch: loop/176-selfupdate-installedjson.

Problem

A live v0.6.6 gateway install reported success but silently did not swap the gateway exe. Root cause: the gateway self-update test harness wrote FAKE versions (gateway 9.9.9 into installed.json, 9.9.8 into update-pins.json) into the production %LOCALAPPDATA%\cc-director\config\setup state. UpdatePlanner then judged installed 9.9.9 >= release 0.6.6 -> UpToDate -> the swap never ran while the installer reported success.

Two independent defenses (as specified)

Defense 1 - isolate the test harness (no production writes)

scripts/test-gateway-selfupdate.ps1 now takes a -Root override and exports CC_DIRECTOR_ROOT to a throwaway temp root before launching the self-update helper. InstallLayout.Default() already honors CC_DIRECTOR_ROOT, so the helper's InstalledManifest.Save / PinStore.Save writes land under the isolated root, never production. The script also (a) refuses to run if -Root resolves to the production root, and (b) hashes the production installed.json / update-pins.json before and after the run and FAILS the run if they changed.

Defense 2 - sanity guard in UpdatePlanner

InstalledStateReader now records the exe's real on-disk FileVersion separately from the recorded (installed.json) version. UpdatePlanner.Plan detects when the recorded installed version is strictly NEWER than the newest released version (the test-pollution signature), logs a loud WARNING via EngineLog, and prefers the exe's real FileVersion for the decision instead of reporting UpToDate. If the exe carries no readable stamp to fall back on, it re-applies the released build rather than trusting the discarded fake version.

CenCon impact

No architecture or security-posture drift. The change is confined to the CcDirector.Setup.Engine container's update-planning logic and its test harness; no container boundaries, data flows, or DT-rules are affected. No docs/cencon/ update required.

Note on the live end-to-end run

The full test-gateway-selfupdate.ps1 end-to-end run (which builds the gateway and spawns processes binding ports) was NOT executed on this busy shared machine to avoid disturbing the live fleet (Gateway 7878, Cockpit 7470, Director 7887 - all confirmed responding 200 at the end). The isolation mechanism the script depends on is proven hermetically and repeatably by SelfUpdateIsolationTests under net10.0, asserting the production files are byte-identical to the hashes above.

I believe this is finished.
