What was implemented

The Gateway tray left-click flyout was rebuilt on every open by GatewayTrayController.BuildFlyoutModel, which read _host.Registry.ListDirectors().Count synchronously on the UI thread, and the Open Cockpit action resolved the Tailscale front-door URL by shelling the tailscale CLI (TailscaleIdentity.TryGetFrontDoorBaseUrl, up to a ~5 second blocking timeout). Both blocked the flyout from painting when the registry read was slow or the Tailscale probe stalled.

The change moves both reads onto background heartbeats that refresh cached fields:

  • New src/CcDirector.Gateway/Tray/GatewayTrayFlyoutCache.cs - a pure, thread-safe cache (mirroring the testable-helper pattern of GatewaySignInTraySurface). Holds the Director count and the front-door base URL. DirectorCountDisplay returns the count or the "..." placeholder until the first heartbeat resolves it; FrontDoorBaseUrl is null until resolved or when Tailscale is unavailable.
  • GatewayTrayController - two background heartbeats started in Start(): a fast one (every 2s) refreshing the cheap registry count, and a slower one (every 15s) running the slow Tailscale CLI probe. BuildFlyoutModel now reads only _flyoutCache.DirectorCountDisplay; OpenCockpit now reads only _flyoutCache.FrontDoorBaseUrl. Neither does a synchronous registry read or CLI probe.
  • New src/CcDirector.Gateway.Tests/Tray/GatewayTrayFlyoutCacheTests.cs - 8 unit tests covering the placeholder, resolved values, the zero-vs-unresolved distinction, the null front-door (Tailscale unavailable) case, and a timing test proving a flyout-style read returns in under 100 ms while a 500 ms "slow probe" runs in the background.

Structural proof (acceptance criterion 1)

The only ListDirectors and TailscaleIdentity references in GatewayTrayController.cs now live exclusively in the two heartbeat refresh helpers - never in BuildFlyoutModel or OpenCockpit:

Live tray-open timing - honest limitation (QA gate)

A live tray-open timing screenshot/recording was not capturable from this automation session, and it has not been faked. Two honest reasons:

  • The flyout is a native notification-area (tray) UI element of the Avalonia devthrottle-gateway.exe process; it cannot be pixel-captured from a headless automation session.
  • Launching a second Gateway to capture it would bind the Gateway port and touch the per-user autostart registry entry, colliding with the user's live Gateway - which the CenCon isolation rules forbid. (The per-session slot isolation script targets cc-director.exe Directors, not the Gateway tray app.)

The instant-open behavior is therefore proven structurally (criterion 1 grep), by the clean build, and by the unit tests (including the slow-probe timing decoupling test). The live tray-open timing screenshot/recording for criteria 2, 3, and 5 is left as the QA gate for the QA Agent to verify in the running app.
