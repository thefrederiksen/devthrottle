Issue #394 - Voice playback observability + no-mic voice-turn test harness (Phase 1)

Branch: issue-394-voice-observability  |  Scope: Phase 1 only (Phases 2 and 3 are explicit follow-ons, out of scope).

The voice path "cuts out" intermittently and was hard to diagnose because the two places it breaks - server-side TTS reliability and on-device playback - had almost no observability or automated coverage. This change adds the observability and the automated no-microphone harness so the failure modes are measurable and regression-protected. No behaviour change to the shipping voice path beyond added logging and a test seam.

Reconciliation with #389 (merged minutes earlier, PR #612)

Issue #389 already made TtsService's HttpMessageHandler injectable and added src/CcDirector.Core.Tests/Voice/TtsServiceTests.cs covering the single-chunk, multi-chunk byte-exact join, stalled-chunk, and 5xx-chunk paths via an injected fake handler. The #394 "Host-side TtsService tests" acceptance criterion is therefore already satisfied by #389; this change does NOT duplicate that seam or those tests. I re-ran them as evidence: 9/9 passed. (Per the issue note, the obsolete "assert the old 180s stall" regression target was intentionally NOT reintroduced - #389 changed that behaviour and the new timeout/retry behaviour is what is tested.)

What was implemented (the genuinely novel Phase 1 work)

1. Android playback observability

  • phone/CcDirectorClient/Platforms/Android/AndroidReplySpeaker.cs - on PlayAsync logs byte count AND an estimated duration; logs the TERMINAL outcome with elapsed wall time and played-vs-estimated duration, distinguishing Completed (played to end) / Error (with MediaPlayer what/extra) / Interrupted (Stop or cancel); registers an audio-focus-change listener and logs gain/loss/duck during playback.
  • phone/CcDirectorClient/Voice/IReplySpeaker.cs - PlayAsync now returns Task<PlaybackOutcome> so the caller can record the result.
  • phone/CcDirectorClient/Voice/VoiceConversation.cs - the existing PollAndSpeak done line now carries audioBytes, the playback result, and played-vs-estimated seconds.
  • New pure, dependency-free helpers so the wording and the duration maths are testable off-device (the real Android MediaPlayer cannot run headless): PlaybackOutcome.cs, Mp3Duration.cs (estimates a CBR MP3's length from its frame headers), PlaybackLog.cs (the ASCII-only log-line formatter).

2. No-microphone end-to-end harness

  • src/CcDirector.Gateway.Tests/VoiceTurnNoMicE2EHarnessTests.cs - drives the Gateway's real voice-turn/submit + poll + audio-fetch pipeline using the TEXT field (no microphone, no transcription, no live key). A small in-process STUB Director the Gateway discovers and dials answers the SSE turn with a reply event carrying non-empty MP3 bytes, so everything from the Gateway's submit route through RunTurnAsync, the slim poll, and the dedicated audio endpoint is the real shipping code. Asserts the poll reaches stage=reply with non-empty audio of a plausible size, and runs N=10 iterations reporting a pass rate so multi-chunk flakiness would show up as a number.

The played=3.1s/8.0s on the last turn is the cutout this issue makes measurable; before this change only the START "bytes=N" line existed and a cutout was invisible.
