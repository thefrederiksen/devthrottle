Issue #298 - [Loop] Issue-level claim to prevent two loops implementing the same issue (duplicate-prevention)

Developer Agent proof. Branch loop/298-issue-claim. This is a PROCESS / SKILL-DOC change (no C# touched), so the proof is a DETERMINISTIC demonstration of the claim mechanism against a THROWAWAY test issue, not a slot-5 running-app launch. The live fleet (Gateway 7878, Cockpit 7470, Director 7887) was never touched.

The problem

On 2026-06-10 two concurrent implementation-loops both selected the oldest flow:ready-dev issue (#199) and both implemented it - duplicate PRs #294 / #296 and a near-collision pushing the same branch. The loop's "one consumer per LIST" guard (epic #270 AC4) does not guard at the ISSUE level. There was no lock on issue selection.

What was implemented (the mechanism)

  • New repo label flow:in-progress - the claim marker. Created on the repo (gh label create).
  • Claim at selection (implementation-loop Step 0.2): the instant a loop selects an issue it transitions flow:ready-dev -> flow:in-progress in one gh issue edit and records a timestamped CLAIM ... comment naming the claimant.
  • Selection reads flow:ready-dev only - a claimed (in-progress) issue is invisible to every other loop's selection set.
  • Verify-after-claim: GitHub labels are NOT an atomic compare-and-swap, so after claiming, the loop re-reads the issue's CLAIM comments and confirms its own is the OLDEST. The loser of a race (a later CLAIM comment) backs off and selects a different issue. Comment createdAt order is the deterministic arbiter. This is honest best-effort with a deterministic loser-back-off - the residual sub-second window is documented, no false claim of lock-free atomicity (DEVELOPMENT_METHOD.md Section 4a).
  • Release on every terminal path: the Developer hand-off swaps flow:in-progress -> flow:ready-qa (and reject / escalate remove it too); PASS ends flow:done, escalations end flow:needs-human. flow:in-progress is never a terminal state - the loop's Step 4 claim-release gate enforces it.
  • Stale-claim sweep (Step 0.0): a flow:in-progress issue whose newest CLAIM comment is older than 60 min (a crashed claim) is reclaimed flow:in-progress -> flow:ready-dev, so an issue is never stranded invisible forever. A fresh claim is protected from the sweep.

Design honesty (the subtlety called out in the issue)

GitHub label operations are not an atomic compare-and-swap. The mechanism does NOT claim perfect atomicity. It uses a best-effort single-edit claim (which closes the WIDE window we hit on #199 - the label flips the instant work starts) PLUS a verify-after-claim re-read that resolves the remaining narrow window by claim-comment order (oldest wins, loser backs off). The residual exposure is only the sub-second window between two loops selecting at the same instant, and the re-read resolves even that deterministically. True mutual exclusion would require an external lease (e.g. Gateway-held) - explicitly out of scope and documented as such in Section 4a.
