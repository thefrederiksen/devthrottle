Issue #349 - Voice Audio Cues (Start / Stop / Error Beeps)

Developer Agent proof report | Branch: issue-349-voice-audio-cues | PR: #354 | Date: 2026-06-11

What Was Implemented

SCOPE

Added audio cues (beeps/tones) to the voice recording flow in the Android phone client for eyes-free driving confidence. Three distinct sounds: a start beep when the mic opens, a higher-pitch stop beep on successful submit, and an error tone on turn failure.

NEW FILES
  • phone/CcDirectorClient/Voice/IAudioCue.cs - Interface: PlayStart / PlayStop / PlayError
  • phone/CcDirectorClient/Voice/NullAudioCue.cs - No-op stub for non-Android / tests
  • phone/CcDirectorClient/Platforms/Android/AndroidAudioCue.cs - ToneGenerator implementation
  • phone/CcDirectorClient.Tests/AudioCueTests.cs - 5 unit tests
MODIFIED FILES
  • phone/CcDirectorClient/Controls/VoiceSessionView.xaml.cs - Added _audioCue field, optional Configure parameter, 3 call sites
  • phone/CcDirectorClient/FifoPage.xaml.cs - IAudioCue injected + passed to Voice.Configure
  • phone/CcDirectorClient/MauiProgram.cs - DI registration (AndroidAudioCue on Android, NullAudioCue elsewhere)
  • phone/CcDirectorClient.Tests/CcDirectorClient.Tests.csproj - Link IAudioCue.cs and NullAudioCue.cs for test project

Acceptance Criteria

Criterion How the code satisfies it Verification
Distinct beep heard on Android when recording starts _audioCue?.PlayStart() called in HandleRecordAsync after permission granted and foreground service started, immediately before VoiceDictationDialog.PromptAsync. AndroidAudioCue fires ToneType.PropBeep (150ms) on a background thread. Functional test on Android device: tap Ask Agent, hear beep when dialog opens
Different tone heard on successful SUBMIT _audioCue?.PlayStop() called when VoiceDictationDialog.PromptAsync returns a non-null UtteranceAudio. AndroidAudioCue fires ToneType.PropBeep2 (100ms, higher pitch). Functional test on Android: submit a recording, hear distinct higher-pitch tone
Error tone heard when a turn fails (offline / transcription error) _audioCue?.PlayError() called in the catch (Exception) block of RunTurnAsync. AndroidAudioCue fires ToneType.PropNack (300ms, unmissable). Functional test: disable network, submit - hear NACK tone; also log shows "[AndroidAudioCue] PlayTone(...)"
No audio from NullAudioCue on non-Android / in tests NullAudioCue has three empty method bodies. Registered via DI on non-Android targets (#else branch in MauiProgram). Test project links NullAudioCue directly. 5 AudioCueTests all pass: PlayStart/PlayStop/PlayError do not throw, re-entrant calls safe
Does not block the UI thread All three AndroidAudioCue methods dispatch to Task.Run and return immediately (fire-and-forget). The background thread sleeps for durationMs + 50ms before releasing ToneGenerator; the UI thread is never involved. Code review: PlayStart/PlayStop/PlayError each return after _ = Task.Run(...) with no awaits on the calling thread
