1. What was implemented

The bulk Save/Load Workspace feature is upgraded from "re-open repos in fresh sessions" to a real state transfer with an optional, per-session handover note written by the wingman running as a REAL Claude session. Reopening a workspace can optionally seed each restored session from its note so a new Director picks up where the old one left off.

New files

FilePurpose
src/CcDirector.Core/Sessions/WorkspaceHandoverWriter.cs Core orchestration. For one session it drives the issue #509 SessionAskRunner engine (a real driver-backed session, never --print/-p), reads the wingman's answer back from the transcript, and persists it through HandoverScanner.WriteNew (the same parser the Handovers tab uses). Both the ask and the persistence are injected seams, so it is fully unit-testable with fakes. Per-session failures (repo gone, agent cannot answer, ask times out) return a WorkspaceHandoverResult.Fail rather than aborting the whole save.
src/CcDirector.Core.Tests/Sessions/WorkspaceHandoverWriterTests.cs Six unit tests covering the happy path, missing repo, empty answer, ask throwing, the handover prompt contract, and the title builder.

Changed files

FileChange
src/CcDirector.Core/Sessions/WorkspaceDefinition.cs Added WorkspaceSessionEntry.HandoverPath (nullable) - the optional reference to the handover doc the wingman wrote at save time. Null means "no handover; restore fresh".
src/CcDirector.Avalonia/SaveWorkspaceDialog.axaml(.cs) Added the "Include handover documents" toggle and a one-line explanation. Carries AgentKind through SessionData, and exposes IncludeHandovers + SelectedSessions so the caller can write notes off the dialog code path (each note is a slow real-session ask).
src/CcDirector.Avalonia/LoadWorkspaceDialog.axaml(.cs) Added the "Seed each session from its handover note" toggle, enabled only when the selected workspace actually has notes; exposes SeedFromHandovers.
src/CcDirector.Avalonia/MainWindow.axaml.cs WriteWorkspaceHandoversAsync (new) runs after the Save dialog when the toggle is on: writes one note per session via WorkspaceHandoverWriter, records each path on its entry, re-saves. LoadWorkspaceAsync gained a seedFromHandovers parameter and reuses the existing InjectHandoverPromptAsync to seed each restored session from its note.
src/CcDirector.Avalonia/WorkspaceProgressDialog.axaml.cs Added UpdateHandoverProgress so the save shows "Writing handover notes - session N/M" while the wingman works.
src/CcDirector.Core.Tests/WorkspaceStoreTests.cs Extended the load/save round-trip test to assert HandoverPath persists and that an entry without one round-trips as null.

2. Acceptance criteria - traced to the exact code

Acceptance criterionCode that satisfies itHow a human verifies (section 5)
AC1. Save with handovers writes one note per live session, via the wingman, with no /handover skill dependency and no -p. The Save dialog "Include handover documents" toggle sets SaveWorkspaceDialog.IncludeHandovers (SaveWorkspaceDialog.axaml.cs). When set, the File menu lambda calls WriteWorkspaceHandoversAsync (MainWindow.axaml.cs:957), which loops every selected session and calls WorkspaceHandoverWriter.WriteForSessionAsync. That method calls SessionAskRunner.AskAsync - a real driver-backed session (issue #509), whose BuildLaunchSpec never emits --print/-p - and then persists the answer with HandoverScanner.WriteNew. There is no call to the /handover skill anywhere in this path; the only --print/-p tokens in the new code are comments documenting their absence. The default agent kind ClaudeCode (ClaudeDriver) declares both TranscriptRead and PreassignedSessionId, the two capabilities the ask requires. Steps 1-4
AC2. Open with seeding starts each session from its note; with and without the toggles both work. The Load dialog "Seed each session from its handover note" toggle is enabled only when the selected workspace has at least one HandoverPath (LoadWorkspaceDialog.SetSeedAvailability) and sets SeedFromHandovers. The File menu lambda passes it to LoadWorkspaceAsync(workspace, seedFromHandovers) (MainWindow.axaml.cs). Inside the restore loop, a session is seeded only when seedFromHandovers && entry.HandoverPath is present, by reusing the existing InjectHandoverPromptAsync (the same seeding path the New Session dialog's Handovers tab already uses). With the toggle off, or for an entry without a note, the session restores fresh - today's behavior - so both toggle states work. Steps 5-7

4. Code-review notes (self-review against the standards)

  • No fallback programming. Per-session failures use the documented Result Objects for Expected Failures pattern (WorkspaceHandoverResult), not a swallowing catch - one bad session yields "no handover for that session" with a logged reason, the rest still save.
  • No null-forgiving operator (!) anywhere in the new code; null is handled with explicit is null / is not null checks.
  • Enterprise logging. Entry/exit/failure are logged via FileLog.Write with [ClassName] Method: context formatting throughout WorkspaceHandoverWriter and the new MainWindow methods.
  • Responsive UI. The slow per-session asks run off the dialog code path, behind a WorkspaceProgressDialog showing "Writing handover notes - session N/M"; the dialog is closed in a finally.
  • Core has no UI dependency. The orchestration lives in CcDirector.Core with injected seams; the WPF/Avalonia layer only wires it up.
  • VisualStyle compliance. The two new toggles use the dialog palette (foreground #CCCCCC primary / #888888 hint, accent buttons unchanged), match the existing dialog grid/margins, and add no new hard-coded colors outside the documented tokens. The Save dialog height was bumped 420->480 to fit the toggle.
  • Naming + async conventions. *Async suffixes, _camelCase fields, verb-noun methods, Task-returning helpers.
  • Tests. Every new public Core method is exercised by a unit test using Arrange-Act-Assert and the Method_Scenario_Result naming convention.
  • CenCon impact: no drift. This is a feature inside existing containers (Desktop app + Core sessions). It introduces no new container, no new external dependency, and no change to the security posture - it reuses the issue #509 real-session engine that is already on main. No docs/cencon/ manifest or security-profile update is required.

5. How a human should test this (runtime QA, deferred to the operator)

Build a slot exe from this branch and launch it via the project's per-session isolation flow (or the operator's normal test path), then:

AC1 - Save writes one note per live session via the wingman

  1. Open two or three sessions in the Director, each in a real git repo, and do a little work in each so there is something to summarize.
  2. File -> Save Workspace... Tick "Include handover documents (the wingman writes a note per session)", name the workspace, click Save.
    Expected: a progress dialog appears showing "Writing handover notes - session 1/N", "2/N", ... cycling through the sessions.
  3. Watch the director log (%LOCALAPPDATA%\cc-director\logs\director\director-*.log).
    Expected: for each session you see [WorkspaceHandoverWriter] WriteForSessionAsync followed by [SessionAskRunner] AskAsync and session closed, and finally WriteWorkspaceHandoversAsync: '...' re-saved with N/N handover notes. There must be no --print or -p in any launched command, and no /handover skill invocation.
  4. Inspect the saved workspace JSON (in the workspaces folder) and the handover folder.
    Expected: each session entry has a HandoverPath pointing at a real markdown note whose body is a plain-prose handover summary (what we did / current state / next steps), not a transcript dump.

AC2 - Open with seeding (and without)

  1. Close all sessions (or use a second Director). File -> Load Workspace..., select the workspace you just saved.
    Expected: the "Seed each session from its handover note" checkbox is now enabled (because the workspace has notes).
  2. Tick the seed checkbox and click Load.
    Expected: each session reopens and, once Claude has started, a prompt is auto-sent referencing its handover file (@...handover.md ... summary of what was done ... what we should work on next); the log shows [MainWindow] LoadWorkspaceAsync: seeding session ... from ... and InjectHandoverPromptAsync: sent handover prompt for each seeded session.
  3. Repeat the Load without ticking the seed checkbox (and, separately, load a workspace saved without handovers).
    Expected: sessions reopen fresh with no auto-sent prompt - today's behavior - confirming both toggle states work and the feature degrades cleanly.
