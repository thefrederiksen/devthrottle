Issue #290 - [Gateway] redeploy-gateway.ps1 shared build/stage tree can silently ship a stale build

Developer Agent proof report · branch issue-290-isolated-worktree-sha-assert · RESULT: PASS

What was wrong

scripts/redeploy-gateway.ps1 published the Gateway from the shared working tree into a shared stage dir (local_builds\_gw-deploy). Two redeploys, or a concurrent branch switch in the checkout, could clobber each other so the Gateway came up running an older build than HEAD — while GET /cockpit still returned HTTP 200. HTTP 200 proves a Gateway is up, not which build, so any post-deploy verification (including automated QA) could validate against stale code.

What was implemented (decided approach: option 1 + option 2)

  • Isolated-source publish. The deploy now pins HEAD to a commit SHA, creates a throwaway git worktree --detach at that exact commit, and publishes from the worktree into a per-run stage dir (GUID). A branch switch or edit in the main checkout after that point cannot change what this run ships, and two redeploys cannot corrupt each other's staged output. The worktree is always removed afterward.
  • Post-deploy build-identity assertion. After swap + relaunch + the existing GET /cockpit smoke check, the script reads the running Gateway's GET /healthz version field (which is AppVersion.Full and already carries +<sha> via SourceLink) and asserts its commit SHA matches the commit it just published. On mismatch it throws STALE BUILD: published commit X but the RUNNING Gateway reports Y and exits non-zero. No new endpoint was needed — /healthz already reports the SHA.
  • Latent bug surfaced & fixed. Publishing from a clean worktree exposed that --self-contained false alone trips NETSDK1150 with PublishSingleFile + a RID; the shared tree only avoided it by reusing cached build state. Fixed by setting the property -p:SelfContained=false so it flows to every project in the graph.

Existing behavior preserved: graceful POST /shutdown, file swap, relaunch --managed, and the GET /cockpit smoke check all still run and still gate success.

Verdict

I believe this is finished. The redeploy now publishes from an isolated git worktree pinned to the resolved HEAD commit and, after relaunch, asserts the running Gateway reports exactly that commit — turning "/cockpit 200" into "/cockpit 200 AND this is the build I shipped". The stale-deploy case fails loud, proven against a real self-hosted Gateway. Full solution build is clean (0 warnings, 0 errors).
