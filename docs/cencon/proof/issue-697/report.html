Issue #697 - Fall back to an ephemeral loopback port when the 7879-7898 range is exhausted

CcDirector.ControlApi enhancement Branch issue-697-ephemeral-fallback

What was implemented

When the fixed Control-API port range (7879-7898) is genuinely exhausted, the production loopback host now binds an ephemeral loopback port instead of disabling the Control API. The actual bound port is read back after Kestrel starts and is advertised through the existing channels (instances registration, Gateway registration, Tailscale Serve), so local and remote access keep working - just on a non-fixed port. A successful fallback is not a failure, so the desktop "Control API down / free a port" notification never fires; a genuine bind failure still reports exactly as before.

CenCon impact

No architecture or security drift. The Director keeps the same loopback-only + Tailscale Serve posture; the only difference on genuine exhaustion is that the bound port is OS-assigned instead of from the fixed range. The raw port remains loopback-only and is never exposed on the LAN or the Tailscale interface, so no security_profile.yaml rule is affected and no docs/cencon/ document needs updating.
