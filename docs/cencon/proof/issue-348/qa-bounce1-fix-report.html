Defect Summary (from QA)

In VoiceConversation.SpeakTurnAsync, the "reply" TurnUpdate was emitted at line 108 BEFORE the "summarizing" TurnUpdate at line 114. TalkPage's "reply" handler set VoiceStatusLabel.Text = "Speaking...", then "summarizing" fired and overwrote it with "Summarizing..." - after "Speaking..." was already visible. Actual sequence: Thinking... -> Speaking... -> Summarizing... instead of the required Thinking... -> Summarizing... -> Speaking...

Root Cause Analysis

Two interacting problems:

  1. VoiceConversation.cs: The "reply" and "summarizing" events were in the correct order (reply before summarizing), but there was no explicit "speaking" event to set the status AFTER summarization completed and BEFORE SpeakAsync was called.
  2. TalkPage.xaml.cs: The "reply" handler set BOTH the VoiceReplyLabel.Text (correct) AND VoiceStatusLabel.Text = "Speaking..." (wrong - mixed screen-update and status-update concerns in a single handler).

Because "reply" set "Speaking..." immediately when the reply text arrived (before summarization), and then "summarizing" overwrote it, the user saw the inverted order.

Fix Applied

1. VoiceConversation.cs (lines 107-125 after fix)

Added an explicit "speaking" TurnUpdate stage emitted AFTER SummarizeForVoiceAsync completes and BEFORE SpeakAsync. The "reply" event now only carries the raw text for screen display; it no longer implies a status transition. 

var rawReply = result.SpokenText();

// "reply" updates the on-screen reply label only - it does NOT advance the
// status indicator. The status must follow the actual processing order:
// Thinking -> Summarizing -> Speaking. See AC1.
onUpdate?.Invoke(new TurnUpdate("reply", rawReply));

// Step 5: summarize the raw reply into 2-3 spoken sentences of plain prose.
onUpdate?.Invoke(new TurnUpdate("summarizing", "Summarizing..."));
var spokenSummary = await SummarizeForVoiceAsync(session, rawReply, ct);

// "speaking" fires after summarization completes and before TTS begins,
// so the status label reads "Speaking..." only while audio is actually playing.
onUpdate?.Invoke(new TurnUpdate("speaking", "Speaking..."));
await SpeakAsync(session.TailnetEndpoint, spokenSummary, ct);

2. TalkPage.xaml.cs (lines 484-492 after fix)

Removed the VoiceStatusLabel.Text = "Speaking..." assignment from the "reply" handler. Added a new "speaking" case that sets the status label. The "reply" handler now only updates the screen label. 

case "reply":
    // Update the on-screen reply label only.
    // Status is updated by the "speaking" stage after summarization.
    VoiceReplyLabel.Text = u.Text;
    VoiceReplyCard.IsVisible = true;
    break;
case "speaking":
    VoiceStatusLabel.Text = "Speaking...";
    break;
