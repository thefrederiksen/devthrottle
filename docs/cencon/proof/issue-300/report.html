Defect found and fixed during the live demo (mapping table sharpened): the spec's D-3 pinned two templates (Basic: To Do/Doing/Done; Agile: New/Active/Closed) probed by the item's CURRENT state name. The live proof project Z_Archive_DataWarehouse turned out to use the Scrum template (To Do / In Progress / Done) - "To Do" matched Basic's initial state, but the Basic in-progress transition ("Doing") was rejected by DevOps. The current-state probe is therefore ambiguous between Basic and Scrum. Fix, staying within D-3's "static mapping, no dynamic discovery" intent: the table now pins all three common templates (Basic / Scrum / Agile) and the probe reads the work item TYPE's allowed state list (az devops invoke --area wit --resource workitemtypestates, one static lookup) to pick the row; an unrecognized state set still escalates needs-human. Documented in the implementation-loop skill and DEVELOPMENT_METHOD.md Section 7b.
