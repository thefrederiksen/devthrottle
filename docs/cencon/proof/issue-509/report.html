Proof - Issue #509

[Wingman] Drivers: ask a question via a real session and read back a formatted answer

Branch: issue-509-session-ask · Container: core_services (CcDirector.Core)

What was implemented

A single public helper, SessionAskRunner (in src/CcDirector.Core/Drivers/SessionAskRunner.cs), runs a short throwaway agentic question over a REAL driver-backed session and reads back a parseable answer, then tears the session down. This is the engine that replaces the metered claude --print one-shot path, so the work bills against the user's subscription (issue #509; the engine that #510 / #511 / #512 build on).

The flow uses the existing driver protocol, nothing new:

  1. IAgentDriver.BuildLaunchSpec — spawn arguments (never --print / -p);
  2. spawn an ISessionBackend and wait for the CLI to paint and settle;
  3. IAgentDriver.SubmitAsync — submit the prompt, which asks the agent to wrap its answer in a fixed delimiter pair;
  4. IAgentDriver.ReadWidgets — read the reply from the transcript (the answer channel, never the screen) once it stops growing;
  5. extract exactly the delimited block;
  6. tear the backend down (logged session closed).

Output-format contract (documented assumption resolved)

The agent is told to emit its answer between a unique delimiter pair, each on its own line: 

===DEVTHROTTLE-ANSWER-BEGIN===
<the answer>
===DEVTHROTTLE-ANSWER-END===

The helper extracts exactly the text between the first opening marker and the next closing marker, trimmed. If the markers are missing it fails loud (InvalidOperationException) rather than falling back to the whole reply (no-fallback rule). Other resolved assumptions: helper lives in CcDirector.Core (callable without the Gateway, does not reuse HostedAgent); default timeout 60s (SessionAskRunner.DefaultTimeout); a kind whose driver lacks TranscriptRead returns the same typed unsupported result as RawCli.
