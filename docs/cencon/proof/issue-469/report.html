Issue #469 - [Gateway] Device registration via local pairing code (per-device keys; no other path)

Developer Agent proof report. Replaces the shared "fleet token" enrollment with a deliberate, locally-authorized per-device enrollment. The Gateway mints a 4-digit, 5-minute, single-use pairing code shown ONLY in the GatewayApp host window (Anchor B - local presence is the root of trust). A new Director submits {deviceId, machineName, pairingCode}; the Gateway verifies and consumes the code, issues a unique per-device key, records the device, and the Director writes the key to its local credential file. The old shared-token-only path and the secret-embedding QR endpoints are closed.

What was implemented

  • Gateway (CcDirector.Gateway): PairingCodeService (4-digit / 5-min / single-use, CSPRNG, thread-safe), DeviceRegistry (unique per-device key per machine, persisted to devices.json under the config root, supports later revocation via status), DeviceEnrollmentEndpoint (POST /devices/register, GET /devices). AuthMiddleware accepts a per-device key as a valid Bearer alongside the shared token.
  • Old path closed: removed GatewayPairingEndpoint + PairingPayload (the /pair/qr.png and /pair/payload endpoints that embedded the shared token); the Cockpit "Connect a phone" panel is now a retired-notice. Shared-token-only registration is rejected.
  • GatewayApp (CcDirector.GatewayApp): PairingWindow - the corner-window pairing UI (Register button -> 4-digit code + live countdown -> joined confirmation), wired into the tray flyout as "Register a new device". The code is shown only here.
  • Director (CcDirector.Avalonia): ConnectToGatewayDialog - Gateway URL + 4-digit code entry (Join disabled until exactly 4 digits), POSTs enrollment, writes the per-device key, shows "Registered as <machine>". Reached from Settings > Gateway > "Connect to Gateway...".
  • Contracts: DeviceRegistrationRequest / DeviceRegistrationResponse / RegisteredDeviceDto.
  • ControlApi / Core: GatewayEnrollmentClient (the enroll call) and GatewayCredentialStore (writes the key to gateway-token.txt + config.json - the same file the local cc-* tools read). OperationResult<T> added per CodingStyle.
  • Tests: PairingCodeServiceTests (8), DeviceRegistryTests (6), GatewayCredentialStoreTests (3) - all pass.

CenCon impact

This changes the security posture: credential model moves from a single shared fleet token to unique per-device keys, and new-device enrollment now requires a local-presence pairing code instead of presenting the shared secret. This strengthens the posture (per-device revocability, no secret in a QR/link, attributable identity) and violates no blocking security rule. The docs/cencon/architecture_manifest.yaml / security_profile.yaml should be refreshed by the Support Agent to describe the per-device-key enrollment; no architecture container was added or removed (the new code lives inside the existing gateway, gateway_app, avalonia_ui, control_api, core_services, gateway_contracts containers).

I believe this is finished. The full solution builds clean (0 warnings, 0 errors), 17 new unit tests pass, all 10 API-level acceptance criteria pass against a real running GatewayHost, the loopback Control API accepts the on-disk credential (200), and all 5 UI states render as specified in the mockup. The secret-embedding QR endpoints are removed and shared-token-only enrollment is rejected.
