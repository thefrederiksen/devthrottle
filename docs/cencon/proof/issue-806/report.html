Issue #806 - Mobile Foundation: React + TypeScript PWA served by the Gateway

Developer Agent proof report. Branch issue-806-mobile-foundation. Repo thefrederiksen/devthrottle. This is Issue 1 (foundation) of the mobile app plan (docs/architecture/mobile/IMPLEMENTATION.md).

What was implemented

  • New mobile/ project at the repo root: Vite + React + TypeScript + vite-plugin-pwa. Outside the C# src/ tree (not in the .NET solution).
  • Release-gated MSBuild target (BuildMobileApp) on CcDirector.Gateway.csproj that runs npm ci && npm run build and stages mobile/dist/** into the Gateway's wwwroot/m/. Gated by RunMobileBuild (Release / publish or explicit -p:BuildMobile=true), so a routine Debug dotnet build never runs npm.
  • Gateway serves /m as static files with the per-machine token injected into index.html (the /voice __GATEWAY_TOKEN__ pattern); a single catch-all handler also does SPA deep-link fallback. The shell is exempt from the global auth gate (it carries no secret), so it loads with auth on or off.
  • OpenAPI document at /openapi/v1.json (AddOpenApi / MapOpenApi); /sessions annotated .Produces<List<SessionDto>>. openapi-typescript generates the typed client into mobile/src/api/schema.ts.
  • Home / roster page: a "needs you" group above the full session list, using the Gateway /sessions aggregator and a faithful TypeScript port of the C# SessionOrdering triage. Tapping a row opens a session-detail placeholder bound to the id.
  • Mobile-redirect middleware: a phone User-Agent browser-navigation (Accept: text/html), not already under /m, gets a 302 to /m/; a desktop UA falls through to the Cockpit unchanged.
  • PWA: installable (web app manifest), service worker precaches the app shell and caches the last /sessions so a cold offline open renders the last-known roster.

CenCon impact

No architecture/security drift requiring a docs/cencon/ manifest change beyond this proof.
