Issue #646 - Workstation install requires pairing to a Gateway before finishing

Gateway Centralization Phase 3. A Workstation-role install ("I already have a gateway") now cannot finish without a verified gateway connection. The mandatory pairing step reuses the existing device pairing-code flow (POST /devices/register) and persists the issued per-device key to config.json.

This proof was refreshed after rebasing onto the merged sibling PR #679. PR #679 introduced tools/cc-director-setup/Services/WizardStepFlow.cs - a pure, unit-testable navigation policy that drops the in-wizard Sign-in step for the Workstation role. This issue's mandatory "Connect" step is now integrated INTO that same WizardStepFlow policy as the exact inverse of Sign-in (Connect applies to a fresh Workstation install; Sign-in applies to a Gateway install). There is ONE role-aware rail and no parallel/ad-hoc navigation logic.

How it was integrated onto WizardStepFlow

  • WizardStepFlow now models BOTH role-aware steps: StepSignIn = 3 (Gateway-only, from #679) and the new StepConnect = 5 (fresh-Workstation-only). SignInApplies(role) and ConnectApplies(role, isUpdate) are exact inverses; VisibleSteps/NextStep/PrevStep take (role, isUpdate) and skip whichever role-aware step does not apply, with no gaps. The full id set is 1 Welcome, 2 Prerequisites, 3 Sign in, 4 Privacy, 5 Connect, 6 Skills, 7 Install, 8 Complete.
  • MainWindow drives the rail through those WizardStepFlow members only - the ad-hoc 8-step rail, RequiresGatewayPairing, ApplyConnectStepVisibility, and the duplicate NextVisibleStep/PrevVisibleStep from the pre-rebase branch were removed. The sidebar uses #679's position-based rendering, so the visible circles renumber 1..N and skipped rows + their trailing connector lines collapse with no gap, for either role.
  • GatewayPairingRunner (engine) and GatewayConnectStep (UI) are unchanged from this issue's original work: verify reachability + pairing via /devices/register, issue a per-device key, persist gateway URL + key to config.json via GatewayCredentialStore.SaveEnrolledKey, and gate the wizard's Next on IsVerified/PairingVerified. They are now wired through the WizardStepFlow rail.

CenCon impact

No architecture or security drift. The gate reuses the existing /devices/register enrollment surface, the existing per-device-key trust model, and GatewayCredentialStore (the same credential file + config.json gateway block the Director already uses). No new endpoint, no new secret path, no new pairing mechanism. The rebase consolidated navigation into the one WizardStepFlow policy #679 introduced (less code, fully unit-tested), removing the duplicate rail logic.

I believe this is finished: the mandatory Workstation gateway-pairing step is integrated onto the single WizardStepFlow rail (Connect as the inverse of Sign-in), all five acceptance criteria are met and proven against a disposable gateway, the rail tests and engine tests pass, and the full solution builds clean.
