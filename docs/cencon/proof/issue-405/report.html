Issue #405 - [Voice] Client resilience: poll-loop + submit retry/backoff (survive spotty data)

Branch: issue-405-voice-resilience  |  Surface: phone client (.NET MAUI), phone/CcDirectorClient  |  Proof type: xUnit unit tests against a fake voice-turn channel (pure-logic phone-client change - no Director slot needed).

What was implemented

The Gateway async voice-turn pipeline is designed to survive a signal drop (submit once, the Gateway caches the result ~10 min, the phone just polls), but the client did not implement that resilience: a single dropped poll or submit threw straight out of the turn. This change makes the client live up to the design.

  • New resilient seam VoiceTurnRunner holding the submit-retry + poll-retry/backoff/deadline loop, depending only on a new IVoiceTurnChannel interface (implemented by DirectorVoiceClient over real HTTP). The clock and the inter-poll delay are injected (VoiceTurnRetryPolicy) so the loop is MAUI-free and timer-free and unit-tested off-device with instant waits.
  • Poll retry/backoff: each poll is wrapped; a transient failure (network error, 5xx, request timeout) waits a backoff (1.5s steady -> up to ~5s) and re-polls rather than throwing out of the loop.
  • Deadline aligned with the server TTL: the loop gives up only at an overall ~10-minute deadline (VoiceTurnTimeoutException) or on a terminal stage.
  • Transient vs terminal classification: a 5xx is transient; the error stage and a 410 Gone (session exited) stop immediately; a 404 after holding a valid turn id is a clean VoiceTurnExpiredException ("please resend"), never an unhandled crash. DirectorVoiceClient.PollVoiceTurnAsync now throws VoiceTurnHttpException carrying the status code so the runner can classify.
  • Submit retry: VoiceTurnRunner.SubmitAsync retries a bounded number of times (default 4) with backoff before surfacing a failure, so a drop mid-upload does not discard the recorded clip.
  • VoiceConversation.SpeakTurnAsync now delegates its agent path to the runner, keeping the same onUpdate stage callbacks and reply-audio playback.
