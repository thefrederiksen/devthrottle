Issue #391 - Machine-level agent tool library

Developer Agent proof. CenCon Development Method. Branch issue-391-agent-tool-library.

What was implemented

  • Built-in catalog of the 5 known agent tools (ClaudeCode, Pi, Codex, Gemini, OpenCode) with a recommended default command-line preset and default model per tool - new AgentToolCatalog in CcDirector.Core/Agents.
  • Claude Code default changed to the STANDARD command line (no --dangerously-skip-permissions). The "Automatic (skip permissions)" preset is offered but OFF by default. AgentOptions.DefaultClaudeArgs default changed from the skip-permissions flag to an empty string.
  • Per-tool machine-level config (selected preset, free-text args override, default model, enabled flag) persisted in config.json under agent.tools.<key> - new AgentToolConfig in CcDirector.Core/Configuration, written via CcDirectorConfigService.MergePatch (machine-level, no gateway call).
  • Tools UI: the Settings dialog "Agents" tab (the existing machine-level agent-CLI library surface, backed by AgentKind/ToolDetectionService) gains, per tool: a command-line preset dropdown, a default-model field, a free-text args override, and an Enabled checkbox. The per-tool Test button (PASS/FAIL with version) was already present and is unchanged.
  • Launch wiring: App.ApplyConfiguredToolPresets composes Claude's effective default args (preset + --model) from the saved config at startup; saving in the dialog re-applies it live. The New Session dialog now passes null (not an empty string) when it specifies no Claude flags, so the configured default applies.
Note on the issue's "extend ToolsView" assumption. The issue named Controls/ToolsView.axaml, but that page is the unrelated cc-* CLI catalog (cc-outlook, cc-vault, ... 30+ tools); it is not tied to AgentKind. The machine-level agent-CLI library (the AgentKind set, per-tool path, Detect/Test, backed by ToolDetectionService) already lives in the Settings dialog "Agents" tab. The new per-tool preset/model/enabled fields were added there - the architecturally correct, lowest-risk home - per the issue's own flagged assumption ("This extends the existing ToolsView ... a design choice flagged for the human"). No new page was created.

Build & tests

  • BUILD dotnet build cc-director.sln - Build succeeded, 0 Warning(s), 0 Error(s).

Root cause addressed by serializing the CC_DIRECTOR_ROOT-mutating tests into one non-parallel xUnit collection (ConfigEnvSerial) and pinning that test fixture's stand-in shell args. Single-threaded the full suite was always green, confirming a load/timing artifact, not a logic regression.

CenCon impact

No architecture or security drift. The change is additive (new Core config model + catalog + Settings UI fields). The Claude default change reduces privilege (a fresh Claude no longer auto-skips permission prompts), which strengthens the security posture rather than weakening it. No new container; no blocking security rule touched.
