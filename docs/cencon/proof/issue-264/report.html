Issue #264 - Deterministic rewrite of the flaky PromptInjectionWatcher wingman-source test

Repo: thefrederiksen/cc-director · Branch: loop/264-flaky-promptinjection-test · Developer Agent (CenCon)

What was implemented

The test SessionStatusWingmanTests.PromptInjectionWatcher_pushes_extracted_text_via_wingman_source was [Fact(Skip)] because it was timing-flaky: it wrote bytes to the buffer and slept 1500 ms, relying on the byte-arrival debounce to fire a single scan that happened to read a resolved grid. If that one scan read a not-yet-yielding grid it never re-scanned, so the push intermittently never landed; and the final assert on PendingPromptText raced the session's own source="user" write of the same field.

The test was rewritten to be deterministic, with no production behavior change (RequestImmediateScan() already existed at src/CcDirector.Core/Wingman/SessionStatusWingman.cs:340). The rewrite:

  1. Drives the internal PromptInjectionWatcher directly (reachable via the existing InternalsVisibleTo("CcDirector.Core.Tests")), so there is no spurious byte-arrival scan racing the explicit one.
  2. Confirms the grid is resolved before triggering: it asserts that the exact inputs the watcher's tick reads (SnapshotScreenRowsWithCursor() + PromptInputLineExtractor.ExtractUserAuthoredInput(...)) already extract to the expected text. The single scan therefore cannot read a not-yet-yielding grid.
  3. Fires exactly one scan via RequestImmediateScan() and waits on the "wingman"-source event through a TaskCompletionSource (signalled by the push). There is no fixed Task.Delay-then-assert; the 10 s Task.WhenAny is only a hang guard.
  4. Asserts on the captured "wingman"-source value, not PendingPromptText (which the session also writes with source="user" - the original race).
