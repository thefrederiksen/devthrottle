Primary defect: register response is parsed from the wrong JSON shape

Severity: blocking. Device registration fails on every attempt against the real (merged) DevThrottle cloud endpoint, because the desktop client parses a flat response body while the merged cloud endpoint returns an enveloped body.

What the merged cloud endpoint actually returns. The register endpoint is implemented and merged at thefrederiksen/devthrottle_internal: website/api/v1/devices.js. Both the rotate (200) and insert (201) paths return an enveloped body with the key and the masked record nested under data:

// website/api/v1/devices.js
return res.status(200).json({ data: { device_key: key.raw, record: toRecord(updated) } });   // line 132
return res.status(201).json({ data: { device_key: key.raw, record: toRecord(inserted) } });  // line 142

So the real register response is:

{ "data": { "device_key": "dtd_live_...", "record": { "id": "...", "name": "...", "platform": "...", ... } } }

What the desktop client parses. CcDirector.Core/Account/DeviceRegistryClient.RegisterAsync reads device_key and the masked record fields from the root object, not from data:

// DeviceRegistryClient.cs (lines 254-263)
var root = JsonNode.Parse(json) as JsonObject
    ?? throw new InvalidOperationException("device register response was not a JSON object");

var deviceKey = StringField(root, "device_key")
    ?? throw new InvalidOperationException("device register response had no string 'device_key'");
var device = ParseRecord(root);

Consequence. Against the real cloud, root is { "data": { ... } }. StringField(root, "device_key") is null (the key lives under data.device_key), so RegisterAsync throws InvalidOperationException("device register response had no string 'device_key'") on every call. The Gateway never obtains a key, never stores one, and the heartbeat then skips (it requires a stored device key), so last-seen never advances. The failure is permanent and deterministic - it is a parse mismatch, not a transient cloud outage.

Why the unit tests do not catch it. The developer's stub (DeviceRegistryClientRegisterHeartbeatTests) returns a flat body ({"id":...,"name":...,"device_key":"..."}) that matches the buggy parser, so all 11 Core tests pass while masking the production defect. The test even asserts this flat shape is "the one cloud-contract assumption proven here against the stub" - but it proved the wrong shape. This is the exact stub-matches-the-bug trap.
