Proof - Issue #857

[Gateway] On sign-in, register this Gateway as a device with the cloud and store the per-device key

Branch issue-857-register-on-signin · Repo thefrederiksen/devthrottle · Built on the merged DeviceRegistryClient (#854) and the Gateway sign-in flow (#637).

QA bounce fix (2026-06-30) - cloud "data" envelope mismatch

Defect (QA FAIL): DeviceRegistryClient.RegisterAsync parsed device_key and the record from the root of the JSON response, but the merged cloud endpoint (devthrottle_internal: website/api/v1/devices.js) returns them enveloped under data: { "data": { "device_key": "...", "record": { ... } } }. Against the real cloud, register threw "device register response had no string 'device_key'" on every call, so the key was never stored and the heartbeat short-circuited - failing all five acceptance criteria in production. The original stub tests passed only because they used the flat (buggy) shape.

Fix (verified against the merged cloud contract devices.js):

  • RegisterAsync now reads device_key from data.device_key and the masked record from data.record (cloud devices.js lines 132/142: json({ data: { device_key, record } })).
  • ListDevicesAsync (GET, merged in #854) had the same latent envelope bug - it parsed a top-level JSON array, but the cloud returns { "data": [ ... ] } (devices.js line 178). Fixed in this PR for consistent envelope handling. This corrects merged #854 behaviour (the account device-list proxy GET /account/devices would otherwise throw against the real cloud).
  • HeartbeatAsync (POST, returns { "data": { "recorded": true } }) and RevokeDeviceAsync (DELETE, returns { "data": { "id":..., "revoked":true } }) are status-code-only - they never parse the response body (200 -> true, 404 -> false) - so they were already correct and needed no client change.
  • Introduced two shared helpers, DataObject / DataArray, that unwrap the { "data": ... } envelope and throw clearly when it is absent (no fallback - a contract-violating body surfaces as a clear failure).
  • Stub fixtures corrected to the real enveloped shape so the tests now actually CATCH this mismatch: DeviceRegistryClientRegisterHeartbeatTests (register -> { data: { device_key, record } }, heartbeat -> { data: { recorded: true } }), GatewayDeviceRegistrationTests stub (same), and AccountDevicesEndpointTests.StubCloudHandler (GET -> { data: [ ... ] }, DELETE -> { data: { id, revoked } }). With the corrected parser these fixtures are required; reverting the parser to read from root now fails the tests.

Result: build clean (0 warnings, 0 errors); 11 Core register/heartbeat tests + 30 Gateway device tests pass against the corrected real-cloud envelope fixtures (see fix-test-output.txt). The live signed-in real-cloud round-trip remains the QA gate.

What was implemented

When the Gateway signs in (the #637 browser loopback completion path), and as a first-launch/retry safety net on the background heartbeat, the Gateway now registers itself as a device with the DevThrottle cloud account and stores the cloud-issued per-device key locally. A periodic heartbeat keeps the account device list's last-seen fresh. Every cloud call degrades gracefully: a failure leaves the Gateway signed in and running, logs the failure, and retries on the next heartbeat. The raw per-device key is never written to the log (security rule DT-05).

Files changed

  • src/CcDirector.Core/Account/DeviceRegistryClient.cs - extended the existing injectable-HttpClient client with RegisterAsync (POST /api/v1/devices/register) and HeartbeatAsync (POST /api/v1/devices/heartbeat), plus the CloudDeviceRegistrationRequest / CloudDeviceRegistrationResult records. No second client or base URL - reuses the same DEVTHROTTLE_API_URL egress base.
  • src/CcDirector.Gateway/Account/GatewayInstallId.cs (new) - the Gateway's single stable per-machine install id, persisted at config/director/gateway-install-id.txt. The idempotency anchor (no pre-existing Gateway install id existed to reuse; this establishes the one canonical id, not a competing one).
  • src/CcDirector.Gateway/Account/GatewayDeviceKeyStore.cs (new) - stores the issued per-device key + its install id at config/director/account-device-key.json (the #469 device-key location). Key value never logged.
  • src/CcDirector.Gateway/Account/GatewayDeviceRegistrationService.cs (new) - EnsureRegisteredAsync with the two idempotency guards (in-run + relaunch/key-present).
  • src/CcDirector.Gateway/Account/GatewayDeviceHeartbeatService.cs (new) - the background heartbeat loop (mirrors GatewayTokenRefreshService): ensure-registered then advance last-seen; single boundary try/catch = graceful degradation.
  • src/CcDirector.Gateway/Account/GatewaySignInService.cs - added an optional best-effort onSignedIn hook fired (detached) after a successful sign-in.
  • src/CcDirector.Gateway/GatewayHost.cs - constructs the two services when the credential service is present, wires the sign-in hook, starts the heartbeat in StartAsync, disposes it in StopAsync.
  • Tests: CcDirector.Core.Tests/Account/DeviceRegistryClientRegisterHeartbeatTests.cs, CcDirector.Gateway.Tests/Account/{GatewayDeviceRegistrationTests, GatewayInstallIdTests, GatewayDeviceKeyStoreTests}.cs.

QA gate - the one thing the automated proof cannot cover

Live real-cloud signed-in round-trip.

The cloud device endpoints live in the separate, already-merged repo thefrederiksen/devthrottle_internal (#81 register, #83 heartbeat). The automated proof here drives a faithful in-process stub of that contract; staging a real signed-in DevThrottle account from automation is not available to this run, so the proof does not fake a live call. QA must verify, against the real cloud with a real signed-in Gateway:

  1. A fresh sign-in creates exactly one device row (GET /account/devices or the cloud directly).
  2. The stored key file appears under %LOCALAPPDATA%\cc-director\config\director\account-device-key.json (value redacted) and no Gateway log line carries the raw key.
  3. A relaunch does not add a second row.
  4. Two GET /account/devices reads a heartbeat apart show a newer last-seen.
  5. Confirm the cloud's register response field name for the raw key. This implementation parses device_key (the one cloud-contract field-name assumption; the masked record fields id/name/platform/.../key_prefix/key_last4 match the existing #854 list shape). If the cloud names it differently, only the one string constant in RegisterAsync needs to change.
