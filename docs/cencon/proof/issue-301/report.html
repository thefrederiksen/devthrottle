What was implemented

WorkListStore (the fleet's named work-list store, #273) now persists every list - items, order, consumer claim - to ONE plain JSON file, worklists.json, in the Gateway data dir (%LOCALAPPDATA%\cc-director\worklists.json, the keyvault.json precedent / D-1), and reloads it on startup (D-2):

  • Write-through on every mutation (Create / AppendItem / Reorder / RemoveItem / Claim / Release) with an atomic temp-file + rename; a failed save is a logged error that propagates, never a silent skip.
  • Reload-on-start: missing file = empty store + log line (the normal first boot). Corrupt file = quarantine: the bytes are renamed to worklists.json.corrupt-<stamp> (preserved for the operator, never silently overwritten), an explicit error is logged, and the store starts empty so the Gateway still boots. Chosen recovery behavior + why: refusing to boot would turn one bad file into a fleet-wide Gateway outage; quarantining keeps the Gateway up while losing nothing - the operator can inspect/hand-restore the quarantined file. The next write-through writes a fresh file at the original path.
  • Stale-claim policy (D-2): a persisted consumer claim is by definition stale after a restart (the claiming runner died with the Gateway). On load it is released with an explicit FileLog line naming the list and the dead consumer token, and the released state is persisted immediately. A new runner re-claims and continues from the persisted order. No auto-respawn of dead runners (out of scope per the spec).
  • Testability (D-3): the store path is a REQUIRED constructor argument (no caller can silently land on the real user's file); GatewayHost gains a workListsPath override mirroring keyVaultPath. Every Gateway test fixture passes an isolated temp path. The data-dir override for whole-process isolation already existed (CC_DIRECTOR_ROOT) and is what the isolated-instance proofs below use.

Files changed: src/CcDirector.Gateway/WorkListStore.cs, src/CcDirector.Gateway/GatewayHost.cs, src/CcDirector.Gateway.Tests/WorkListStorePersistenceTests.cs (new), plus isolated workListsPath in 19 existing test files.
