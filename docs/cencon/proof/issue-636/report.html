Issue #636 - Account credential service on the Gateway (store, validate, identity)

Developer Agent proof. Branch issue-636-gateway-credential-service. Gateway Centralization Phase 2 foundation: a Gateway-hosted credential service that stores the DevThrottle access-plus-refresh token pair encrypted at rest, validates it locally, and exposes the signed-in identity. The existing CcDirector.Core.Account types (WindowsProtectedTokenStore, JwtAccessTokenValidator, DevThrottleAccountService, JwtIdentityReader) are reused as-is from the Gateway, not duplicated.

What was implemented

  • New CcDirector.Gateway/Account/GatewayAccountFactory.cs - the Gateway-side wiring that builds the reused DevThrottleAccountService over a WindowsProtectedTokenStore and an AuthEventLog rooted under the Gateway config directory, honoring the DEVTHROTTLE_JWT_SIGNING_SECRET signing secret and seeding from DEVTHROTTLE_TEST_SEED_TOKEN. The access/refresh tokens are never logged on any path.
  • CcDirector.Core/Storage/CcStorage.cs - added GatewayDevThrottleCredentialBlob() (config/gateway/devthrottle-credential.bin) and GatewayDevThrottleAuthEventsLog(), keeping the Gateway's account storage distinct from the per-Director store under config/director.
  • CcDirector.Gateway/GatewayHost.cs - constructs the credential service and exposes it as the Account property. On Windows it builds the Windows Data Protection-backed service; on a non-Windows host it stays null (the operating-system credential store is Windows-only for now, the issue's stated assumption). A test seam lets a test inject its own service over an isolated store.
  • CcDirector.Gateway.Tests/Account/GatewayAccountServiceTests.cs (8 tests) plus a local GatewayTestJwt helper.

CenCon impact

No architecture drift: the change reuses the existing CcDirector.Core.Account container types and adds a thin Gateway wiring class plus two storage-path helpers. It strengthens DT-01 (Insecure Data Storage) - the Gateway credential is encrypted at rest via Windows Data Protection - and respects DT-05 (Information Disclosure) - tokens are never logged (proven by criterion 4). No blocking security rule is violated; no docs/cencon/ file required an update.
