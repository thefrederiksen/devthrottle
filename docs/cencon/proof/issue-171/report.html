Issue #171 - FileLog midnight rollover loses all writes

Developer Agent proof report (CenCon Development Method). Branch issue-171-filelog-rollover. Area: CcDirector.Core.

Confirmed root cause

Both suspects in the issue were confirmed by reading src/CcDirector.Core/Utilities/FileLog.cs:

  • (a) Writer thread dies on any exception. The entire consuming foreach sat inside a single try. After the midnight rollover, any exception (or any transient write failure) escaped the loop and permanently terminated the background writer thread, so all logging stopped - the new day's file stayed at 0 bytes.
  • (b) Busy queue never flushes. The writer used AutoFlush=false and flushed only when _queue.Count == 0. A busy long-running Director keeps the queue non-empty, so buffered lines were never flushed to disk.

What was implemented

FileChange
src/CcDirector.Core/Utilities/FileLogWriter.cs (new, internal) Extracted the dequeue/rollover/flush engine with an injectable clock (Func<DateTime>) so the rollover is unit-testable. Per-line try/catch inside the loop logs and continues (the thread outlives any single bad write). Bounded flush: flush when the queue drains or when more than FlushInterval (1s) has elapsed since the last flush. Opens the file with FileShare.Read so live log viewers can read it.
src/CcDirector.Core/Utilities/FileLog.cs Now a thin static facade over FileLogWriter wired to DateTime.Now and the real log directory. Public API (Start/Write/Stop/CurrentLogPath) unchanged - no consumer changes required.
src/CcDirector.Core.Tests/Utilities/FileLogWriterTests.cs (new) Four regression tests pinning the three acceptance criteria plus path computation.

Live-app proof (slot 5)

Built cc-director5.exe via scripts\local-build-avalonia.ps1 -Slot 5 and launched it via the cc-director-launch scheduled task (never from the agent's own process tree). The Director (PID 17332) wrote through the refactored FileLog -> FileLogWriter engine: its live log went from 0 bytes to 16,684 bytes and flushed under continuous startup load - the exact "0-byte stays 0" symptom no longer occurs - and the file was readable with shared access while the writer held it open. The slot-5 Director was then stopped (path-verified) and the task removed. (The raw live log is not committed because it contains machine-local user paths - PII for this public repo.)
