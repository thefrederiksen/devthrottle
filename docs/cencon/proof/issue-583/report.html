Proof Report - Issue #583

[Account] Director credential store and offline logged-in check - store account tokens in the operating system credential store, verify offline, refresh online.

What was implemented

A greenfield DevThrottle credential and authentication service in the Core layer (CcDirector.Core, the core_services container) plus the Windows operating system credential store binding (the native_apis container). This is the single foundation the startup gate (#580), first-run login (#581), and account area (#582) build on. It is distinct from ClaudeAccountStore (the Claude sign-in account, a different concern).

  • Account/DevThrottleTokens.cs - the access-plus-refresh token pair record.
  • Account/IProtectedTokenStore.cs - the operating system credential store binding (interface, so the macOS Keychain can be a later drop-in).
  • Account/WindowsProtectedTokenStore.cs - the Windows Data Protection (DPAPI) implementation: serializes the pair to JSON, encrypts it with the current user's data-protection key via System.Security.Cryptography.ProtectedData (CurrentUser scope), and writes an opaque binary blob. The raw token never touches disk in plain text.
  • Account/JwtAccessTokenValidator.cs - local-only validation (no network): verifies the access token's HMAC-SHA256 signature against the configured signing secret and checks exp. Distinguishes a genuinely-ours expired token (renewable) from a tampered/wrong-signature token (never logged in).
  • Account/ITokenRefresher.cs - the one network-touching seam: exchanges a refresh token for a fresh pair. The offline logged-in check never calls it.
  • Account/AuthEventLog.cs - append-only JSON-lines authentication-floor event log (logged-in on first store, logout on clear). Records only kind + timestamp, never the token.
  • Account/DevThrottleAccountService.cs - the orchestrator: StoreTokens, IsLoggedIn() (local, no network), RefreshIfNeededAsync (background renew, keeps the cached credential when offline), Logout.
  • Path helpers added to Storage/CcStorage.cs; the System.Security.Cryptography.ProtectedData package added to CcDirector.Core.csproj; the new components recorded in docs/cencon/architecture_manifest.yaml.

How it was proven

Two independent gates:

  1. Unit tests - 22 tests in CcDirector.Core.Tests/Account/ (the full Account namespace ran green: 45 passed, 0 failed - the count includes the pre-existing ClaudeAccountStore tests).
  2. Live end-to-end proof harness - a standalone program that exercises the REAL DevThrottleAccountService with the REAL WindowsProtectedTokenStore (real Windows DPAPI) against a real temporary config directory - no storage test doubles. Output captured in proof-output.txt; the service's own enterprise log lines captured in director-log-excerpt.log (paths redacted of the local username). The on-disk DPAPI blob begins with the canonical Windows DPAPI magic header 01000000D08C9DDF0115D1118C7A00C0, confirming real operating-system encryption.

CenCon impact

Added the DevThrottleAccountService family to the core_services container and the Windows Data Protection credential store to the native_apis container in docs/cencon/architecture_manifest.yaml (both were named in the issue's Affected Containers). No blocking security rule (CCD-01..CCD-07 in security_profile.yaml) is violated: no hardcoded credentials, no null-forgiving operator, no .Result/.Wait(), no PII paths in source; the token is never written to a log. The signing secret is supplied to the validator at construction (from configuration / the Key Vault at the wiring point), never hardcoded.

Scope note (assumptions confirmed by the issue): the browser sign-in page, the token hand-back protocol, and account resolution are owned by the internal-repository backend and are OUT of this issue. This Director-desktop side is implemented and proven against a test-issued HS256 token pair, exactly as the issue authorizes. The live backend exchange is supplied later as an implementation of ITokenRefresher (wired with #581). The tokens are treated as Supabase tokens (a signed access JWT plus a long-lived refresh token).

Conclusion

I believe this is finished. Every acceptance criterion is met, proven twice (unit tests + a live run against the real operating-system credential store), the full solution builds clean (dotnet build cc-director.sln - 0 warnings, 0 errors, under TreatWarningsAsErrors), and the CenCon manifest is updated in the same change.
