Issue #856 - Add a device becomes account sign-in (QR/deep-link); pairing code demoted to fallback

Surface: Gateway tray (CcDirector.GatewayApp) + CcDirector.Gateway + CcDirector.Core. Branch: issue-856-add-device-signin. Build: succeeded, 0 warnings, 0 errors (dotnet build cc-director.sln).

What was implemented

  • Re-framed the "Add a device" window (PairingWindow.axaml/.axaml.cs). The idle state now LEADS with "Sign in with your DevThrottle account": a QR code plus a plain sign-in link, with copy stating that signing into the same account is what joins the device here. The 4-digit pairing code from #469 is moved below a clearly secondary heading, "NO ACCOUNT HANDY? USE A PAIRING CODE INSTEAD", behind a "Show a pairing code" button.
  • QR carries a plain URL only - never a secret. New helper CcDirector.Gateway.Pairing.DeviceSignInQrCode.RenderPng renders the QR with QRCoder (already referenced) and rejects anything that is not an absolute http/https URL, so a pairing code or token can never be encoded. (Consistent with #469 having removed the old secret-embedding QR endpoints.)
  • Single source of truth for the sign-in URL. New FirstRunLoginCoordinator.ResolveSignInBaseUrl() returns the DEVTHROTTLE_SIGNIN_URL value or the documented default https://devthrottle.com/signin - with NO loopback callback and NO secret. BuildSignInUrl was refactored to use it.
  • Tray action renamed from "Register a new device" to "Add a device" (GatewayTrayController).
  • No new endpoint added - the web/QR sign-in start endpoint already exists as POST /account/sign-in (#853), shared with the Cockpit sign-in buttons, so this issue reuses it per its own scope note ("if not already added there").
