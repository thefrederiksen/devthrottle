Issue #637 - Gateway sign-in: browser loopback flow on first launch

Developer Agent proof. Branch issue-637-gateway-signin. Gateway Centralization Phase 2: the browser loopback sign-in that used to run on the Director now runs ON THE GATEWAY (plan decision #2 - forced sign-in at the Gateway's first launch). The reused CcDirector.Core.Account types FirstRunLoginCoordinator + LoopbackLoginListener (the same browser-opener + loopback mechanism the Director used) are driven from the Gateway, and the captured token is stored through the Gateway credential service from issue #636 (GatewayHost.Account). Tokens are never logged (security rule DT-05).

What was implemented

  • New CcDirector.Gateway/Account/GatewaySignInService.cs - drives the reused Core FirstRunLoginCoordinator over the Gateway-hosted credential service: opens the system browser at the configured sign-in address, captures the credential the sign-in completion hands back on the loopback callback, and stores it through the credential service. Adds the Gateway-side single-flight guard (an auto-prompt at launch and a tray click never run two browser hand-offs at once) and the IsSignedIn() / GetIdentity() queries the tray reads. A no-op login reporter is injected (login telemetry forwarding is the Gateway's own relay, issues #628/#630, not this flow). Access and refresh tokens are never logged on any path.
  • New CcDirector.Gateway/Account/GatewaySignInTraySurface.cs - the pure, UI-framework-free tray decisions (ShouldPromptOnLaunch, ShouldShowSignInAction, AccountRowValue), kept in the library so the tray surface logic is unit-testable without an Avalonia UI thread.
  • CcDirector.Gateway/GatewayHost.cs - constructs the GatewaySignInService over Account and exposes it as the SignIn property (null on a host with no credential service - a non-Windows host).
  • CcDirector.GatewayApp/GatewayTrayController.cs (tray surface) - on host start, auto-prompts the browser sign-in when the Gateway has no stored credential (PromptSignInIfNeeded); adds the "Sign in to DevThrottle" flyout action that starts/retries it; adds an Account status row; cancels any in-flight sign-in on quit. The sign-in run is detached with a boundary try-catch so a failure never crashes the tray and stays retryable.
  • CcDirector.Gateway.Tests/Account/GatewaySignInServiceTests.cs - 10 tests (7 service + 3 tray surface), reusing the existing GatewayTestJwt helper and the injected-browser-opener + real-loopback + stand-in-completion pattern of the Core FirstRunLoginCoordinatorTests.

CenCon impact

No architecture or security drift. This reuses the #636 Gateway credential service and the Core FirstRunLoginCoordinator + LoopbackLoginListener as-is; it adds a Gateway-side service plus a tray surface within the already-documented Gateway containers. Tokens are never logged (DT-05 honored, carried over from #636). The relocation matches the documented Phase 2 plan decision #2 (forced sign-in at the Gateway's first launch).

Conclusion

I believe this is finished. All five acceptance criteria and the token-never-logged security rule are met and proven end to end against the real tools/devthrottle-dev-signin tool, with 10 dedicated tests and the full Gateway suite passing, on a clean full-solution build (0 warnings, 0 errors).
