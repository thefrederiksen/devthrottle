2. Design scrutiny: endpoint-level enforcement vs global middleware

The Developer enforced the token AT THE ENDPOINT on the grounds that production gateways run authEnabled=false. QA scrutinized and CONFIRMED this design call:

  • Holds in production mode. Both production construction sites - GatewayTrayController.StartHostAsync (new GatewayHost(_port)) and GatewayWorker.ExecuteAsync - pass token: null, so GatewayHost.Token = GatewayAuth.LoadOrCreate(): a 32-byte crypto-random token that can never be empty (empty/missing file regenerates). That token is what Map closes over, and the check runs inside the handler regardless of authEnabled. QA's own harness booted the real GatewayHost with authEnabled=false and observed the 401s directly.
  • Consistent with other protected routes. The endpoints reuse the exact AuthMiddleware.HasValidToken (Bearer header OR cc-gateway-token cookie, Ordinal compare) the global middleware uses - one mechanism, one error body. The cookie path was verified accepted (Submit_GatewayCookie_IsAcceptedLikeOtherProtectedRoutes). Note: in production the OTHER routes are unprotected (middleware off) - these two routes are now strictly stronger, which is precisely the issue's intent.
  • No token-comparison weakness found. QA added 4 adversarial probes beyond the Developer's trace (see qa-curl-trace.txt, QA-EXTRA sections): EMPTY bearer (Authorization: Bearer ) -> 401 (no empty-token bypass); case-variant token (PROOF-TOKEN-369) -> 401 (Ordinal, case-sensitive); empty cookie value -> 401; wrong cookie -> 401. The compare is not constant-time, but that is byte-identical to the pre-existing global middleware - consistent, not a regression (noted as a possible future hardening, non-blocking).
  • No existence leak. The auth gate runs before any session/turn lookup on both routes, so an unauthenticated caller cannot probe which sessions or turn ids exist.
