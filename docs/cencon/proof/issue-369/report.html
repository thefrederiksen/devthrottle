Proof - Issue #369: Security: require API key on Gateway voice-turn endpoints

What was implemented

The Gateway's async voice-turn routes (POST /sessions/{sid}/voice-turn/submit and GET /sessions/{sid}/voice-turn/{turnId}, added by #376) are now token-gated with the Gateway's existing auth mechanism: AuthMiddleware's Bearer-header-or-cookie check against the per-machine gateway token. No new auth scheme was invented - the exact validation logic was extracted from AuthMiddleware.Run into AuthMiddleware.HasValidToken and reused verbatim by both voice-turn handlers.

Why endpoint-level enforcement matters: the production Gateway (GatewayTrayController.StartHostAsync and the dev GatewayWorker) constructs GatewayHost with authEnabled: false, so the global AuthMiddleware pipeline never runs in production - the voice-turn routes were reachable over the tailnet with no credentials at all (the exact exposure the issue describes). The gate now sits inside the handlers themselves, so it holds in production mode; when authEnabled is true the global middleware runs first and the endpoint check is a harmless second validation of the same token.

Phone compatibility (verified, not assumed): phone/CcDirectorClient/Voice/DirectorVoiceClient.NewClient() (lines 58-67, merged in #378) sets Authorization: Bearer <token> on every client it hands to SubmitVoiceTurnAsync/PollVoiceTurnAsync, so a phone with its token configured keeps working unchanged. The cookie path (cc-gateway-token, the Cockpit browser mechanism) is accepted too, because the reused check is bearer-OR-cookie - identical to every other protected route.
