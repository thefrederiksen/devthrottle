Issue 839 - Centralize all transcription on the Gateway behind one endpoint

Release notes

The Gateway is now the single owner of speech-to-text. Every module that needs audio turned into text sends the audio to one Gateway endpoint, POST /transcription, and gets the text back. No module resolves the transcription mode, reads the key, picks a provider, or talks to OpenAI on its own. The duplicate "figure out the mode, then read the key" resolvers are collapsed into one owner, and the standalone config.json Voice.OpenAiKey copy is removed so the key vault is the only transcription key store. This closes the split that let a recorded note fail with "no key is set" even when a valid key was present.

What was implemented

  • One owner: GatewayTranscriptionService (new) is the single place that resolves the configured mode and the key and runs the right provider - in-process Whisper for on-device mode, or the resolved OpenAI-compatible batch endpoint for the remote modes - with an optional validated dictionary-correction step. Resolution lives in one method, Resolve().
  • One endpoint: POST /transcription (new, TranscriptionBatchEndpoint) is the single audio-to-text HTTP face: raw audio body, content-type header, optional ?correct=true; returns { text, mode, model }.
  • Batch callers routed through it: the phone Notes worker (via a thin GatewayServiceRecordingTranscriber), the Settings "Test it" button (now posts to /transcription), on-device mode (the Notes path now supports local Whisper too), and the Gateway wingman-voice batch paths (/wingman/transcribe, /wingman/utterance/complete) all use the one owner.
  • Duplicate resolvers removed: RecordingEndpoints.ResolveSelectedMethod, the /transcription/test inline block (TranscriptionTestEndpoint deleted), and the wingman endpoint's own TranscribeBytesByModeAsync/TranscribeBytesAsync (which had hardcoded the OpenAI URL/model and ignored DevThrottle) are gone. TranscriptionRoutingEndpoint now resolves through the same single owner.
  • One key store: the config.json Voice.OpenAiKey loading is removed; OpenAiKeyResolver no longer reads an in-process key copy and instead reads the local key vault when standalone, so the key vault is the only transcription key store (gateway-attached pulls the remote vault via routing; standalone reads the local vault file).

CenCon impact

No architecture or security-posture drift. The change reduces surface (one resolver, one key store) and preserves the security invariants: the bring-your-own OpenAI key is still only ever paired with the OpenAI base URL (composed from the one pure TranscriptionEndpointResolver), and the transcription-integrity rule (CodingStyle section 16 - only TranscriptEditEngine / CleanupOrchestrator may change the words) is unchanged; the architecture guard still passes. The accepted tradeoff is stated: transcription is now Gateway/vault-owned, consistent with the thin-Director direction.

Scoping note on criterion 3. The transcription key now lives only in the key vault. The separate AgentOptions.OpenAiKey field used by desktop text-to-speech (a different OpenAI capability, not transcription) is retained but is no longer loaded from config.json; in production it is unset and the documented OPENAI_API_KEY seed is what feeds it. This keeps the live voice / TTS path the issue says to leave alone untouched.

Proof: running instance (loopback REST through a real Gateway host)

Because the transcription endpoints are Gateway-side (the Gateway tray host, not the Director the isolation script launches), the faithful running-instance proof boots a complete real GatewayHost on an ephemeral loopback port and drives POST /transcription and GET /transcription/routing over real HTTP through the whole host pipeline.
