Issue #621 - Make cron first-class for agents + humans

Developer Agent proof. Branch issue-621-cc-cron. cc-cron CLI (cc-tools family) wrapping the Gateway cron REST surface, plus the Cockpit Schedule page surfaced via a documented, discoverable opt-in.

What was implemented

  • New cc-cron CLI under tools/cc-cron/ (Python/typer/rich, same structure as cc-settings): list, get <id>, runs <id>, create (one-off --at/--tz or recurring --cron/--tz, with --machine, --repo, and --seed OR --worklist), run <id> (run-now), enable/disable <id>, delete <id>, and an endpoint diagnostic. A thin REST consumer only - it owns no job state and runs no scheduler (Scope OUT honored: no engine/store/CronJobDto/CronRunRecord change).
  • Gateway base URL discovery with no hard-coded port: resolves from the single configured source of truth gateway.url (via cc_shared.config, the same value the desktop and Cockpit use); falls back to the documented loopback default only when no gateway URL is configured, mirroring C# CockpitUrlResolver.LocalhostDefault. A global --gateway override is available. The Gateway's 400 validation message is echoed verbatim; an unreachable Gateway yields a clear error, never a stack trace.
  • Registered in tools/registry.json ("ship": true) with a pyproject.toml, cc-cron.spec, build.ps1, and unit tests (tests/test_gateway.py, 13 tests passing).
  • Cockpit Schedule page surfaced via documented opt-in (AC5, conservative route per the recorded decision): the Schedule nav item is hidden by default (style="display:none" in NavMenu.razor, the same v1-alpha posture as the other hidden items); the /schedule route stays fully functional and is reached by the documented, discoverable opt-in of navigating to it directly. NOT graduated to default-on.
  • Docs: a CC_TOOLS.md entry for cc-cron (with the full flag table and the endpoint-discovery / error behavior) plus a one-paragraph "Scheduling" section pointing humans at the Schedule page (and how to opt in) and agents at cc-cron.

How it was verified (real Gateway, isolated)

A real Gateway REST surface was exercised, NOT a mock. An isolated dev console Gateway was launched on a free port (127.0.0.1:17900) with CC_DIRECTOR_ROOT pointed at a scratch directory, so the user's cron store and running Directors/Gateway were never touched. The Cockpit was run standalone on 127.0.0.1:17470 pointed at that test Gateway. cc-cron resolved the base URL from an isolated config.json (proving the production discovery path end-to-end). All test processes were torn down afterward.

Note on the "Error:" cases in the transcript: cc-cron prints exactly one Error: <message> line to stderr and exits non-zero. There is no Python stack trace. (PowerShell additionally decorates a native program's stderr with its own "python.exe : ..." block; that is PowerShell, not cc-cron.)

CenCon impact

No architecture or security drift. cc-cron is a pure REST consumer of the already-shipped Gateway cron surface; no cron engine/store/CronJobDto/CronRunRecord contract changed, no new security surface was added (it reuses the existing gateway.token Bearer model and the same-machine loopback path). The Cockpit change is a single nav-visibility toggle. No docs/cencon/ manifest or security-profile update is required.
