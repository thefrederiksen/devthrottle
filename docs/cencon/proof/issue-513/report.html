Issue #513 - DevThrottle dictation uses the batch Whisper path

Branch: issue-513-batch-whisper-transport  |  Role: Developer Agent (CenCon)

QA-defect fix (2026-06-17) - now re-handed to QA. The first QA pass found a live defect: against the deployed Gateway (0.9.12, pre-#513) DevThrottle resolved transport=batch but model=gpt-4o-transcribe - the resolver self-healed the transport from the mode for an older Gateway but trusted the Gateway's stale model verbatim, leaving the exact 404 model_not_found combination this issue exists to eliminate. Fixed: in OpenAiKeyResolver.ResolveEndpointFromGatewayAsync, when the Gateway omits transport (proving it is a pre-#513 Gateway whose model field is equally untrustworthy) the resolver now derives both the transport and the provider-correct model from the authoritative mode. When the Gateway does serve transport (a current #513 Gateway), its model is trusted as-is. See the QA-defect fix proof below.

Release Notes

DevThrottle-mode dictation now works by routing through the batch Whisper path instead of OpenAI's Realtime WebSocket. The transcription routing target now carries a transport dimension and a provider-correct model, and the dictation pipeline honors the transport. This is the proper fix for the model/endpoint mismatch found after #506 - not a proxy-side model-rewriting hack.

  • BYO / OpenAI → transport realtime, model gpt-4o-transcribe, api.openai.com (unchanged).
  • DevThrottle / Groq → transport batch, model whisper-large-v3, devthrottle.com.
  • In DevThrottle mode the dictionary-cleanup LLM pass is skipped (raw Whisper transcript is returned); no cleanup call is routed to OpenAI or devthrottle.com.
  • The proxy stays honest - no model rewriting; the client now sends the correct model because the routing provides it.
