Issue #407 - Move reply audio out of the poll JSON into a dedicated resumable fetch

Developer Agent proof · branch issue-407-resumable-audio · built on recovery checkpoint d26e3db · captured 2026-06-15 on SOREN_NORTH · READY FOR QA

Summary. The reply audio no longer rides inside the status poll. The poll is now slim - it advertises audioReady / audioLength but carries no bytes - and a dedicated, resumable endpoint GET /sessions/{sid}/voice-turn/{turnId}/audio serves the raw audio/mpeg with HTTP range support (200 full / 206 partial / 404 unknown-or-expired). The phone fetches the audio separately with a resume/retry loop that re-requests only the missing tail after a drop. The legacy inline audioBase64 is retained for one release behind an opt-in flag so older phones still work.
