1. What was implemented

A new Director Control API verb, POST /sessions/{sid}/execute-action (src/CcDirector.ControlApi/ControlEndpoints.cs). The body is a WingmanAction JSON (none | type | send_keys | submit - the existing contract type, unchanged); the handler calls WingmanActionExecutor.Execute (the single write chokepoint in CcDirector.Core, unchanged) and returns the WingmanActResult. There is zero decision logic in the path: no WingmanService call, no LLM, no ClaudePath requirement. This is the Phase-1B missing verb from docs/architecture/TARGET_IMPLEMENTATION_PLAN.md section 3 and the Phase-3 entry point for the wingman decide/execute split. POST /sessions/{sid}/wingman/act is untouched.

Mechanical status-to-HTTP mapping: executor session_gone → 410 with a clear error and nothing injected; executor bad_request → 400; ok / suppressed → 200. Invalid sid → 400, unknown session → 404, missing body → 400. Token-gated by the existing DirectorAuth middleware (the route is not in PublicPaths).
