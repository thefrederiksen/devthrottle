Issue #725 - Director self-allocated a Windows-reserved port - Developer Proof

Root cause confirmed: on this machine, port 7882 is inside a Windows TCP excluded range (netsh int ipv4 show excludedportrange protocol=tcp lists 7882 7882). A raw bind probe read it as free, so the allocator handed it out; the System process shadowed the socket and 404'd every request, while the Director logged "listening".

What was implemented

  • Root-cause fix - PortAllocator skips Windows excluded ranges. Reads the netsh exclusion ranges once (best-effort, Windows-only), and the allocation scan + the reused-previous-port path skip any port inside them. ParseExcludedRanges / IsExcludedPort are pure + unit-tested.
  • Belt-and-braces - ControlApiHost self-probe. After Kestrel starts, the host calls its own /healthz over loopback and verifies the body identifies THIS Director. The "Kestrel listening" line is now gated on that probe; on failure it logs a loud SELF-PROBE FAILED line (never a misleading "listening") and releases the port so a restart re-picks.

Note on the AC2 negative path. A fully-live "bound-but-unservable" reproduction is now self-defeating: the root-cause fix means the allocator no longer hands out the reserved port, so the self-probe-failure branch is hard to trigger on this machine. The branch is in place and gates the log; its trigger (a non-serving bound port) is the exact condition the allocator fix prevents. The self-probe logic and the excluded-range parser are unit-tested.

CenCon impact

Touches control_api (PortAllocator + the host's post-start verification). No architecture/security drift; it makes the Director fail loudly instead of silently serving 404s, and avoids reserved ports.

Statement

I believe this issue is finished. The root cause (allocating a Windows-excluded port) is fixed and proven live - the Director skips 7882 and binds a serving port - and the self-probe ensures the Director never again claims "listening" on a dead port. Build clean; 21 tests pass.
