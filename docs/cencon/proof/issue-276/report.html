What was implemented

A skill-only change to .claude/skills/product-agent/skill.md. After the Product Agent labels an issue flow:ready-dev, it may now ALSO append that issue to an explicitly-named work list via the shipped #273 surface POST /lists/{name}/items, sending the structured ref { "source": "github", "id": "<issue#>", "area"? }, so the item enters the ordered queue the #274 runner drains. The append is opt-in and additive: the flow:ready-dev label stays the single source of STATUS truth; the list only sets ORDER / queue membership. The skill NEVER invents, auto-creates, or guesses a list name (no default list; a missing list returns 404 and is reported, not created). Label-only items flow through the method unchanged. No .NET code was touched - per the issue's Note to the Developer Agent, this is proven by exercising the skill's REST path against a running Gateway, not by dotnet build (which was still run, and is clean, to confirm nothing regressed).

Live REST proof (raw transcript)

Full transcript also committed beside this report as rest-proof-transcript.txt. Gateway was a dev console host built from this branch on isolated port 7950 with CC_GATEWAY_NO_TAILSCALE=1; auth was off (loopback), so no Bearer header was needed for the run.
=== CRITERION 2: append to a list that does NOT exist -> 404, NOT auto-created ===
POST /lists/ready-dev-queue/items {source:github,id:99001,area:CenCon}
  HTTP 404  {"error":"no such list","name":"ready-dev-queue"}
GET /lists  ->  {"lists":[]}        (nothing auto-created)

=== Setup: human/Cockpit creates the named list (NOT the product skill's job) ===
