Issue #389 - TTS per-request timeout, retry, and overall budget

Title: [Voice] TTS stalls up to 180s and returns no audio when an OpenAI request hangs (no per-request timeout/retry)

Branch: issue-389-tts-per-request-timeout

Type: Backend resilience fix in CcDirector.Core. Proof target (per the issue's acceptance criteria) is unit tests.

What was implemented

The root cause was that TtsService.GenerateAsync relied solely on the shared HttpClient.Timeout = 180s with no retry, so one stalled OpenAI /v1/audio/speech chunk blocked the whole voice turn for up to 180 seconds and then returned empty audio. The fix, in src/CcDirector.Core/Voice/TtsService.cs:

  • Per-request timeout (~30s/chunk): each chunk runs under a CancellationTokenSource.CreateLinkedTokenSource(ct) + CancelAfter(PerRequestTimeout). The shared client no longer carries the 180s ceiling (its timeout is Timeout.InfiniteTimeSpan; the linked tokens are the only deadlines).
  • Retry once on transient failure: a per-request timeout, a 5xx, or an empty body is flagged transient and retried a single time. A 4xx is permanent and is not retried.
  • Overall TTS budget (~60s): a second linked CTS with CancelAfter(OverallBudget) spans all chunks (including the retry), so a reply can never block for minutes.
  • Fail fast to text with logging: on final failure the call returns promptly and FileLog records elapsed milliseconds + failing chunk index + reason. The upstream text-only fallback (in VoiceTurnEndpoint) is unchanged.
  • Injectable HttpMessageHandler: a second constructor TtsService(AgentOptions, HttpMessageHandler) makes the timeout/retry behaviour unit-testable without the network. The existing TtsService(AgentOptions) constructor and all five call sites are unchanged.
  • Success path byte-identical: the MP3 chunk concatenation logic is unchanged (the null-forgiving ! in the old loop was replaced with explicit null checks per CodingStyle.md).
