Issue #589 - Migrate desktop dictation to whole-audio batch (remove realtime streaming + partials)

Branch issue-589-desktop-dictation-batch. Builds on the merged shared pipeline (#587, PR #596), the completeness gate (#586, PR #594), and the shared UI component (#588, PR #598).

What was implemented

Desktop dictation (the Speak dialog) no longer streams audio live to a realtime model and no longer shows a live partial transcript while the user speaks. It now captures the whole turn's audio locally, and on Insert / Send / Review it sends the whole clip once through the ONE shared BatchTranscriptionPipeline (issue #587) using the user-selected transcription method, applies the validated dictionary corrector only, and then inserts (Insert), auto-submits (Send), or shows for review (Review).

  • New src/CcDirector.Avalonia/Voice/BatchDictationRecorder.cs - whole-audio capture + single batch transcription for desktop dictation. Buffers every PCM16 chunk from MicAudioCapture locally for the whole turn; on stop it wraps the whole clip in one WAV blob (WavWriter) and calls BatchTranscriptionPipeline.TranscribeAsync exactly once. There is no OpenAiRealtimeProvider, no DictationPipeline, no LivePreviewTranscriber, and no OnPartial on this path.
  • src/CcDirector.Avalonia/Voice/SpeakDialog.axaml(.cs) - the dialog now uses BatchDictationRecorder. Removed the live-partial rendering (OnPartial, RenderTranscript during recording, the _currentPartial accumulator) and the realtime connect/pause-resume segmenting. The transcript box stays blank while recording and transcribing; text appears only after the single batch transcription completes. The former Pause/Resume button is replaced by a Review button (single capture has no resume).
  • The streaming SpeakService is left intact for the OTHER (alpha-hidden) surfaces that genuinely need continuous listening (the wake-word tester and the Voice tab). Dictation no longer shares the streaming orchestrator - that is exactly the divergence this issue removes.
  • Tests: WavWriterTests (RIFF/WAV header + payload round-trip) and DesktopDictationBatchProofTests (one batch call, no realtime socket, byte-identical with no dictionary term) in CcDirector.Avalonia.Tests; the loopback guard allowlist was extended for the new file.

Note on the live-UI screenshot

The live Speak dialog requires a physical microphone and a signed-in, transcription-key-configured Director, and a desktop screen capture in this shared environment would include the operator's unrelated personal applications (a privacy/PII risk on a public repo). The test Director (cc-director13.exe, slot 13) was built from this worktree and launched via its own scheduled task to confirm the build runs; it was then torn down. The three states above are reproduced from the actual XAML labels/colors and code, and the behavioral criteria (one batch call, no realtime socket, byte-identical text, empty-capture gate) are proven deterministically by the tests, which drive the exact code path the dialog uses. QA with a mic + configured key can confirm the live states visually.

CenCon impact

Updated docs/cencon/architecture_manifest.yaml flow dictation_flow: the desktop SpeakDialog path now records that it captures locally and transcribes via one whole-audio batch call through the shared BatchTranscriptionPipeline (no realtime streaming, no live partial preview, completeness gate enforced). No security posture change - the same OpenAiKeyResolver routing and key handling are used; the dictation path now makes plain HTTP batch calls instead of a realtime WebSocket.

Conclusion

I believe this is finished. Desktop dictation is whole-audio batch only: no realtime socket, no live partials, one batch transcription per turn via the user-selected method, dictionary-only correction (byte-identical with no dictionary term), and an explicit failure for an empty/interrupted capture. The full solution builds clean and the proof tests pass.
