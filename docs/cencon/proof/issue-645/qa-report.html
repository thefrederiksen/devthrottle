Verification approach

Each criterion was verified independently in the QA Agent's own worktree on the PR branch (D:\ReposFred\dt-645-qa), not by trusting the Developer's report. The role-selection and provisioning logic was traced through the source and confirmed by the test suites the QA Agent built and ran itself.

Install-OUTCOME note (AC#1 / AC#4): a genuine end-to-end install was deliberately NOT run. The installer writes to the shared per-user root %LOCALAPPDATA%\cc-director (InstallLayout.Default()), starts the always-on Gateway tray process, registers a Windows autostart Run key, and writes a Tailscale front-door mapping. Those are machine-global side effects that no environment override isolates; running it would collide with the user's REAL running Gateway/Director and live sessions. An isolated throwaway install is therefore NOT safely feasible, so AC#1 and AC#4 are verified at the LOGIC level (the role -> provisioning chain that lays down the gateway), per the QA task's explicit allowance. AC#2 and AC#3 are fully verified.
