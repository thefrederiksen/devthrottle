Issue #846 - Session-number backfill: startup wiring + on-demand endpoint

Branch: issue-846-backfill-numbers-wiring

Date: 2026-06-29

What was implemented

The existing, unit-tested SessionManager.BackfillNumbers() had no production caller. This change wires it in two ways:

  1. At Director startup. ControlApiHost.StartAsync calls _sessionManager.BackfillNumbers() once, right after StartSessionStateServices() and BEFORE the port bind (so a Director whose Control API cannot bind still numbers its existing sessions). The assigned count is logged.
  2. On a running Director (no restart). New POST /admin/backfill-numbers on the Director Control API runs the same backfill and returns { "assigned": N }. It is not in DirectorAuth.PublicPaths, so the Bearer token (or login cookie) is required when auth is enabled.
  3. Gateway reachability. New POST /directors/{id}/backfill-numbers on the Gateway forwards to the owning Director's /admin/backfill-numbers. It mirrors the existing /directors/{id}/settings route (a Director-id-keyed registry lookup), because the per-session /sessions/{sid}/{**rest} catch-all would otherwise treat backfill-numbers as a session id and 404.
