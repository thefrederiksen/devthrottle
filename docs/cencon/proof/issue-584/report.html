Developer Proof - Issue #584

[ControlApi] Agent management REST API under /settings/agents with full Settings-UI parity

Branch: issue-584-agents-rest-api. All work done in an isolated git worktree. Proof captured against a live test Director (own slot 13, own worktree build, self-allocated Control API port 7898), driven over the loopback Control API. The live user config.json was backed up before and fully restored after; real Windows user paths in the captured transcripts have been redacted to C:\Users\<user> (this is a public repository).

What was implemented

  • New endpoint file src/CcDirector.ControlApi/AgentsEndpoint.cs mapping the dedicated /settings/agents REST surface, wired into the Control API host alongside the existing /settings routes (ControlApiHost.cs).
  • Library (the agent.entries array): GET /settings/agents (list), GET /settings/agents/{id} (get one), POST /settings/agents (add), PATCH /settings/agents/{id} (update one or more values of one entry), DELETE /settings/agents/{id} (remove), POST /settings/agents/{id}/enabled (toggle enabled), POST /settings/agents/reorder (reorder by id list).
  • Parity sub-routes reusing the SAME Core services the Agents tab uses (no second implementation): POST /settings/agents/detect (ToolDetectionService.DetectTool), POST /settings/agents/quick-check (ToolDetectionService.TestToolAsync + persists the validation status via BuildValidationPatch), POST /settings/agents/command-line (AgentToolConfig.ResolveEffectiveCommandLineArguments), GET /settings/agents/catalog (AgentToolCatalog + AgentDrivers).
  • All writes persist through AgentEntryStore.SaveEntries → the non-lossy CcDirectorConfigService.MergePatch write authority - the identical store the Settings dialog Agents tab uses, so REST and the dialog stay behaviorally identical and unrelated config sections are never clobbered.
  • 18 new end-to-end tests in src/CcDirector.Gateway.Tests/AgentsEndpointTests.cs (all pass).

CenCon impact

This change adds REST routes to the existing control_api container and reuses the existing core_services (AgentEntryStore, AgentToolConfig, ToolDetectionService, AgentToolCatalog, AgentDrivers, CcDirectorConfigService). It introduces no new container, no new architectural shape, and no new security posture: the surface is loopback-only and subject to the existing auth middleware exactly like /settings, it handles no credentials or secrets, and it introduces no new agent value. No docs/cencon/ drift.
