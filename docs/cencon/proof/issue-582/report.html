Issue #582 - In-app account area: identity, log out, and a usage-telemetry opt-out toggle

Developer Agent proof. Branch issue-582-account-area. Built clean (dotnet build cc-director.sln -> Build succeeded, 0 warnings, 0 errors). Verified in a running, per-session-isolated test Director (slot 13) driven via UI Automation, with screenshots captured by PrintWindow (PW_RENDERFULLCONTENT) and the credential/telemetry state read straight from disk.

What was implemented

  • Account area (new "Account" tab in the Settings dialog). Shows the signed-in DevThrottle identity - email and authentication provider - read locally from the cached access token's claims (no network call), a Log out action, and the usage-telemetry opt-out toggle.
  • Identity reader. JwtIdentityReader decodes the Supabase-shaped access-token payload (the email claim and app_metadata.provider) entirely locally; DevThrottleAccountService.GetIdentity() surfaces it. Returns null (an explicit "Not available" state) when no credential is stored or the email claim is absent - no fabricated identity.
  • Log out. Clears the credential store and records a logout event (the existing DevThrottleAccountService.Logout() from #583), then returns the Director to the account gate (#580) in-process. Because the main window's close is wired to shut the whole Director down by design, the swap shows the gate as the new main window and hides the old windows rather than closing them, so the process stays up on the gate.
  • Usage-telemetry toggle. TelemetrySettings persists telemetry.enabled in config.json (default on) via the non-lossy CcDirectorConfigService.MergePatch. UsageTelemetry.Record(...) gates the richer usage path on that flag; the always-on authentication floor (AuthEventLog) records login/logout regardless. Telemetry is greenfield - this introduces the flag, the always-on floor's separation, the gate on the richer path, and a local usage sink that proves the gate (the server-side pipeline is the internal-repository child).

avalonia_ui (CcDirector.Avalonia) containers. It adds a configuration key
(telemetry.enabled) and a local usage sink; no new container, no architecture drift. No
security rule is violated: the usage sink and the auth-event log record only an event name/kind and a
timestamp - never the access/refresh token (DT-05) and nothing about the user's work; the identity
reader is local-only. The architecture manifest (last_updated 2026-06-21) and security profile
