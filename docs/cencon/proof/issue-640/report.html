Proof - Issue #640: Token refresh owned by the Gateway

Outcome: I believe this is finished   All five acceptance criteria proven.

Token refresh is now the Gateway's job. A real ITokenRefresher implementation (GatewayHttpTokenRefresher) replaces the no-op BackendUnavailableTokenRefresher that was wired into the Gateway-hosted credential service. When the cached access token is expired-but-well-formed and a refresh endpoint is configured (DEVTHROTTLE_REFRESH_URL), the Gateway exchanges the refresh token for a fresh access-plus-refresh pair and stores it through the credential service - in the background, never blocking startup or request handling. When no endpoint is configured, or it is unreachable, the cached credential is kept and the unavailability is logged clearly (no fallback that hides the failure).

What was implemented

FileChange
src/CcDirector.Gateway/Account/GatewayHttpTokenRefresher.cs (new) The real ITokenRefresher on the Gateway. Gated on DEVTHROTTLE_REFRESH_URL: when set, POSTs { "refresh_token": "..." } and parses { "access_token", "refresh_token" }; when unset it reports refresh unavailable (returns null). A connectivity error / non-success status / response missing either token is treated as "refresh unavailable" (null) and logged WITHOUT the token values.
src/CcDirector.Gateway/Account/GatewayTokenRefreshService.cs (new) The background refresh loop: a timer that periodically calls DevThrottleAccountService.RefreshIfNeededAsync fire-and-forget. Start() returns immediately (first sweep after a short delay), so it never blocks Gateway startup or request handling. The timer callback owns its try/catch so a refresh failure never crashes the timer thread.
src/CcDirector.Gateway/Account/GatewayAccountFactory.cs Wires GatewayHttpTokenRefresher (short-timeout HttpClient) into the credential service in place of the no-op refresher. Adds the RefreshUrlEnvVar constant.
src/CcDirector.Gateway/GatewayHost.cs Constructs the background refresh service when a credential service exists, starts it in StartAsync (after the host is listening, non-blocking), and disposes it in StopAsync.
src/CcDirector.Gateway.Tests/Account/GatewayTokenRefreshTests.cs (new) Five unit tests covering refresh-on-expiry against a local stub, no-endpoint kept-cached, unreachable-endpoint kept-cached, valid-token no-op (zero exchanges), and the refresher's no-endpoint null.

Backend dependency (provisional contract). The backend refresh-exchange endpoint is a flagged dependency (the issue's stated assumption). The refresh is therefore GATED on configuration (DEVTHROTTLE_REFRESH_URL) and proven against a local stub, exactly as the issue's proof target asks. The wire shape ({ refresh_token } in, { access_token, refresh_token } out) is Supabase-shaped and provisional; it is isolated in GatewayHttpTokenRefresher and easy to adjust when the backend contract lands.
