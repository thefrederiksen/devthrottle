Issue #591 - Bring the phone recorder onto the completeness gate + selected method

Summary. The phone recorder now goes through the shared completeness gate (issue #586, already enforced - reused unchanged) and routes transcription through the ONE shared batch pipeline (issue #587) using the user-SELECTED transcription method, instead of being hardcoded to OpenAI. The one-minute segmented capture, resumable upload, and assembly model are unchanged. The phone now reacts to a server gate refusal (HTTP 409) by re-sending exactly the named segments, closing a "stuck forever" gap so a kill-and-resume run delivers the full segment set with zero audio loss.

What was implemented

  • New SelectedMethodRecordingTranscriber (CcDirector.Core/Recording): an IRecordingTranscriber that resolves the user-selected method (base URL + key + model) FRESH per transcribe, transcribes each segment with ONE whole-segment batch POST to the resolved method via BatchTranscriptionPipeline (returning RAW text only), and applies the validated dictionary corrector ONCE on the assembled raw concatenation. No baked-in OpenAI URL, no free-text cleanup.
  • BatchTranscriptionPipeline.TranscribeRawAsync (additive): the transcription half of the shared pipeline exposed on its own, so callers that batch-transcribe several segments then run the dictionary corrector once keep using the single transport.
  • RecordingEndpoints.BuildService (CcDirector.Gateway): now constructs SelectedMethodRecordingTranscriber, resolving the configured mode + vault key in-process exactly like the Gateway's /transcription/routing endpoint - so switching the mode in the Cockpit changes the base URL the recording transcription POSTs to (BYO -> api.openai.com, DevThrottle -> devthrottle.com), with no restart.
  • Phone gate-driven resume: RecordingUploadGate.RequeueIndicesForResend (pure decision), IncompleteUploadException in IngestUploader (surfaces the 409 missing/bad indices), and AndroidAudioRecorder re-arms exactly the named segments and re-runs the upload pass.
