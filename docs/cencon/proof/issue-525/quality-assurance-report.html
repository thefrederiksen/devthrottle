Issue 525 driver slash command quality assurance report

Passed

This report covers the driver-aware slash command implementation and the tests that prove it works without starting the Director desktop application.

What changed

  • Each agent driver now owns slash command metadata through IAgentDriver.SlashCommands.
  • DriverCapabilities still controls bottom action buttons only.
  • PiDriver exposes a Pi built-in command catalog from PiSlashCommands.
  • ClaudeDriver exposes the existing Claude built-in command catalog.
  • GenericDriver and CursorDriver expose no built-in slash commands until verified.
  • The Avalonia composer asks for commands using the active session agent kind, so Pi no longer receives Claude-specific autocomplete.
  • Claude native command interception now only runs for Claude sessions, so Pi commands such as /model are not routed into Claude dialogs.

How dynamic command discovery works

Command sourceBehavior
Driver built-in commandsVersioned in code. For Pi this is PiSlashCommands, checked against the local Pi documentation.
Pi project prompt templatesDiscovered from .pi/prompts/*.md. A file named review.md becomes /review.
Pi project skillsDiscovered from .pi/skills/*/SKILL.md and .agents/skills/*/SKILL.md. A skill named browser becomes /skill:browser.
Pi global prompt templates and skillsDiscovered from the documented global Pi and shared agent locations.
Pi extension registered commandsNot dynamically discoverable yet because Pi does not currently expose a stable external list. We should add this later through a Pi machine-readable command list or a small Pi extension that writes registered commands to a file.
Dynamic update detail: commands are discovered from disk and cached per agent and repository path. If a prompt or skill is added after the first read, the provider must be invalidated. The test InvalidateCache_PiPromptAddedAfterFirstRead_RefreshesCommands proves that a newly added Pi prompt appears after cache invalidation. The desktop does not yet watch those folders live; that would be a separate user interface refresh improvement.

Driver harness

The harness is the new RecordingSessionBackend. It implements the session backend interface but starts no process. It records raw bytes and submitted text so driver behavior can be checked safely.

Driver actionHarness proof
Pi cancelRecords one byte: 0x1B, the Escape key.
Pi clear contextRecords submitted text: /new.
Pi interruptThrows a clear unsupported operation because Control plus C is unsafe for Pi.
Generic cancelRecords one byte: 0x1B.
Generic interruptRecords one byte: 0x03, the Control plus C byte.
