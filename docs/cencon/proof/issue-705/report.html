Issue #705 - Session Intercommunication - Developer Agent Proof

Branch: issue-705-session-intercomm
Summary: Sessions can list and message each other across the fleet by relaying through their own Director. The fleet token never reaches the agent process.

What was implemented

  • GatewayClient forwarding (src/CcDirector.ControlApi/GatewayClient.cs): ListFleetSessionsAsync, SendPromptToFleetAsync, FanoutToFleetAsync - reuse the Director's already-authenticated client; throw on failure so the boundary returns a clear error.
  • Director relay endpoints (ControlEndpoints.cs): GET /fleet/sessions, POST /fleet/send, POST /fleet/broadcast. Local targets deliver directly; remote targets relay through the Gateway; unknown target with no Gateway is a clear error (no fallback). Sender header stamped by the Director from its own session record.
  • Pure framing helper (FleetMessaging.cs) and contracts (FleetMessageRequest.cs).
  • Host wiring (ControlApiHost.cs): a Func<GatewayClient?> provider so the relay always reads the current client.
  • Spawn awareness (SessionManager.cs): CC_FLEET_TOOLS env note (NOT a credential).
  • Three Python tools: cc-sessions, cc-whoami, cc-send (+ shared cc_shared/director.py), registered in docs/cli-reference.md.
  • Tests: 8 C# (framing + endpoint validation) and 7 Python (resolver, incl. ambiguity) - all passing.

The sender name (devthrottle-wt-705) and machine were resolved by the Director from its own session record - not taken from the caller - confirming the anti-spoof stamping.

Proof environment note. This is a headless CLI/REST feature with no GUI surface, so the proof artifacts are terminal transcripts rather than a screenshot. The test Director was a per-session isolated slot-14 build (cc-director14.exe), launched via its own scheduled task, and torn down after proof. The recipient used for Transcript C was a throwaway RawCli (cmd) session in the isolated Director; multi-line text is delivered via the agent driver's @file paste convention (the same mechanism handover uses), which is why the content appears in the recipient's delivered input file. No message was injected into any real fleet session.

CenCon impact

Adds three endpoints to the CcDirector.ControlApi container, three forwarding methods on its Gateway client, and three cc-* tools. No security-posture regression - it specifically keeps the fleet token server-side and never exposes it to an agent process. No change to architecture or security blocking rules. No CenCon doc drift.

Statement

I believe this issue is finished. Every acceptance criterion is met, the solution builds clean with zero warnings, all 15 new tests pass, and the relay, clear-error, tools, and live framed delivery were proven against a running test Director.
