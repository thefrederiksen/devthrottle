Issue #717 - cc-ask (ask another session, get the answer back) - Developer Agent Proof

Branch: issue-717-cc-ask. Built on the merged #705 fleet messaging.

What was implemented

  • GatewayClient.AskFleetAsync - relays a question to the Gateway's prompt route with WaitForIdle=true and returns the captured Output. Uses a dedicated long-timeout HttpClient (the shared client's 10s timeout is too short for a waiting ask).
  • POST /fleet/ask (ControlEndpoints) - frames the question (reusing #705 FrameForSender); with a Gateway, relays prompt-with-wait (uniform for local and remote); standalone, captures a local target's reply via buffer-cursor + Idle wait (AskLocalAsync). Timeout -> 504; unknown/unreachable -> clear error.
  • FleetAskRequest / FleetAskResponse contracts.
  • cc-ask Python tool (single-target; rejects all); shared cc_shared/director.py gained a per-call timeout. Registered in docs/cli-reference.md.
  • Tests: 3 new endpoint-validation tests (11 fleet tests total, all passing).

The question was delivered, cc-ask waited for the target to finish its turn, captured the output produced during that turn, and printed it back to the asker. A cmd RawCli session is not a natural answerer, so the captured answer is its deterministic reaction to the question (the agent driver delivers multi-line text via the @file paste convention). The full deliver -> wait-for-idle -> capture -> return round-trip is exercised; a real agent recipient answers in prose over the identical path.
