Issue #587 - One shared batch transcription pipeline ready for QA

Repository: thefrederiksen/devthrottle  |  Branch: issue-587-shared-batch-transcription  |  Date: 2026-06-21

What was implemented

Built the ONE shared batch transcription pipeline every surface calls (design gap items G1, G4, G5). The change:

  • New shared Core entry point src/CcDirector.Core/Transcription/BatchTranscriptionPipeline.cs: takes a complete (already-gated) audio blob plus the resolved method, runs ONE whole-audio batch POST {baseUrl}/audio/transcriptions with the resolved key and model, then applies only the validated dictionary corrector. No streaming/partial transcription path. Returns the raw transcript, the corrected transcript, and the exact list of dictionary terms that changed.
  • VoiceService rewired src/CcDirector.Core/Voice/VoiceService.cs: removed the hardcoded whisper-1 / api.openai.com endpoint and the two calls to the free-text language-model cleanup (WingmanService.CleanVoiceTranscriptAsync). The transcription method now comes from the Gateway routing resolver (OpenAiKeyResolver.ResolveEndpointAsync), and the only post-transcription transform is the dictionary corrector via the shared pipeline.
  • Change reporting added src/CcDirector.Core/Dictation/CleanupOrchestrator.cs: CleanupOutcome now carries ChangedWords (the accepted find/replace edits that actually changed text), so the pipeline can report which words changed. The legacy 3-argument constructor is kept, so existing call sites are untouched.

The free-text cleanup method (WingmanService.CleanVoiceTranscriptAsync) was REMOVED from the transcription path but the method itself is left in place (its own unit tests still pass); per the issue's flagged assumption, any routing decision becomes a separate non-text step on a copy and never touches the used transcript.

Live proof - the running Director uses the new shared pipeline

A real Director (isolated slot 14, built from this worktree, launched via its own scheduled task) was driven through the Control API POST /voice/command with a small audio blob. The voice path went through the new pipeline using the resolver-selected method (the provider returned 400 only because the dummy bytes are not valid audio - proving the request reached the resolved transcription endpoint, not a hardcoded whisper-1 call). 

[BatchTranscriptionPipeline] TranscribeAsync: bytes=12, mode=byo, model=gpt-4o-transcribe
[VoiceService] Transcribe FAILED: Transcription returned 400: { "error": { "message": "Audio file might be corrupted or unsupported" ... } }

The log message prefix is [BatchTranscriptionPipeline] and the error text is "Transcription returned 400" (the new pipeline), where the old code logged [VoiceService] Whisper-specific text. The mode/model pair (byo / gpt-4o-transcribe) is exactly what the routing resolver chose - the hardcoded whisper-1 is gone.

CenCon impact

Architecture posture changed and the manifest was updated in the same change: docs/cencon/architecture_manifest.yaml - the VoiceService description was rewritten (whole-audio batch via the shared pipeline, no hardcoded whisper-1, dictionary-only), a new BatchTranscriptionPipeline component was added, and the Resumable Voice Utterance Flow trust path was updated to drop the free-text cleanup step. No security rule changed; the existing "bring-your-own OpenAI key is never sent to devthrottle.com" rule is reinforced (routing comes from the one pure resolver, and the pipeline presents routing.ApiKey only to routing.BaseUrl). security_profile.yaml last_verified (2026-06-09) is within its 30-day threshold.
