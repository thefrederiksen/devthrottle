Issue #854 - Gateway account device-list proxy

[Gateway] Proxy the account device list + revoke (GET/DELETE /account/devices)

A LOCAL Gateway proxy so the Cockpit never holds the account token or calls the cloud directly. The Gateway reads its own stored account token (DevThrottleAccountService.GetAccessTokenForForwarding - the same egress credential it already uses to forward telemetry/login), calls the cloud device registry (GET /api/v1/devices / DELETE /api/v1/devices/:id, contract devthrottle_internal#81/#82), and returns a token-free local DTO. Distinct from, and leaves untouched, the local pairing registry GET /devices (issue #469).

What was implemented

  • src/CcDirector.Core/Account/DeviceRegistryClient.cs (new) - injectable-HttpClient cloud egress client; base URL reuses AccountTelemetryClient.ApiBaseUrlEnvVar / DefaultApiBaseUrl (DEVTHROTTLE_API_URL, default https://devthrottle.com) - the same cloud base the Gateway already targets, no new hard-coded URL. ListDevicesAsync + RevokeDeviceAsync attach the Bearer token; the token is never logged.
  • src/CcDirector.Gateway.Contracts/AccountDevicesDto.cs (new) - AccountDeviceDto, AccountDevicesResponseDto{signedIn,devices}, RevokeDeviceResponseDto{signedIn,id,revoked}. No token field exists on any of them (security rule DT-05).
  • src/CcDirector.Gateway/Api/AccountDevicesEndpoint.cs (new) - maps GET /account/devices and DELETE /account/devices/{id}; signed-out yields an explicit signedIn:false envelope; cloud-unreachable yields a clear 502 (logged); this-device marker computed by matching the record name to the host machine name.
  • src/CcDirector.Gateway/GatewayHost.cs - wired near the existing AccountStatusEndpoint / AccountLogoutEndpoint mapping, under the same host-wide Gateway auth middleware.
  • src/CcDirector.Gateway.Tests/Account/AccountDevicesEndpointTests.cs (new) - 9 stub-cloud unit tests over a real Kestrel host + real credential service.
  • src/CcDirector.Gateway.Tests/GatewayHostTests.cs - 2 wiring tests over the real in-process GatewayHost.

CenCon impact

No drift. The new endpoint is consistent with the existing Gateway-as-single-egress architecture (it sits next to /account/status and /account/logout); there is no per-endpoint component registry in architecture_manifest.yaml to update. Security rule DT-05 is honored: no token field on any Cockpit-facing DTO and no token value in any log line. Security-profile drift and manifest staleness checks both PASS.

QA gate (live real-cloud end-to-end)

The signed-in, real-cloud path could not be staged from this automation session (no real signed-in DevThrottle account + live cloud device registry available here). It is proven against an in-process stub of the documented cloud contract (devthrottle_internal#81/#82), exercised through the real endpoint on a real Kestrel host and the real credential service. The live real-cloud end-to-end - a signed-in Gateway returning its actual account devices, a real revoke disappearing from a real subsequent list - is the QA verification gate. The list-envelope shape assumed here is a top-level JSON array of masked records; QA should confirm the cloud's actual envelope.

Conclusion

I believe this is finished. All five acceptance criteria plus the auth-gate requirement are met with proof; the build is clean with zero warnings; the change honors DT-05 and the no-fallback / try-catch-at-boundaries coding standards.
