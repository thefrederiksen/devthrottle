Issue #535 - [Voice] /m/ session: return to list after sending to the agent; stay after sending to the wingman

Surface: Cockpit mobile assets src/CcDirector.Cockpit/wwwroot/m/ (m.js, index.html, sw.js). Client-only; no server-side change.

Build: dotnet build cc-director.sln -> Build succeeded. 0 Warning(s) / 0 Error(s).

What was implemented

  • Post-send destination now depends on the target.
    • Send to agent (text askAgentBtn -> runAgentTurn, and voice mode "agent" -> onRecordingStopped -> sendEntry -> runAgentTurn): returns to the list view promptly on send (not after the agent turn completes). The text handler fires runAgentTurn then calls showList(); the voice handler calls showList() right after sendEntry for an agent recording.
    • Ask the wingman (text askWingmanBtn -> runWingmanDirect, and voice mode "wingman"): stays on the session view to hear the spoken answer (unchanged behavior, now explicit).
  • No-silent-failure for the agent send across navigation. runAgentTurn was rewritten to be durable: it sets blocked[sid]=true (so the originating list row shows "- sending...") and, on a connection/server delivery failure while the user is off the session view, it retries with backoff until the turn lands. So a text (or voice) agent send that fails after we navigate to the list is never silently lost - the row visibly shows it is still sending and it keeps trying. If the user is still on the session view, the existing on-screen failure + Retry is shown instead.
  • Cache-bust bumped v17 -> v18 in index.html (m.js?v= / m.css?v= + on-screen label) and sw.js (cache name + pre-cached SHELL), plus the m.js header comment.
  • Not changed here (owned by #532, already in main): clearing the textarea on tap and the wingman "sent" wording. This change builds on top of them.
