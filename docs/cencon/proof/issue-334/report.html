Code Review Findings

F1 - TreeView Virtualization Missing (filed as follow-up #issue)

Severity: Medium - Performance degradation on large repos, not a crash

Location: GitChangesView.axaml lines 159-163 and 191-195 (StagedTree and ChangesTree)

Avalonia's TreeView does not virtualize by default. With 2,000+ items, all nodes are instantiated and laid out in a single pass when ItemsSource is set. On a fixture with 2,001 entries, setting ItemsSource can take 200-500ms depending on machine, which is measurable but not catastrophic at the file counts in daily use.

Note on daily-use scale: Normal repos rarely have 2,000 individually tracked changed files. In practice, untracked directories appear as a single entry (?? dir/) when using -u normal. The -u (which expands to --untracked-files=all) flag in the current code shows every untracked file individually - this is the scenario that generates large file counts.

Recommendation: File a follow-up issue to add a virtualizing panel and consider whether -u should use --untracked-files=normal for very large repos. Small fix, low risk, not blocking this review.

Filed as structural finding per issue scope ("structural ones get filed and linked").

F2 - Brushes Not Frozen (non-issue in Avalonia)

Location: GitChangesView.axaml.cs lines 34-39

The static brush fields are created as new SolidColorBrush(...) without Freeze(). In WPF, non-frozen brushes accessed from background threads throw. In Avalonia, SolidColorBrush is not frozen in the WPF sense - the rendering pipeline is thread-safe for immutable brushes created on the UI thread and only accessed from the UI thread (via data bindings). No cross-thread brush access occurs here - all UI updates flow through DispatcherTimer ticks on the UI thread. Not a bug in Avalonia context.

F3 - Post-await UI Access (non-issue in Avalonia)

Location: GitChangesView.axaml.cs lines 175-192 (RefreshAsync)

After await _provider.GetStatusAsync(_repoPath), the method sets StagedTree.ItemsSource and other UI properties directly. This is safe because RefreshAsync is only called from PollTimer_Tick (an async void event handler on a DispatcherTimer which fires on the UI thread) and from Attach() via fire-and-forget _ = RefreshAsync(). Avalonia maintains the UI SynchronizationContext so await continuations resume on the UI thread. Not a bug.

F4 - BuildTree Allocates Per-Refresh (observation)

Location: GitChangesView.axaml.cs lines 195-236

Every refresh allocates a Dictionary<GitFolderNode, Dictionary<string, GitFolderNode>>. At 2,000 files this allocates roughly 2,000 GitFileLeafNode objects + folder nodes. The cache guard (_lastRawOutput comparison at line 171) prevents unnecessary rebuilds when git output hasn't changed. In practice the timer fires every 15 seconds and the comparison short-circuits most rebuilds. Acceptable, no fix needed.
