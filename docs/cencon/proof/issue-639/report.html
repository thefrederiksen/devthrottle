Issue #639 - Gateway forwards telemetry with its own account token

Branch: issue-639-gateway-telemetry-token  |  Containers touched: CcDirector.Gateway, CcDirector.Gateway.Tests (plus a small read-only accessor on CcDirector.Core.Account.DevThrottleAccountService).

What was implemented

  • The Gateway is now the single egress and attaches its OWN account token. A new IGatewayTelemetryTokenSource supplies the Gateway's stored account token, read from the Gateway credential service (DevThrottleAccountService, issue #636) via the new GatewayAccountTelemetryTokenSource adapter. The credential service gained one read-only accessor, GetAccessTokenForForwarding(), which returns the stored access token when the Gateway is signed in (same local valid-or-renewable check as IsLoggedIn) and null otherwise. The token value is never logged.
  • The token is resolved at forward time, inside TelemetryRetryQueue.TryForwardAsync. When a token source is wired: signed in -> attach the Gateway token (any leftover per-event stored Bearer is ignored); not signed in -> the forward is DEFERRED (the event stays at the head of the queue, FIFO preserved) and flushes once the Gateway signs in. When no token source is wired (a host with no credential service) the queue keeps its Phase 1 behaviour (forward the stored per-event Bearer), so nothing on non-Windows hosts regresses.
  • The login relay (POST /telemetry/login) no longer requires or forwards an inbound Authorization header. An inbound Director Bearer, if still present, is ignored (logged only as present/absent, never its value) and the event is enqueued with no stored bearer.
  • The director-startup forward (POST /telemetry/director-startup) shares the same queue, so it too now attaches the Gateway token at forward time and defers while not signed in.
  • GatewayHost wiring: the credential service (Account) is resolved before the telemetry queue is built, and when present the queue is constructed with a GatewayAccountTelemetryTokenSource.

CenCon impact

No architecture drift: the containers (Gateway, Gateway.Tests) and the cloud wire contract (Bearer + source:"app") are unchanged - only whose token is attached changes. The security posture is preserved and reinforced: the Gateway token is never logged (rule DT-05), and a Director token is no longer forwarded to the cloud, removing a credential-leakage path.

I believe this is finished. All five acceptance criteria are met and proven, the full solution builds clean with zero warnings, and the existing telemetry tests were updated to the new contract (the inbound Director Bearer is ignored).
