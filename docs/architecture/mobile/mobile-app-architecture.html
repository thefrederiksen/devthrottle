Mobile App - Architecture Report

A new React + TypeScript Progressive Web App for the phone, served by the Gateway, talking to the existing REST API.
Status: proposal for sign-off  |  Date: 2026-06-28  |  Supersedes: the Blazor /mobile page, the vanilla /m/ app, and the /voice/ page

This is the plan for one real mobile experience, built once. We pick React + TypeScript, ship it as an installable offline Progressive Web App, serve its static files directly from the Gateway (the always-on front door), and let it drive the existing Gateway REST API. The desktop Cockpit stays as it is. When a phone opens the Cockpit URL, the front door detects the device and redirects it to the mobile app automatically; flipping the browser to "Desktop site" is the deliberate escape hatch back to the full Cockpit.

1. The decision, and why

Three stacks were on the table. The deciding constraint is that the phone is often on a questionable network, and the app will grow to hold many features over time (the goal is to operate sessions from the phone, and to build the app itself from the phone).

The honest trade we accept: React adds a TypeScript/Node build toolchain to a repo that is otherwise pure C#. That is a one-time setup cost. We pay it because mobile is a real, growing, phone-first app, and React is the lightest, most resilient, most maintainable way to build it. The C# backend stays the single source of truth (see the typed client below).

1b. Pages and version scope

We start small and make each page genuinely good before adding the next - the phone app gets used and developed daily, so every screen has to earn its place. The shape mirrors the desktop Cockpit: a home that shows who needs attention, drill into a session, and operate it.

Why serve the mobile app from the Gateway, not the Cockpit? Today /m/ is served by the Blazor Cockpit child process. We move serving to the Gateway because the Gateway is the always-on front door and the mobile app has nothing to do with Blazor. This makes the mobile app independent of the Cockpit process (it stays up even while the Cockpit is restarting or being retired) and lets the Gateway inject the auth token into the page directly.

3. Device detection and forwarding

The redirect lives at the Gateway front door, as a small middleware that runs before the fallback proxy - the same spot the existing browser-aware page routing already uses. On a browser navigation (a request that says Accept: text/html) whose path is not already under /m:

  • If the User-Agent looks like a phone (Android / iPhone / mobile), respond with 302 Found to /m/.
  • Otherwise fall through unchanged to the desktop Cockpit.

The escape hatch is free: Android's "Desktop site" toggle rewrites the User-Agent to a desktop signature, so that request no longer matches the phone test and falls straight through to the full Cockpit. No separate setting, no cookie to manage.

Why User-Agent and not screen width? The redirect has to happen at the front door before any app loads, so we cannot measure the viewport in JavaScript first (that would mean loading the heavy desktop app and then bouncing). User-Agent is the only signal available server-side at navigation time, and it is exactly what the "Desktop site" toggle changes - so it doubles as the escape hatch. Width-based logic still applies inside the mobile app for layout.

4. Where it lives in the repo, and how it is built

The React project sits at the repo root, outside the C# src/ tree so it does not confuse the .NET solution.

      wwwroot/m/                  <- mobile build output lands here; Gateway serves it at /m

Build wiring. An MSBuild target on the Gateway project runs the web build during a normal dotnet build/publish, so a developer (or CI) never has to remember a separate step:

<!-- CcDirector.Gateway.csproj -->
<Target Name="BuildMobilePwa" BeforeTargets="Build">
  <Exec Command="npm ci"          WorkingDirectory="$(SolutionDir)mobile" />
  <Exec Command="npm run build"   WorkingDirectory="$(SolutionDir)mobile" />
  <!-- vite outputs to mobile/dist; copy into the Gateway wwwroot it serves -->
  <ItemGroup><MobileDist Include="$(SolutionDir)mobile\dist\**\*" /></ItemGroup>
  <Copy SourceFiles="@(MobileDist)" DestinationFolder="wwwroot\m\%(RecursiveDir)" />
</Target>

Vite content-hashes asset filenames, so cache-busting is automatic; the PWA service worker handles update-on-next-launch.

Build-time only, release pipeline only - no new runtime dependency. Node.js and npm are needed only to turn the React/TypeScript source into static files. That happens once, on the machine that builds the release (the release pipeline / CI). It is the same category of tool as the .NET SDK you already need to compile at all.
  • Machines that RUN the Gateway (your dev VMs, end users): nothing extra to install. The built static files ship inside the release bundle, and the Gateway already serves static files. The browser runs the JavaScript, not the server. The auto-update path carries the pre-built files like any other release asset.
  • Ordinary C# developers who never touch the mobile app: also no Node needed. The web build is wired to run only at release/publish time (a Condition on the MSBuild target, or a CI step), not on every local dotnet build. So a routine C# build stays Node-free.
  • Only the release builder installs Node + npm, once.
The MSBuild target shown above is therefore gated to publish/release configurations rather than firing on every Build, so the dependency lives in the build pipeline alone.

5. The typed API client - C# stays the source of truth

The worry with a separate-language front-end is keeping the request/response shapes in sync. We remove that by hand by generating the client:

  1. The Gateway emits an OpenAPI document describing its endpoints and the CcDirector.Gateway.Contracts DTOs (built into .NET 10 via Microsoft.AspNetCore.OpenApi / MapOpenApi).
  2. A codegen step (openapi-typescript) turns that document into TypeScript types and a typed fetch client under mobile/src/api/.
  3. The React app calls the API through that client. If a C# DTO changes and the front-end is not updated, the TypeScript build fails - the mismatch is caught at compile time, not in production.

So the C# backend remains the one place a shape is defined; the front-end consumes a faithful, type-checked copy.

6. Authentication and trust boundary

The trust boundary is unchanged from the phone/voice surfaces today. The app is only reachable through the Gateway front door, which sits behind the Tailscale tailnet (authenticated, TLS). Some endpoints (for example the voice-turn endpoints) are token-gated even when global auth is off, so:

  • The Gateway serves the mobile index.html with the per-machine Gateway token injected - the same pattern the /voice page already uses today.
  • The app sends that token as a Bearer header on its API calls.

No new auth model; the mobile app reuses the one the phone already relies on.

7. Offline and "app-like" behavior (the PWA layer)

Installable + offline shell

vite-plugin-pwa generates a web app manifest and a service worker that caches the app shell, so it launches from the home screen and opens even with no connectivity (showing last-known state).

Offline outbox

Voice recordings and prompts queue in IndexedDB and upload when connectivity returns - a clean reimplementation of the behavior the vanilla /m/ app proved (retry-forever upload, drive-safe).

The existing vanilla app is not wasted - it is our behavioral spec. It already encodes the hard-won mobile details (gesture-safe autoplay, the session color logic, the offline upload outbox). We port the behavior, not the code.

9. Phased rollout

  1. Scaffold. Create mobile/ (Vite + React + TS + vite-plugin-pwa). Gateway emits OpenAPI; wire the codegen step and the MSBuild build/copy. Gateway serves the build at /m with token injection. Ship a one-screen "sessions roster" to prove the whole pipe end-to-end on the phone.
  2. Forwarding. Add the Gateway device-redirect middleware (mobile UA -> /m) and confirm the Desktop-site escape hatch.
  3. Core features. Session detail / conversation history, voice (the current /m/ behavior), then progressively the rest - built and tested from the phone.
  4. Cutover. Once parity is reached, delete the Blazor /mobile page, the vanilla /m/, and /voice/, and drop the Cockpit "Mobile" nav item.

10. Open decisions for sign-off

  • Path: keep the mobile app at /m (short, already in muscle memory), or give it a fresh path like /app? This report assumes /m.
  • TypeScript: assumed yes (type-safety against the API is the main reason). Plain-JS React is possible but loses the compile-time DTO check.
  • Component styling: hand-rolled CSS (matches the current dark look, zero deps) vs a small mobile component kit. Recommendation: start hand-rolled to keep the bundle tiny.
  • Native MAUI client (phone/CcDirectorClient): out of scope here; it keeps using the same REST API. The PWA is the near-term path because it deploys and updates instantly.
