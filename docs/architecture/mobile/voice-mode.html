Mobile Voice Mode

Implementation document - DevThrottle mobile PWA (the app served at /m). Draft for review before any code is written.

Voice Mode turns a running Claude Code session into something you can listen to and answer hands-free from your phone. A read-only Wingman watches the session, and at the end of every turn it speaks a clear, plain-English narrative of what happened and what (if anything) it needs from you - including when it is waiting on a list of choices. The spoken clip is rendered to audio on the Gateway, downloaded to the phone, and only then offered for playback so you never tap a play button and then sit waiting for a download.

Headline finding from the research: most of the backend for this already exists and is proven on the desktop Cockpit voice tab and the native phone app. The Wingman, the per-turn narrative (with single/multiple selection support), the text-to-speech, the per-session audio cache, and the "which sessions are ready" endpoint are all built. The new work is almost entirely in the mobile PWA UI, plus one new behavioral rule: the phone must hold the audio before the play-triangle appears.

1. The idea, in plain English

  • You open a session and tap a Voice mode tab. On that screen there is a Switch to voice mode button.
  • Pressing it tells the Wingman to start watching this session. The Wingman is a helper that sits on top of your real session - it never types for you and never changes your work; it only reads.
  • The moment you enter voice mode, it begins explaining: it waits for the Wingman, the Wingman comes back with "here is what just happened and here is what it wants," and - if you are still on the voice screen - it speaks it to you.
  • From then on this happens automatically on every following turn. You do not press anything to get the next narration.
  • The narration focuses on the conversation and what is being asked of you. Tool calls are background noise and are not the point.
  • If the session is waiting on several choices (a numbered menu, "pick any that apply"), the narration says how many options there are and what they are, and the screen shows them as buttons you can press.
  • If you have left the voice screen and gone back to the session list, the session's card turns yellow while the Wingman is reading and turning the turn into speech.
  • When the Wingman is finished, the card shows a play-triangle so you know a voice message is waiting. You tap it to listen.
  • The triangle does not appear until the audio has been downloaded to the phone. "Finished," from the phone's point of view, means the audio is sitting on the phone ready to play - not merely rendered on the Gateway. We never want you to tap and then wait for a download.

2. How it works end to end

One loop, repeated for every turn of the primary session while voice mode is on for that session:

PRIMARY SESSION GATEWAY (Wingman warm brain) PHONE (PWA, voice mode on) --------------- ---------------------------- -------------------------- turn runs ........ bytes .....> turn-end detected (ActivityState -> WaitingForInput / WaitingForPerm) | v reads the new turn's history (conversation first; tool calls are de-emphasized) | v builds the spoken narrative - headline / what happened - needsYou.statement - selectionMode: single|multiple - options[] (key, send, note) ---- card goes YELLOW here -----> [ session card: yellow + spinner ] | (VoiceGenerating = true) v TTS (OpenAI "nova") -> MP3 cached at /voice-audio/ (VoiceAudioReady = true) | v /wingman/voice/ready lists ----- phone fetches the MP3 bytes ---> [ download to phone cache ] and stores them locally | v on voice screen: auto-play + show options on the list: NOW show the play-triangle ---> [ card: green + play-triangle ] (phone-ready, not just gateway-ready)
The one genuinely new rule (Section 2 highlight): there are two notions of "ready." The Gateway is ready when the MP3 is rendered and cached. The phone is ready only after it has pulled those bytes down and stored them locally. The play-triangle and any auto-play are gated on phone-ready. Until the bytes are local, the card shows the yellow "working" state, not a triangle.

Reply interaction (how you answer by voice)

  • There is no hold-to-talk. You tap Respond; the control switches into the recording interface - exactly like the existing Speak dictation - you speak, then you tap Send.
  • The dictation panel for a reply has Cancel, Pause / Resume, and Send. It does not have Insert (mockup F). Send transcribes and sends the text straight into the session (the same POST /prompt path the Send button uses, with the transcript run through the existing dictionary-correction engine).
  • For this version there is a single reply target: Respond (transcribe -> into the session). A second target, Talk to the Wingman (ask the Wingman directly rather than driving the session), is planned but explicitly out of scope for now - leave room for it, do not build it yet.

7. Open decisions for review

  1. One warm brain or one Wingman per session? Today the Gateway runs a single shared warm-brain session for translation (cheaper, proven). Your description ("a session that should always be started with the original session") could mean a dedicated companion per primary session. Recommendation: keep the shared warm brain for v1; revisit only if narratives bleed across sessions.
  2. Where does the phone store the audio? Cache Storage API (Workbox runtime cache keyed on the audio URL) vs an explicit IndexedDB blob store we manage. Recommendation: Cache Storage keyed by session + turn, so "phone-ready" is simply "the response is in the cache."
  3. How does the phone learn a new turn finished? Piggy-back on the existing 5s roster poll (it already carries VoiceGenerating/VoiceAudioReady) vs a dedicated voice poll/stream. Recommendation: roster poll for the list; a slightly faster poll only while the voice screen is open.
  4. Auto-play policy. Auto-play only when the voice screen is foreground and the clip is phone-ready; never auto-play from the list. Confirm this matches your intent.
  5. Replying by voice (DECIDED). Tap Respond -> the existing dictation interface (no hold-to-talk) -> Cancel / Pause / Send, no Insert -> Send sends into the session. A future "Talk to the Wingman" target is out of scope for now.
  6. Per-turn cost. Every turn triggers an Opus narration + a TTS call for every voice session. Do we cap which sessions get voice (only "needs you"?) or narrate every turn of every voice session? Recommendation: only sessions explicitly switched into voice mode, which is already the model.

8. Out of scope (for this first version)

  • Fully conversational back-and-forth voice (continuous listen). v1 is: narrate on each turn, answer by tapping options or dictating.
  • Background narration when the PWA is fully closed (depends on platform push; not assumed here).
  • Any change to the Wingman charter. It stays read-only; answering a choice goes through the existing single write chokepoint (menu-press / the action executor).
