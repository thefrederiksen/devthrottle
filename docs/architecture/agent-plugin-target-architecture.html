Target Architecture: Agent Plugin Framework

A proposed clean architecture for controlling Claude Code, Codex, Gemini, Pi, OpenCode, Cursor, Grok, Copilot, and future CLIs without central switch statements or hard-coded tool catalogs.

Core principle: every CLI integration is a plugin. The plugin owns what makes that CLI special: settings, launch, permissions, models, runtime verbs, slash commands, transcript/history, and verification status. Product surfaces consume plugin contracts.

The plugin is the single authoritative object for a CLI. It can be implemented as one class internally, or as a small bundle of interfaces, but it must be registered as one coherent capability package.

3. Recommended Interface Family

IAgentPlugin

Top-level registration unit. Carries identity and returns the specialized providers below. The host talks to this first.

  • Id, DisplayName, Kind
  • Capabilities
  • Settings, Launch, Runtime, Transcripts

IAgentSettingsProvider

Replaces the central tool catalog. Settings UI renders this schema and stores values generically.

  • permission modes
  • command presets
  • models and model flags
  • custom fields and validation

IAgentLaunchProvider

Composes launch command from saved settings. No Settings dialog, New Session dialog, or Control API endpoint knows flags.

  • detect executable
  • version probe
  • compose argv/env
  • resume/session id support

IAgentRuntimeDriver

Owns live terminal control after the process starts. Buttons and REST verbs come directly from this contract.

  • submit
  • cancel / interrupt
  • clear context
  • history / resume / fork

IAgentTranscriptProvider

Owns history and usage for each CLI. It may be absent; when present, the History tab and Wingman consume it.

  • resolve active transcript
  • parse conversation
  • usage and tools
  • context reset rollover

IAgentCommandProvider

Supplies slash commands and composer help. This remains CLI-specific and plugin-owned.

  • slash command list
  • docs snippets
  • feature flags
  • version-aware availability

5. Ownership Matrix

Concern Target Owner Reason
Display name, icon, docs, install help Agent plugin manifest A plugin should describe itself without app code changes.
Executable detection and version probe Plugin launch provider Detection is CLI-specific and often version-specific.
Command-line presets and permission modes Plugin settings provider Presets are part of the CLI integration, not a central catalog.
Model picker and model flag Plugin settings provider Models and flags vary by CLI and evolve independently.
Process lifecycle and PTY custody Framework runtime host Shared mechanical custody should stay CLI-agnostic.
Submit, Stop, Interrupt, Clear, History Plugin runtime driver Keystrokes and slash commands are CLI protocol, not UI behavior.
Transcript lookup and parsing Plugin transcript provider Every CLI stores history differently.
Button visibility and REST verb availability Framework reads plugin capabilities Surfaces render declared capabilities; they do not special-case CLIs.

6. Capability Model

Capabilities should be richer than a flat enum. A plugin should report both whether a feature exists and how verified it is.

Supported

Implemented and tested for this CLI/version. UI may show the button normally.

Experimental

Available behind an explicit flag or warning. Useful for new plugin development.

Unsupported

Known absent or unsafe. UI hides it and API returns a clear unsupported response.

Unknown

Not verified for this CLI/version. Treated as unsupported until proven.

7. Example: Codex As A Plugin

Plugin-Owned Settings

Codex declares its own permissions control, command-line flags, model behavior, and validation.

  • Standard: normal Codex permissions behavior
  • Full access: --sandbox danger-full-access --ask-for-approval never
  • model options if Codex exposes a stable launch flag
  • version probe and install instructions

Plugin-Owned Runtime

Codex declares which live actions are valid and exactly how to perform them.

  • Stop: verified soft-cancel keystroke
  • Interrupt: verified hard interrupt behavior
  • Clear context: submit /clear if verified for this CLI version
  • History: only if a safe picker/resume action is verified

9. What This Replaces

The target architecture removes central knowledge from the host. There should be no central list that says "Codex has these presets" or "Cursor has this yolo flag" or "Gemini has no clear-context button." Those facts belong to the plugin. The host only knows how to load, validate, store, and execute plugin contracts.

Current Pattern To Retire

  • central AgentToolCatalog for CLI presets
  • central ToolDetectionService switch logic
  • central AgentDrivers.For(...) switch as the only registry
  • UI branches for specific CLIs

Target Pattern

  • plugin manifest and registry
  • plugin-owned settings schema
  • plugin-owned launch/runtime/transcript adapters
  • host-rendered generic UI from plugin capabilities

10. Architectural Decision

Do not merge everything into one huge interface. Use one top-level plugin registration unit that exposes focused subcontracts. That gives us one source of truth per CLI without creating a monolithic interface every plugin must implement completely.

One Plugin

Every CLI has one package and one registration point. The product thinks in agents, not in scattered services.

Many Facets

Settings, launch, runtime, transcripts, and commands stay separate internally because they have different lifecycles.

Optional Providers

A plugin can support runtime but not transcripts, or settings but not model selection, with explicit capability status.

11. Migration Direction

Phase Goal Result
1. Define contracts Create plugin interfaces and capability schema without changing behavior. Current CLIs can be wrapped by adapter plugins.
2. Wrap built-ins Turn Claude, Pi, Cursor, Copilot, Codex, Gemini, OpenCode, Grok into built-in plugins. No central preset or driver switch needed for built-ins.
3. Move settings ownership Replace AgentToolCatalog with plugin settings providers. Settings UI becomes schema-rendered and plugin-driven.
4. Move detection ownership Replace central detection switch with plugin launch providers. New CLIs can define their own install/detect/version story.
5. External plugin loading Load plugin packages from a configured folder with manifest validation. Adding a CLI no longer requires changing Director source code.

12. Summary

The target architecture is not "one driver class does everything" and not "many unrelated central catalogs know pieces of each CLI." It is one plugin per CLI, with focused provider interfaces underneath. The framework owns custody and rendering. The plugin owns CLI knowledge.
