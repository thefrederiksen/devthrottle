CC Director - Security Flows

How every machine, agent, and client gets authorized access to a REST API. Target model: per-device keys issued by the Gateway.
Design / explainer doc - diagrams and flows only, not the implementation plan. Status: PROPOSED (per-device-key model approved).

1. The trust model in one paragraph

The Gateway is the single issuer and registry of credentials. Every machine that runs a Director holds exactly one credential file on disk containing a per-device key that the Gateway issued to that machine. The Director uses that key to call the Gateway; the local agents and cc-* CLI tools on the same machine read the same file to call their local Director's loopback Control API. A device only obtains a key by completing a pairing-code enrollment against the Gateway. Tailscale (or any network) is treated as transport security only - never as authorization. Every REST endpoint requires a key; there is no "trusted network, auth off" path.

KEY per-device key - unique per machine, individually revocable
GW Gateway - issuer + registry + API-key vault
CTRL Director Control API (loopback :7879)
CODE pairing code - short-lived, one-time, shown on a trusted screen

3. The four scenarios

1A Director enrolls (joins the Gateway)

A freshly installed Director has no key. It enrolls once, gets a per-device key, and writes it to the local credential file. This is the "register a new device" pop-up flow.

New Director Gateway Cockpit (trusted screen) | | | | 1. request to join | | | (machine name, | | | device id) | | |--------------------->| | | | 2. show PAIRING CODE | | |----------------------->| [ 4 - 7 - 2 - 9 ] | | (short-lived, OTP) | | | | | 3. submit code (typed/confirmed by the human) | |--------------------->| | | | (verify code matches, | | | not expired, unused) | | | | | 4. issue per-device KEY (unique, revocable) | |<---------------------| | | | | | 5. write CRED FILE on disk | | | | | 6. normal calls: Authorization: Bearer KEY | |--------------------->| |

Why a code, not the key itself

The pairing code is a short-lived one-time grant - it is NOT the key. The key never travels in a QR or a link. A leaked code is useless after it expires or is used once; a leaked key can be revoked for that one device without touching any other machine.

2Local agents / CLI call the Control API

The "agents" are Claude Code sessions running the cc-* tools and raw curl against http://localhost:7879 on the same machine. They are authorized by reading the local credential file - no separate enrollment.

cc-* tool / curl Director Control API (:7879, loopback) | | | read CRED FILE --> KEY | | | GET /sessions Authorization: Bearer KEY |--------------------->| | | (validate KEY) | 200 OK | |<---------------------|

"Register the agent library" option at Director enrollment

When a Director enrolls (scenario 1), it offers to also drop the credential where the local cc-* tools expect it, so the agent library is authorized in the same step. There is no separate Python SDK to register - the agent library == the cc-* CLI tools plus raw curl, all reading the one file.

3Each additional / remote machine

Identical shape to scenario 1, repeated per machine. The Gateway is the central registry, so machine B, C, ... each enroll with their own pairing code and receive their own distinct, separately-revocable key. No key is ever shared across machines.

Machine A KEY ----\ Machine B KEY -----+---> GATEWAY registry: { A: keyA, B: keyB, C: keyC, ... } Machine C KEY ----/ revoke(B) -> only B loses access

4Browser client (responsive web app - formerly the phone)

The native phone app is replaced by a responsive web app served from the Gateway. It does not enroll as a device and holds no key file. It authenticates by logging in at the Gateway front door (session cookie), the same way the Cockpit does.

Browser (phone/desktop) Gateway front door | | | GET /cockpit (or /app) | |----------------------------->| | 302 -> /login | |<-----------------------------| | POST /login (credentials) | |----------------------------->| | Set-Cookie: session | |<-----------------------------| | authenticated browsing | |----------------------------->|

4. Where the credential lives

One file per machine, read by two kinds of caller:

  • The Director - presents the key to the Gateway (register, heartbeat, verify).
  • The local agents / CLI - present the key to the local Director's loopback Control API.

Proposed location stays under the existing config root (%LOCALAPPDATA%\cc-director\config\director\). The current single shared gateway-token.txt is replaced by a per-device key file issued at enrollment. File permissions should be locked to the current user.

5. What changes vs. today

ConcernTodayTarget (per-device keys)
Credential One shared "fleet token" for the whole deployment A distinct key per device, issued by the Gateway
Control API auth Opt-in; OFF by default in Tailscale mode (trusts the network) Always required on every endpoint; network is transport only
Enrollment Present the shared token == you are in; no approval step Pairing code shown on the trusted Cockpit screen; device must prove it
Pairing artifact QR / link embeds the master secret (full compromise if leaked) Short-lived one-time code; the key never travels in a QR/link
Revocation Impossible per-device - must rotate the secret for everyone Revoke one device in the registry; others unaffected
Audit All callers indistinguishable (one secret) Per-device identity - who did what is attributable

6. Key lifecycle (registry view)

Open design questions (to lock next, one at a time)

  • Pairing-code format and lifetime (digits, how long valid, single-use).
  • Where the Cockpit pop-up appears and who is allowed to approve.
  • Whether browser clients also need a lightweight per-session token or just the login cookie.
  • Migration path off the single shared token without locking out existing machines.
