Register a Director - UI Mockup (Anchor B)

The pairing code is shown in the Gateway's own local app window on the host machine - never over the network. You must be in front of the gateway machine to enroll a device.
Mockup / design only. Per-device-key model, local-presence root of trust. Companion to SECURITY_FLOWS.html.

Where the popup lives and how it starts

The pairing code is shown by the CC Director Gateway app running on the gateway host machine - the small window docked in the bottom corner of that machine's desktop. It is not shown in the web Cockpit and never crosses the network. To register a device you have to be physically present at (or remoted into) the gateway host.

  1. On the gateway host, the user opens the corner app and clicks "Register a new device". The Gateway mints a short-lived code and shows it right there on that machine's screen, then waits.
  2. On the new machine, the user enters the Gateway URL and types the code they just read off the gateway host's screen. The not-yet-trusted machine never sees the code unless a human carried it over.
Why this is the stronger anchor: it breaks the circular dependency. Showing the code on a trusted surface used to require the Cockpit to be authenticated - but the Cockpit is open. By rooting the code in local access to the gateway host, enrollment no longer depends on Cockpit auth at all. Local presence is the hardest thing for a remote attacker (or the joining machine itself) to fake.

The flow, step by step

  1. On the gateway host's corner app, the user clicks Register a new device; a code appears, e.g. 4 7 2 9, counting down.
  2. On the new machine, the Director opens its Connect to Gateway dialog and the user enters the Gateway URL.
  3. The user reads the code off the gateway host screen and types it into the Director, then clicks Join Gateway.
  4. The Director sends {device id, machine name, code} to the Gateway. The Gateway verifies the code (matches, not expired, unused).
  5. The Gateway issues a unique per-device key, records the device, and the Director writes the key to its local credential file.
  6. Both screens confirm: the Director shows "Registered as MACHINE-B"; the gateway corner app shows the device joined.

What the web Cockpit does (and does not) do here

Enrollment never touches the web Cockpit. Because the Cockpit is currently unauthenticated, it is not trusted to mint or show codes, and it cannot approve devices. The authenticated device list and per-device Revoke view belong in the Cockpit only after Cockpit auth ships - until then, listing/revoking devices is also a local-host action in the corner app. Cockpit authentication is tracked as a separate, still-required release blocker.
Still open (next decisions, one at a time):
  • Code format/length and lifetime (shown here as 4 digits / ~5 min, single-use).
  • Headless gateway host: if no interactive desktop session exists, there is no corner window - the code must instead be readable via a local-only command on the host (same local-access anchor, different surface). Decide whether interactive session is required or we add that local CLI.
  • Whether the Director dialog also offers a "register the local agent library" step in this same flow.
