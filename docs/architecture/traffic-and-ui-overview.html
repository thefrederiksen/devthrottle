CC Director - Traffic and UI Architecture

Status: CURRENT  |  Date: 2026-06-12 (updated post-#372)  |  Verified against: main @ commit b949edb (#372 slices 1-2 PR #373, slice 3 PR #375)  |  Audience: anyone touching Gateway / Director / Cockpit traffic
Maps every network flow and UI surface across the three processes. The Cockpit's old three-way traffic split was unified by issue #372: the Cockpit now talks to exactly ONE origin - the Gateway.

1. The three processes (and the fourth UI)

CC Director is three processes plus one in-process UI. Two of them are web servers; one is a desktop app that also happens to host a web server.

ProcessWhat it isPortBind
DIRECTOR Avalonia desktop app + an in-process Control API (Kestrel). Owns the PTYs (sessions). The desktop window IS a UI; the Control API is how everything else reaches the sessions. 7879-7898 loopback only
GATEWAY Fleet aggregator + reverse proxy (YARP) + Tailscale Serve provisioner + Cockpit supervisor. The single front door. 7878 all interfaces
COCKPIT Blazor Server web UI. Never exposed directly - the Gateway reverse-proxies browsers to it. 7470 loopback only
DESKTOP UI The Director's own native window (MainWindow, SessionActionBar, CleanView, TerminalView). Talks to sessions in-process - no HTTP. n/a in-process

Public reach is via Tailscale Serve: https://<machine>.<tailnet>.ts.net/ on 443 maps to the Gateway on 7878. Each local Director's fixed port also gets a Serve mapping so it is reachable at https://<machine>.<tailnet>.ts.net:<port>.

3. About that word: "fallback"

"Fallback" here is a router default route, NOT the fallback-programming anti-pattern.

The Gateway uses YARP app.MapFallback("{*path}", ...). Routing is matched in order: explicit API endpoints first; anything left over (HTML pages, the Blazor circuit, app.css, blazor.web.js) is reverse-proxied to the Cockpit. It is deterministic "if not an API route, it is the web UI" - not "try X, silently degrade to Y." The only soft state is a 503 "Cockpit starting..." interstitial during boot/update, which is explicit and auto-retries.

4. The Cockpit has ONE channel: the Gateway (issue #372)

The Cockpit talks to exactly one origin - the Gateway that served its page. Every call, whatever its nature, lands there; the Gateway routes it onward:

   Browser (Blazor circuit + JS)
        |
   +----+-----------------------------------------------------------+
   |  COCKPIT  ->  GATEWAY :7878  (the ONLY HTTP destination)        |
   |                                                                |
   |  READ aggregate (Gateway answers itself):                      |
   |      roster, turnbriefs, lists, interrupted, directors,        |
   |      create-session, fleet-wide fanout                         |
   |                                                                |
   |  PER-SESSION verbs (Gateway forwards to the owning Director    |
   |  via the /sessions/{sid}/{**rest} catch-all + explicit legs):  |
   |      prompt, keystroke, interrupt, escape, clear-context,      |
   |      history, queue*, git, usage, brief, recap, summary,       |
   |      screenshots LIST + bytes + delete, hold, rename,          |
   |      upload-image, terminal WS /stream, dictate WS             |
   |                                                                |
   |  PER-DIRECTOR verbs (Gateway forwards by Director id):         |
   |      /directors/{id}/settings  (GET + PUT)                     |
   +----------------------------------------------------------------+

The rule is one sentence now: "the Cockpit talks to the Gateway; the Gateway talks to Directors." DirectorClient carries no Director addresses at all - session-scoped methods take a session id, settings take a Director id, and fanout is a single fleet-wide POST /fanout on the Gateway.

History: this used to be a three-way split (the wart this doc originally named)

Before issue #372, per-session REST went DIRECT to each session's advertised TailnetEndpoint while streams/screenshot-bytes went via the Gateway proxy and aggregates went to the Gateway API - three patterns, inconsistently applied (screenshot LIST direct, screenshot BYTES proxied; keystrokes direct, terminal output proxied). The direct leg re-created the loopback-resolution failure #268 had fixed for streams: a Director advertising http://127.0.0.1:788x made a remote browser dial its OWN loopback, so writes silently failed while streams worked. Fixed in three slices: PR #373 (generic catch-all forwarder + owner-cache fast path; Cockpit session verbs repointed) and PR #375 (screenshots list/delete, settings by Director id, fleet fanout; all stale TailnetEndpoint guards swept). A loopback-only Director is now fully drivable from a remote browser.

The Gateway is the single front door. It pulls /sessions from every registered Director in parallel to build the fleet roster (total latency = slowest single Director, not the sum). Directors only push lightweight signals: register / heartbeat / doorbell / verify-nonce. Wingman briefs are generated Gateway-side and pulled by clients.

9. The #372 cleanup: SHIPPED

The unification this doc originally proposed is implemented, QA-verified, and merged: issue #372 (slices 1-2, PR #373, commit eae996d) and follow-up #374 (slice 3, PR #375, commit b949edb).

What shipped

Slice 1 - the Gateway's SessionWsProxyEndpoints gained a catch-all /sessions/{sid}/{**rest} (any HTTP method) that reverse-proxies to the owning Director at the same path, with an owner-cache fast path so the per-keystroke leg costs a dictionary lookup instead of a fleet fan-out (a stale entry self-heals via the live resolve). Slice 2 - DirectorClient repointed at the Gateway base; ~41 session-scoped call sites updated; the TailnetEndpoint guards that blocked loopback-only Directors dropped. Slice 3 - screenshots LIST/DELETE routed by session id, Director settings by Director id (/directors/{id}/settings), fanout collapsed to one fleet-wide Gateway /fanout call; the "no tailnet endpoint" dead-end pane and the vestigial DirectorBase component parameters removed.

What it bought (all verified live and by the round-trip suites):

  • The loopback-resolution failure class is dead - a Director that advertises only http://127.0.0.1:788x is fully drivable from a remote browser, because the browser never dials the Director.
  • One trust/cert/reachability story - the Gateway front door - for writes, reads, streams, and bytes alike.
  • Local dev is trivial - run the Cockpit behind the Gateway and nothing same-origin breaks.
  • One mental model - "the Cockpit talks to the Gateway; the Gateway talks to Directors."

Extension rules going forward: a NEW per-session verb on the Director Control API needs no Gateway change - the catch-all carries it. A new Director-scoped (non-session) verb needs an explicit forwarder keyed by Director id, like /directors/{id}/settings. Never reintroduce a TailnetEndpoint guard or a Director-address HTTP call in the Cockpit - the remaining TailnetEndpoint references there are display-only. The Director Control API itself is unchanged; only who dials it changed: always the Gateway, never the browser.
