CC Director - Traffic and UI Architecture

Status: CURRENT  |  Date: 2026-06-12  |  Verified against: working tree @ v0.6.23 (commit 28b1da3)  |  Audience: anyone touching Gateway / Director / Cockpit traffic
Maps every network flow and UI surface across the three processes, names the one architectural wart, and points at the cleanup.

1. The three processes (and the fourth UI)

CC Director is three processes plus one in-process UI. Two of them are web servers; one is a desktop app that also happens to host a web server.

ProcessWhat it isPortBind
DIRECTOR Avalonia desktop app + an in-process Control API (Kestrel). Owns the PTYs (sessions). The desktop window IS a UI; the Control API is how everything else reaches the sessions. 7879-7898 loopback only
GATEWAY Fleet aggregator + reverse proxy (YARP) + Tailscale Serve provisioner + Cockpit supervisor. The single front door. 7878 all interfaces
COCKPIT Blazor Server web UI. Never exposed directly - the Gateway reverse-proxies browsers to it. 7470 loopback only
DESKTOP UI The Director's own native window (MainWindow, SessionActionBar, CleanView, TerminalView). Talks to sessions in-process - no HTTP. n/a in-process

Public reach is via Tailscale Serve: https://<machine>.<tailnet>.ts.net/ on 443 maps to the Gateway on 7878. Each local Director's fixed port also gets a Serve mapping so it is reachable at https://<machine>.<tailnet>.ts.net:<port>.

3. About that word: "fallback"

"Fallback" here is a router default route, NOT the fallback-programming anti-pattern.

The Gateway uses YARP app.MapFallback("{*path}", ...). Routing is matched in order: explicit API endpoints first; anything left over (HTML pages, the Blazor circuit, app.css, blazor.web.js) is reverse-proxied to the Cockpit. It is deterministic "if not an API route, it is the web UI" - not "try X, silently degrade to Y." The only soft state is a 503 "Cockpit starting..." interstitial during boot/update, which is explicit and auto-retries.

4. The Cockpit reaches a session THREE ways (the wart)

This is the part that is confusing and worth cleaning up. The Cockpit does not have one channel to a session - it has three, chosen per-call:

   Browser (Blazor circuit + JS)
        |
   +----+-----------------------------------------------------------+
   |  COCKPIT                                                        |
   |                                                                |
   |  (A) READ aggregate ........> GATEWAY :7878  (Cockpit:GatewayUrl)|
   |      roster, turnbriefs, lists, interrupted, directors,         |
   |      create-session                                            |
   |                                                                |
   |  (B) WRITE / per-session REST > DIRECT to session.TailnetEndpoint|
   |      prompt, keystroke, interrupt, escape, clear-context,      |
   |      history, queue, settings, git, usage, brief, recap,       |
   |      summary, screenshots LIST, hold, rename, upload-image     |
   |                                                                |
   |  (C) STREAM / bytes ..........> SAME-ORIGIN -> GATEWAY proxy ->  |
   |      terminal WS /stream, dictate WS,         owning Director   |
   |      screenshot BYTES /screenshots/file                        |
   +----------------------------------------------------------------+

The code's stated rule is "reads aggregate through the Gateway; writes and streams go direct to the Director." In practice the split is three-way and inconsistent:

  • screenshot LIST goes direct-to-Director (B), but screenshot BYTES go via the Gateway proxy (C).
  • brief / recap / git / usage go direct-to-Director (B), but turnbriefs go via the Gateway (A).
  • The terminal output stream goes via the Gateway proxy (C, which issue #268 built to kill the loopback-resolution failure), but the terminal keystroke leg posts direct to the Director (B) - re-opening the exact failure mode #268 closed.

Why this bites

Pattern B requires the browser to reach the Director's advertised TailnetEndpoint directly. A Director that can only advertise a loopback URL (http://127.0.0.1:788x) makes a remote phone resolve that to its OWN loopback - so writes silently fail while streams (routed through the Gateway) keep working. It also breaks local dev: running the Cockpit standalone makes "same-origin" point at the Cockpit instead of the Gateway, so streams and screenshots 404 while REST still works. Same root cause, two symptoms.

       ONE url:                           +-----------------------+

The Gateway is the single front door. It pulls /sessions from every registered Director in parallel to build the fleet roster (total latency = slowest single Director, not the sum). Directors only push lightweight signals: register / heartbeat / doorbell / verify-nonce. Wingman briefs are generated Gateway-side and pulled by clients.

9. Cleanup target: one channel, not three

The architecture is sound. The single wart is section 4: the Cockpit reaches a session three ways, and the direct-to-Director leg (B) re-introduces the loopback/cross-origin failure that the same-origin proxy (C) was built to eliminate.

Proposed direction: route ALL session traffic same-origin through the Gateway proxy.

Promote the per-session reads and writes (pattern B) onto the same same-origin Gateway proxy that already carries streams and screenshot bytes (pattern C). The Cockpit would talk to exactly one origin (the Gateway that served its page) for everything session-scoped; the Gateway forwards to the owning Director by session id - the lookup it already does for the WS/screenshot proxy.

What this buys:

  • Kills the loopback-resolution failure class entirely - not just for streams. A Director that advertises only loopback still works for writes, because the browser never dials the Director.
  • Remote write path becomes as robust as the stream path - one trust/cert/reachability story (the Gateway front door), not two.
  • Local dev becomes trivial - run the Cockpit behind the Gateway and nothing same-origin breaks; no standalone-port surprises.
  • One mental model - "the Cockpit talks to the Gateway; the Gateway talks to Directors." The three-way split collapses.

Scope sketch (the cleanup issue tracks the detail): add per-session REST forwarding to the Gateway (generalize the existing SessionWsProxyEndpoints owner-lookup to plain HTTP verbs), point DirectorClient at the same-origin base instead of TailnetEndpoint, and retire the browser's direct-to-Director leg. The Director Control API stays exactly as-is (it is still the thing that owns the PTY); only who dials it changes - always the Gateway, never the browser.
