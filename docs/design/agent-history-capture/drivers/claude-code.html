Driver: Claude Code — history capture

How cc-director captures a full Claude Code conversation and turns it into the canonical model.

The core idea: two decoupled paths

Claude Code runs as a full-screen terminal application (it switches to the alternate screen, ESC[?1049h), so the terminal scrollback is empty — we cannot scrape it. But Claude writes its own complete conversation log to disk as JSON lines. Capturing history therefore splits into two separate, decoupled jobs:

Pointer path · control

The hooks

Tiny, event-driven. Answers only: which transcript file is the live conversation right now? This is the part that survives /clear and compaction.

Content path · data

The transcript reader

Reads the file the pointer names and normalizes it. Answers: what was actually said — user, assistant, tool calls, tool results.

Key idea: the hooks never carry the conversation. They carry only a pointer to where it lives. The conversation itself stays in Claude’s own .jsonl, which we read directly. That is what makes it robust.

What a “hook” is — and the problem it solves

A Claude Code hook is simply a command Claude runs by itself when a lifecycle event happens. We register ours in a settings file and pass it with --settings, which merges with the user’s own hooks (Claude Code issue #11392) — so it never disables their hooks.

The pointer problem (why this exists)

When cc-director launches a session it preassigns the id with --session-id, so it knows the first file. But on /clear and on auto-compaction, Claude silently starts a new session id and a new .jsonl file. Without a signal, cc-director would keep reading the old, frozen file forever. The SessionStart hook fires at exactly those switch moments.

SessionStart sourcefires when
startupa fresh session begins
resumea session is resumed
clearthe user runs /clear
compactthe context auto-compacts (or /compact)

What the hook receives (on stdin)

{
  "session_id":      "bfaa74c9-3e87-4d16-8c23-3dbfc572e9c3",
  "transcript_path": "C:/Users/soren/.claude/projects/.../bfaa74c9-....jsonl",
  "hook_event_name": "SessionStart",
  "source":          "clear",
  "cwd":             "D:/ReposFred/devthrottle"
}

Our script report-session.ps1 does only four things: read that JSON; read CC_DIRECTOR_API + CC_SESSION_ID from the injected environment; POST { claudeSessionId, transcriptPath, source } to /sessions/{id}/claude-hook; then swallow every error and exit 0 (a hook must never block or break the session). The endpoint calls Session.UpdateClaudeSessionPointer(…), setting Session.ClaudeTranscriptPath — that field is the pointer.

Without the t6 hook, cc-director would still be reading <A>.jsonl at t7 and be permanently stale. Verified live: a real /clear moved the pointer from 5af19963… to bfaa74c9… — both the id and the transcript path.
