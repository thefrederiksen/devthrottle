Components

  • Top bar — title, summary N sessions . M red, CARDS/LIST toggle, /api + healthz links
  • Group header — per machine: MACHINE . USER . N SESSIONS
  • Session card — status dot, name, CLAUDECODE badge, repo, state + relative time, full-width Open button
  • Empty area — desktop only, below the left-packed cards
MEDDesktop dead space

Fixed-width cards left-pack into the top-left; roughly 70% of a 1440px viewport is empty, which reads as a broken layout. Use a responsive grid (auto-fill, minmax(320px,1fr)) or a centered max-width column.

MEDStatus is communicated by color alone

The dot (blue/green/red) is the only state glyph; the activity text is separate and not bound to it. Add a dot+text chip so state survives greyscale and color blindness, and two render paths cannot disagree.

MEDUnreachable Directors are not surfaced

The API returns machineErrors, but the page renders only healthy sessions. A remote user gets no signal that a machine is down. Render an inline 'unreachable' row per error.

LOWHeader links wrap on mobile

/api and healthz drop to a second row. Harmless, but these are developer links and could be tucked into a menu on small screens.

LOWRepo line is redundant

Every card repeats cc-director under a name that already contains it. Show repo only when it differs from the group/name.

GOODMobile cards are solid

Full-width stacked cards with a full-width Open button are a good touch layout; nothing to change.

Components

  • Top bar — MANAGER, session name + repo path, Working badge, Cards link
  • Hint line — 'Manager chat. Phase 1: text only...'
  • Chat history — empty in current state (~85% of the screen)
  • Composer — message textarea, Ctrl+Enter hint, Send button
HIGHThe per-Director entry point is a bare relay chat, not a dashboard

Opening a Director directly lands on 'Manager chat. Phase 1: text only' with a huge empty area. It does not list the Director's sessions or repos and offers no way to start a session, so a remote user who opens a Director cannot navigate its work.

HIGHOverlaps with the Session view with no clear distinction

The chat relays to 'the configured agent' (one session). A remote user cannot tell why they would use this instead of opening a session. Either make this a real per-Director session list (the gateway group scoped to one machine), or fold it into the gateway/session flow.

MEDMostly empty on both viewports

Desktop and mobile are ~85% empty. Even as a chat, an empty state with recent context or guidance would earn the space.

MEDWeak wayfinding

Only a 'Cards' link returns to the gateway; no breadcrumb, no session switcher, no indication which session this chat is bound to beyond the title.

GOODClear input affordance

Placeholder, Ctrl+Enter hint, and Send button are unambiguous, and the composer is responsive across viewports.

GOODMobile layout is genuinely mobile-first

Status card, a digested LAST TURN summary, then AGENT ACTIVITY. This is the strongest remote surface and the right pattern for a phone.

MEDDesktop wastes horizontal space

The transcript sits in a ~720px centered column with wide empty gutters on a 1440px screen. The SUPERVISOR LOG could sit beside the transcript (two-pane) to use the width.

MEDMobile composer overlaps its own placeholder

The textarea placeholder ('...Shift+Enter for newline)') is clipped behind the Speak/Send buttons on narrow widths. Layout bug to fix.

LOWDesktop and mobile show different information

Desktop shows the raw transcript; mobile shows a digested LAST TURN summary. If intentional, fine; otherwise the desktop view could adopt the summary card too.

LOWDeprecated tab shown remotely

'Agent (legacy)' is still presented to remote users; consider hiding deprecated tabs on the remote surface.

GOODStrong observability built in

SUPERVISOR LOG (status transitions with reasons) and ASK THE SUPERVISOR are high-value remote affordances, and the Speak button supports mobile voice input.

4. Cross-cutting findingsapply across surfaces

HIGHNo authentication on the remote surfaces

All three surfaces loaded over the tailnet with no login (verified directly). Security rests entirely on the Tailscale network boundary, so anyone on the tailnet can open any session and send input to a live agent. Decide deliberately: is the tailnet boundary sufficient, or should the Director Control API carry the same token/cookie gate the gateway already supports?

HIGHWayfinding is fragmented across three ports

The gateway is on :443; each Director and its sessions are on :<port>. Movement between them is ad hoc (a 'Cards' link here, a back arrow there) and the port hop is visible in the URL. There is no consistent Gateway > Director > Session breadcrumb. Reverse-proxying everything under :443 as /d/{id}/... would unify both navigation and the per-port Tailscale mappings.

MEDColor-only status is a cross-surface theme

Gateway cards and the session badge both lean on color. Establish one shared status chip (dot + text) used everywhere.

MEDDesktop dead space is a cross-surface theme

Both the gateway and the session view collapse content into a fraction of a wide screen. A shared max-width / responsive-grid approach fixes both at once.

GOODVisually consistent product

Coherent dark theme, consistent accent blue, and consistent typography across all three surfaces. It reads as one product.
